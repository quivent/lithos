What Lithos writes onto — the streaming multiprocessor in exact detail

The Critical Distinction

Every Hopper SM contains two separate 256 KB memory blocks. They serve entirely different purposes. Confusing them is a common source of wrong mental models.

Register File — 256 KB

  • 65,536 x 32-bit registers
  • 64 KB per sub-partition x 4
  • 1 cycle access latency
  • Private per thread
  • Feeds ALUs directly
  • Determines occupancy

L1/Shared Memory — 256 KB

  • Unified, configurable split
  • Up to 228 KB shared + rest L1
  • 23 cycle access latency
  • Shared across all threads in SM
  • 32 banks x 4 bytes
  • 128 bytes/clock bandwidth
Same size, completely different hardware. The register file is SRAM tightly coupled to each sub-partition's execution units — one cycle, private per thread, split across four physical blocks. The L1/shared memory is a separate SRAM bank shared across all four sub-partitions — 23x slower but visible to every thread in the SM. Lithos must manage both correctly.

A cycle here is one GPU clock tick. The 400-cycle HBM latency is end-to-end: from the SM issuing ld.global.f32 to the data arriving in a register. During that ~220 ns, each of the SM's four sub-partitions could have issued ~400 other instructions — that budget is the opportunity latency hiding exploits.

Lithos controls stall counts in the control word — every memory access that misses a fast level adds hundreds of stall cycles. Lithos patterns guarantee data is staged to the right level before compute needs it. Stall counts are embedded in the 8-byte control word of each 16-byte instruction.
Lithos patterns guarantee coalesced access — a warp reading 32 contiguous 4-byte values hits one 128-byte cache line. Scattered access can waste 32x bandwidth. Lithos vocabulary words enforce coalescing by construction.

Data Flow Within a Sub-Partition

Every cycle, the warp scheduler examines its pool of up to 16 resident warps. It checks the scoreboard for each — are the operand registers ready? Is the target execution unit free? If a warp is eligible, its next instruction is dispatched.

The dispatch unit reads source operands from the register file (1 cycle), routes them to the correct execution unit, and writes results back to the register file. Load/store units access shared memory (23 cycles) or global memory through the cache hierarchy.

The two datapaths give 32 FP32 operations per cycle (one full warp). When a warp has both FP32 and INT32 instructions, Datapath B handles INT32 while Datapath A handles FP32 — but they cannot both do FP32 simultaneously from different warps.

Lithos controls register allocation — determines how many warps can be resident. At 128 regs/thread: 16,384 / (128 x 32) = 4 warps per sub-partition. At 64 regs/thread: 8 warps. More warps = more latency hiding.
Lithos controls stall counts in the binary control word — the warp scheduler reads the stall count from the control word of each 16-byte instruction to know the minimum wait before the next instruction from that warp can issue. Lithos emits these fields directly into the instruction binary.
Lithos controls shared memory usage — cp.async and TMA bypass the register file, flowing data directly from global memory to shared memory. This preserves the scarce register file for compute, not data staging.

Hopper-Specific: GSP, NVLink-C2C, Cooperative Launch

GSP — GPU System Processor (RISC-V)

Hopper embeds a RISC-V microcontroller on-die called the GSP (GPU System Processor). It handles initialization, power management, and firmware tasks. The GSP must boot and confirm readiness before the GPFIFO submission channel becomes active — any attempt to launch kernels before GSP initialization will hang. The standard NVIDIA driver (nvidia.ko) handles this; lithos.ko replicates the initialization sequence. GSP runs independently of the SM array and is not used for kernel compute.

NVLink-C2C and Address Translation Services (ATS)

GH200 connects the Hopper GPU and Grace CPU via NVLink-C2C at 900 GB/s. With ATS enabled, the CPU physical address equals the GPU virtual address — hardware performs address translation transparently. This means any memory the CPU allocates (via malloc, mmap, or the stack) is GPU-visible at the same address without any explicit mapping step. The BAR4 aperture exposes 128 GB of coherent HBM to the CPU. No cuMemAlloc is needed for weight tensors — CPU allocations are GPU-visible by construction.

Cooperative Launch and Grid-Sync Megakernels

Hopper supports cooperative kernel launch: all CTAs in a grid can reach a grid-level barrier (cooperative_groups::this_grid().sync()), allowing a single kernel to persist across an N-layer loop without returning to the host. Lithos targets 2 cooperative megakernels per model (one per forward pass phase). The grid-sync barrier requires all CTAs to be simultaneously resident, constraining the grid size to at most the number of active SMs (132 on GH200). This trades launch flexibility for zero inter-layer kernel overhead.

Grid-sync failure mode — if the grid is too large to fit all CTAs simultaneously, the barrier deadlocks. Lithos sizes grids to exactly 132 blocks (one per SM) to guarantee full residency. PTX-based grid-sync failures reported earlier were a grid-sizing issue, not a conceptual failure of cooperative launch.
