TensorRT-LLM

Why it is the fastest inference engine, and what that speed costs.

1. What TensorRT-LLM Actually Is

TensorRT-LLM is not "TensorRT but for LLMs." It is a complete inference framework — with its own model definition API, weight loading pipeline, graph compiler, runtime scheduler, memory manager, and multi-GPU orchestration layer. It happens to use TensorRT's compiler as one component in a much larger system.

The architecture has two distinct halves:

The Python Layer (Partially Open Source)

The Python layer defines the model graph using a torch-like API. You write something that looks like PyTorch — Linear, RMSNorm, Attention modules — but instead of executing eagerly, it builds a TensorRT INetworkDefinition graph. Each operation becomes a node. The graph is the input to the compiler.

Model definitions live in Python. Weight loading, checkpoint conversion, calibration, and quantization configuration are all Python. The trtllm-build command orchestrates this phase. This layer is open source on GitHub.

The C++ Runtime (Closed Source)

The C++ runtime loads serialized engine files and executes them. It manages the KV cache, schedules requests, handles multi-GPU communication via NCCL, and launches kernels. No Python executes during inference. The runtime is a compiled binary — you cannot inspect, modify, or port it.

This split is deliberate. The Python layer lets users define and customize models. The C++ runtime lets NVIDIA keep their performance-critical code proprietary. You can see what graph gets built. You cannot see how it gets executed.

Python build phase C++ runtime | | Model definition (torch-like API) Load engine file | | Graph construction (INetworkDefinition) Pre-allocate memory | | Weight binding from checkpoint Schedule requests (in-flight batching) | | TensorRT compiler (fusion, autotuning) Execute kernel sequence | | Serialized engine file (.engine) ——→ Token sampling & output

2. The Build Phase (Why It Takes Minutes)

When you run trtllm-build, six things happen in sequence. Each is expensive. Together they take 5–30 minutes depending on model size and GPU.

01 Graph Definition

The Python model definition executes. Each Linear, RMSNorm, Attention call appends nodes to a TensorRT INetworkDefinition. The result is a directed acyclic graph of operations — hundreds of nodes per transformer layer, thousands for the full model. Weight tensors are bound to specific nodes. Input/output shapes are declared (max batch size, max sequence length, max number of tokens).

This step takes seconds. It is pure Python graph construction.

02 Graph Optimization

TensorRT's compiler applies classical graph optimizations:

Constant folding: any operation on known values (like scale factors or shape calculations) is computed once and baked into the graph.

Dead code elimination: nodes whose outputs are never consumed are removed.

Layer fusion: the compiler runs pattern matching across the graph. When it finds sequences that can be replaced by a single fused kernel — like QKV projection + bias + RoPE, or LayerNorm + residual add — it collapses them into one node. This is where TensorRT's proprietary kernel library matters: it has fused kernels for hundreds of common patterns that no public library provides.

This step takes 10–30 seconds. The pattern matching is the expensive part — the number of possible subgraph matches grows combinatorially.

03 Kernel Selection

For each node in the optimized graph, TensorRT has multiple kernel implementations ("tactics") available. A single GEMM node might have 20+ candidate kernels: different tile sizes (64x64, 128x128, 256x128), different data layouts (row-major, column-major), different algorithms (cuBLAS, CUTLASS, TensorRT's internal kernels). For attention nodes, the candidates include FlashAttention-like kernels, XQA kernels, and standard multi-head attention.

TensorRT must pick one tactic per node. The "right" choice depends on the specific tensor dimensions, the GPU's SM count, available shared memory, and interactions with neighboring kernels.

The candidate enumeration is fast. The selection requires the next step.

04 Autotuning

This is where the minutes go. For each node, TensorRT launches every candidate kernel with representative data and measures wall-clock execution time. A model with 64 transformer layers, each with ~10 fused nodes, and ~20 tactics per node means 64 × 10 × 20 = 12,800 kernel launches just for timing. Each launch includes setup, execution, synchronization, and measurement.

The results are stored in a timing cache. If you rebuild with the same GPU and same shapes, the cache avoids re-timing. But change the GPU (H100 to A100), the batch size, or the sequence length, and the cache is invalidated — full autotuning runs again.

The autotuner selects the fastest tactic for each node. The combined selections form the execution plan.

This step takes 3–25 minutes. It is the dominant cost of the build phase. The --profiling_verbosity detailed flag lets you inspect which tactics were selected and why.

05 Memory Planning

With the kernel sequence fixed, TensorRT computes the exact memory layout. Every intermediate tensor's size is known at build time (because shapes are fixed). The planner assigns memory offsets so that non-overlapping tensors share the same physical memory. Buffer reuse across layers is aggressive — layer N's output buffer can be reused as layer N+2's workspace if the lifetimes don't overlap.

The activation memory size is fixed at build time and queryable via trt.ICudaEngine.device_memory_size_v2. Once built, it cannot change. The runtime pre-allocates this exact amount — no malloc during inference.

This step takes seconds. It is a linear scan over the execution order with a simple interval-scheduling algorithm.

06 Serialization

The execution plan — the optimized graph, the selected kernels (actual compiled CUDA code), the memory layout, and the model weights — is serialized into a single binary blob: the engine file. This file contains everything needed to run inference. No model definition, no Python, no weight files. Just a sequence of kernel launches with pre-assigned memory.

Engine files are not portable. They encode the specific GPU architecture (SM version), the CUDA driver version, and the TensorRT version. An engine built for H100 will not run on A100. An engine built with TensorRT 10.3 may not run with TensorRT 10.4.

Serialization takes seconds. The engine file is typically 1–2x the model weight size (weights are embedded in the engine).

3. The Optimizations (Each One Explained)

A Layer Fusion

What gets fused

The TensorRT compiler identifies and fuses common operation sequences into single kernels. The key fusions in a transformer layer:

QKV projection: three separate Linear layers (Q, K, V) become one GEMM with a single weight matrix read. Instead of three kernel launches reading three weight matrices, one kernel reads a concatenated weight matrix and produces all three outputs. This saves two kernel launch overheads and two weight-read passes.

Bias + activation: the bias add and SiLU/GeLU activation after a linear layer are fused into the GEMM's epilogue. The GEMM kernel writes the final activated output directly — no intermediate tensor materialized in HBM.

LayerNorm + residual add: the residual connection and the subsequent RMSNorm/LayerNorm are fused into one kernel. The kernel reads the residual input, adds it to the layer output, normalizes, and writes the result. One read, one write, instead of three reads and two writes.

Attention + softmax: the QK dot product, scaling, masking, softmax, and V multiplication are fused into a single attention kernel (similar to FlashAttention). Intermediate attention scores never touch HBM.

What cannot be fused

Operations across attention boundaries — the output of attention feeds into a different kernel than what produced K and V. Cross-layer dependencies prevent fusion. AllReduce operations in tensor-parallel setups are synchronization barriers that break fusion chains.

How it decides

Pattern matching on the graph. TensorRT maintains an internal catalog of fusible patterns (subgraph templates). The compiler walks the graph, matches subgraphs against the catalog, and replaces matches with fused nodes. The catalog is proprietary and grows with each TensorRT release.

Impact

A standard transformer layer without fusion requires 20–50 kernel launches. With fusion, this drops to 5–10 launches. Across 64 layers, that is the difference between 3,000+ kernel launches and ~500. Each eliminated kernel boundary saves 1–5 us of dead time plus the HBM round-trip for intermediate data. At 64 layers, fusion alone is worth 30–50% throughput improvement.

B Quantization (FP8, INT8, INT4)

FP8 on Hopper

The H100's fourth-generation tensor cores natively support FP8 (E4M3 format: 4 exponent bits, 3 mantissa bits). This is not emulated — the tensor core MMA instructions accept FP8 inputs directly, at 2x the throughput of FP16/BF16. TensorRT-LLM exploits this by quantizing both weights and activations to FP8, letting the tensor cores run at full E4M3 speed.

Calibration

FP8 quantization requires scale factors that map the FP16 dynamic range into FP8's narrower range. TensorRT-LLM's calibration runs representative data through the model and records the activation distributions. Configuration is via CalibConfig: typically 512 batches, batch_size=1, max_seq_length=2048. The calibrator computes per-tensor or per-channel scale factors that minimize quantization error.

Per-tensor vs per-channel vs per-group

Per-tensor: one scale factor for the entire weight matrix. Fastest (no extra indexing) but least accurate — outlier channels distort the scale for all channels.

Per-channel: one scale factor per output channel. Better accuracy, small overhead for looking up the scale during the GEMM.

Per-group: one scale factor per group of (typically 128) elements within a channel. Used in INT4/GPTQ/AWQ. Best accuracy, more indexing overhead.

SmoothQuant

When activation outliers make quantization difficult, SmoothQuant mathematically redistributes the quantization difficulty from activations to weights. It multiplies activations by a per-channel smoothing factor and divides weights by the same factor. Since weights are static (smoothed offline), this makes activation quantization easier at no runtime cost.

AWQ (Activation-Aware Weight Quantization)

AWQ identifies the 1% of weight channels that are most important to accuracy (determined by activation magnitudes) and protects them by keeping higher precision for those channels. TensorRT-LLM integrates AWQ as a quantization method selectable during the build phase.

C In-Flight Batching

Static batching waits for an entire batch to finish before accepting new requests. If one request in a batch of 32 generates 500 tokens while the others generate 50, the GPU sits idle for 31 slots during the last 450 tokens.

In-flight batching (enabled by default in TensorRT-LLM) operates at iteration-level granularity. At every decode step, the scheduler evaluates:

Which requests have finished? Evict them from the batch immediately. Free their KV cache blocks.

Which new requests are waiting? Insert them into the freed slots. Run their prefill (context) phase while other requests continue decoding.

Is there KV cache capacity? If not, defer new requests rather than OOM.

This is conceptually identical to vLLM's continuous batching. The difference is execution: vLLM's scheduler runs in Python and dispatches to PyTorch. TensorRT-LLM's scheduler runs in C++ and dispatches to pre-compiled kernel sequences.

An additional optimization is the overlap scheduler, which launches GPU work for step N+1 before the CPU finishes processing step N's results. This hides CPU scheduling latency behind GPU computation, yielding up to 22% additional throughput improvement.

D KV Cache Management

Paged KV Cache

Borrowed from vLLM's PagedAttention concept. The KV cache is split into fixed-size blocks (pages). Tokens map to pages non-contiguously, like virtual memory. This eliminates the fragmentation problem: without paging, each sequence needs a contiguous buffer sized for max_seq_length, wasting 60–80% of KV memory on padding. With paging, blocks are allocated on demand.

By default, TensorRT-LLM allocates 90% of remaining GPU memory to the KV cache pool after loading the engine and activation buffers. This can be tuned via maxTokens or freeGpuMemoryFraction.

KV Cache Reuse (Prefix Caching)

If two requests share a common prefix (e.g., the same system prompt), the KV cache for that prefix is computed once and shared. The second request skips the prefill for the shared tokens. This is significant for multi-turn conversations and few-shot prompting where system prompts repeat across every request.

Quantized KV Cache

KV values can be stored in FP8 or INT8 instead of FP16, halving KV cache memory and doubling the number of sequences that fit in memory. Enabled via --use_fp8_context_fmha=True at build time. The attention kernel reads quantized KV values and dequantizes inline.

Impact: up to 1.45x throughput improvement in decode-heavy scenarios (more sequences in flight) and 1.09x in prefill-heavy scenarios (less memory pressure).

E Multi-GPU Parallelism

Tensor Parallelism (TP)

Split weight matrices across GPUs along the column or row dimension. Each GPU computes a partial result, then AllReduce synchronizes. For a 70B model on 2 GPUs, each GPU holds half the weights and does half the compute. The AllReduce after each layer exchanges partial results — typically 8–32 KB per layer for attention output, much more for FFN output.

TP requires high inter-GPU bandwidth. On NVLink 4.0 (H100): 900 GB/s bidirectional. The AllReduce for a 4096-dim hidden state across 8 GPUs takes ~1 us — negligible. But scale to 16 GPUs across two nodes (InfiniBand at 400 Gb/s) and the communication starts limiting scaling efficiency.

Pipeline Parallelism (PP)

Different layers on different GPUs. GPU 0 runs layers 0–31, GPU 1 runs layers 32–63. Communication happens only at the boundary: the output of layer 31 is sent to GPU 1, which processes layers 32–63 and sends the result back. Less communication bandwidth needed than TP, but creates pipeline bubbles — GPU 0 is idle while GPU 1 is computing.

TensorRT-LLM uses NCCL-wrapped plugins (allreduce(), allgather(), send(), recv()) exposed in the Python graph definition, so parallelism is built into the compiled engine.

Communication Overhead

At batch=1, the communication-to-compute ratio is unfavorable: the computation per layer is tiny (one matrix-vector multiply) but the AllReduce cost is fixed. TP scaling at batch=1 typically yields 1.6–1.8x speedup from 2 GPUs, not 2x. At batch=64, compute dominates and scaling efficiency approaches 1.9x per doubling.

F Custom Kernels

TensorRT-LLM does not rely on cuDNN or cuBLAS for its performance-critical operations. It ships custom CUDA kernels for every hot path:

XQA (eXtended Query Attention)

A generation-phase attention kernel optimized for GQA (Grouped Query Attention) and MQA (Multi-Query Attention). Uses tensor cores for the QK dot product and V multiplication, reducing data loading and format conversion overhead. The XQA kernel provides 2.4x more Llama-70B throughput within the same latency budget: from 1,227 tok/s/GPU to 2,941 tok/s/GPU on H200. On 8x H100, throughput increased 1.9x (13,232 to 25,300 tok/s). The kernel includes a JIT codepath for Hopper's QGMMA (Quantized Group Matrix Multiply-Accumulate) instructions.

FP8 GEMM Kernels

Custom matrix multiply kernels that consume FP8 inputs directly, using Hopper's native FP8 tensor core instructions. These bypass cuBLAS entirely. The kernels fuse dequantization scaling into the GEMM epilogue — the scale factor multiply happens in registers after the MMA, not as a separate pass.

Fused MoE Kernels

For Mixture-of-Experts models (Mixtral, DeepSeek), TensorRT-LLM provides fused kernels that handle the gate network, top-K expert selection, token routing, expert GEMMs, and output combination in a single kernel. Supports FP16, BF16, FP8, and NVFP4 precisions. The Autotuner specifically tunes MoE kernel parameters.

How These Compare to FlashAttention

FlashAttention is a single algorithm with a public implementation. TensorRT-LLM's attention kernels are a library of specialized implementations — different kernels for prefill vs decode, for GQA vs MHA, for FP8 vs FP16, for different head dimensions. FlashAttention optimizes the attention computation. TensorRT's kernels optimize the attention computation for the specific deployment configuration, selected during autotuning.

G CUDA Graphs

During the decode phase (generating one token at a time), the kernel sequence is the same every iteration — same kernels, same memory locations, only the data changes. TensorRT-LLM captures this sequence as a CUDA graph: a single object that replays the entire multi-kernel sequence with one CPU-side API call.

Without CUDA graphs, each kernel launch costs 3–10 us of CPU dispatch. With ~500 kernel launches per forward pass (after fusion), that is 1.5–5 ms of pure CPU overhead — potentially more than the GPU compute time at batch=1. CUDA graphs eliminate this: one launch, one dispatch, all kernels fire in sequence.

TensorRT-LLM uses graph padding to handle varying batch sizes. Instead of recording a separate graph for every possible batch size, it pads to the nearest cached size. The overhead of computing on padded (unused) elements is less than the cost of re-recording the graph. This yields up to 22% throughput improvement over non-graphed execution.

The tradeoff: CUDA graphs are rigid. They encode specific memory addresses and tensor shapes. Dynamic shapes (changing sequence lengths during prefill) cannot use graphs — only the fixed-shape decode phase benefits.

4. The Runtime (What Executes)

After the build phase produces an engine file, the runtime is pure C++. No Python interpreter, no garbage collector, no framework overhead.

Execution Loop

The PyExecutor(Worker) process runs in a continuous loop on each GPU rank. Each iteration:

1. Fetch new requests from the input queue 2. Scheduler decides which requests to execute this step 3. KVCacheManager allocates/deallocates page blocks 4. ModelEngine runs the forward pass (the compiled kernel sequence) 5. Sampler selects output tokens (greedy, top-k, top-p, beam search) 6. Finished requests are evicted; results returned

Pre-Allocated Memory

All memory is allocated at startup. The engine file declares its exact activation memory requirement. The KV cache pool is pre-allocated from remaining GPU memory. During inference, no cudaMalloc is called. Every buffer has a pre-assigned address. The kernel sequence writes to pre-determined offsets. This is what makes the execution path deterministic.

Deterministic Execution Order

The engine file contains an ordered list of kernel launches. There is no dynamic scheduling, no graph re-optimization, no runtime fusion decisions. The C++ runtime iterates through the list and launches each kernel in order. The only runtime decision is which requests are in the current batch — everything else was decided at build time.

Engine File to Kernel Launches

The engine file is effectively a program: a sequence of (kernel_function, grid_dims, block_dims, shared_mem, arguments) tuples. The runtime maps this to cuLaunchKernel calls (or CUDA graph replay). The compiled kernel code is embedded in the engine file — no JIT compilation at load time. Load the file, allocate memory at the specified offsets, and start launching.

5. The Tradeoffs (What It Costs)

Build Time: 5–30 Minutes Per Configuration

Change the maximum batch size? Rebuild. Change the maximum sequence length? Rebuild. Move to a different GPU? Rebuild. Upgrade TensorRT? Rebuild. Each rebuild runs the full autotuning pass. For a 70B model, expect ~28 minutes of compilation. If you serve multiple model configurations, you maintain multiple engine files. A matrix of 3 batch sizes × 2 sequence lengths × 2 GPU types = 12 engine files, each requiring a separate build.

Rigidity: Frozen Execution Plans

The execution plan is fixed at build time. Activation memory is pre-computed and cannot change. If a request arrives with a longer sequence than max_seq_length, it is rejected — not handled with a slower fallback. Dynamic shapes require building with padded maximums, which wastes memory for smaller inputs, or maintaining multiple engine files for different size ranges.

Complexity: Dozens of Configuration Parameters

The trtllm-build command exposes dozens of flags: --max_batch_size, --max_input_len, --max_seq_len, --max_num_tokens, --max_beam_width, --use_fp8_context_fmha, --strongly_typed, --context_fmha, quantization type, calibration dataset, parallelism configuration. Getting the combination right for your workload requires experimentation. The model definition API differs from PyTorch/HuggingFace — not all architectures are supported, and porting a new architecture requires rewriting the model definition in TensorRT-LLM's API.

NVIDIA Lock-In

The C++ runtime is closed source. Engine files encode the GPU architecture, CUDA driver version, and TensorRT version. They are not portable between GPU generations (H100 vs A100), between driver versions, or between TensorRT releases. You cannot inspect the generated kernels. You cannot run on AMD, Intel, or Apple hardware. You are locked to NVIDIA's release cadence and deprecation decisions.

Memory Overhead

The engine file embeds the model weights plus the compiled kernels plus the memory plan. It is typically 1–2x the size of the raw model weights. The autotuning workspace at build time can require significant temporary GPU memory — you may need more GPU memory to build than to run.

No DeltaNet Support

TensorRT-LLM has no native support for DeltaNet or linear attention layers. Qwen 3.5-27B's hybrid attention-DeltaNet architecture would require writing custom TensorRT plugins in C++ for the DeltaNet layers. Those plugins would bypass TensorRT's fusion and autotuning — they run as opaque black boxes in the execution graph. This partially defeats the purpose of using TensorRT: the DeltaNet layers would be unoptimized islands surrounded by optimized transformer layers.

7. What Lithos Learns From TensorRT

TensorRT-LLM validates several design decisions that Lithos independently arrived at. It also reveals the limits of the "heavy build, pure execution" philosophy.

Build-Time Kernel Selection and Specialization

TensorRT proves that selecting the optimal kernel for each operation at build time — rather than at runtime — is worth minutes of compilation. Lithos agrees with the principle but rejects the time cost. Lithos selects kernels at load time in microseconds, not minutes. The "autotuning" is replaced by SASS-level kernels that are already optimal for the target architecture — there is no need to time 20 tactics when you have one kernel written directly against the hardware.

Layer Fusion

TensorRT fuses at the graph level: pattern-match subgraphs, replace with fused kernels. Lithos fuses at the vocabulary level: each Forth word compiles to SASS that includes the fused operations by construction. There is no pattern matching because the vocabulary defines the fused operations directly. The result is the same — fewer kernel launches, no intermediate HBM traffic — but the mechanism is compositional rather than template-based.

No Python in the Inference Path

TensorRT's C++ runtime agrees with Lithos's design: Python must not be in the hot path. Every microsecond of interpreter overhead is a microsecond the GPU is idle. Lithos has no Python anywhere — not in the build phase, not in the runtime, not in the scheduler.

Pre-Allocated Memory

TensorRT pre-computes memory layout at build time and allocates everything at startup. Lithos does the same with mmap + pre-allocation: weight files are memory-mapped, activation buffers are pre-allocated, and the KV cache pool is reserved at startup. No malloc during inference.

8. The Key Insight

TensorRT-LLM is fast because it moves all decisions to build time. Kernel selection, layer fusion, memory planning, autotuning — everything that requires intelligence happens once, during a multi-minute compilation pass. The inference path is pure execution: a pre-planned sequence of kernel launches with pre-assigned memory. No decisions, no allocations, no overhead.

This is the right idea. The question is whether it requires minutes.

TensorRT needs minutes because it works at the wrong abstraction level. It starts with a high-level graph (PyTorch-like operations), lowers it through a general-purpose compiler (pattern matching, tactic enumeration), and produces GPU code via CUDA C++ compilation. Each layer of abstraction requires a search through a space of possibilities.

Lithos eliminates the search by eliminating the abstraction layers. The Forth vocabulary defines operations that map directly to SASS instructions. There is no graph to pattern-match — the fused operations are the vocabulary. There are no tactics to enumerate — the SASS kernel is the tactic, written once for each SM architecture. There is no memory layout to compute — the Forth stack machine and pre-allocated buffers define it.

The result is the same: a pre-planned sequence of optimized kernels with no runtime overhead. The build takes microseconds instead of minutes. And because recompilation is instant, Lithos adapts to different batch sizes, sequence lengths, and model architectures without maintaining a library of frozen engine files.

TensorRT proved the destination is correct. Lithos found a shorter path.

Lithos — from lithography. Writing on stone. The process that puts circuits on silicon. Lithos writes on the GPU.
