Kernel

Anatomy of a cubin. Launch to completion. The life of a thread block.

What a Kernel Is

A kernel is a function compiled for the GPU, packaged in an ELF file (the cubin), stored as a symbol in a module. At launch it is replicated across a grid of thread blocks; each block gets its coordinates in blockIdx, each thread its coordinates in threadIdx. Same code, different indices, parallel execution.

The kernel is identified by a CUfunction handle. That handle points into a cubin loaded into GPU memory. Every invocation via cuLaunchKernel reads the same code.

The Cubin File

A cubin is a standard ELF for e_machine = EM_CUDA. It has sections like any other ELF, plus some NVIDIA-specific ones.

Metadata — What the Driver Needs

Before a kernel can launch, the driver asks the cubin four questions. The answers are in .nv.info.KERNEL:

AttributeExampleUsed for
EIATTR_REGCOUNT168Occupancy calc — how many warps fit per SM
EIATTR_MAX_THREADS128Upper bound on blockDim
EIATTR_SHARED_SIZE49152Static shared alloc per block
EIATTR_PARAM_CBANKc[0][0x160..0x210]Where args live in constant bank 0

EIATTR_EXIT_INSTR_OFFSETS[0x28c0]Where EXIT lives (for debuggers)

Lithos generates cubins directly (no nvcc round-trip). The assembler emits these attributes as part of the module write. cuModuleLoad parses them once; every cuLaunchKernel thereafter is metadata-free.

The SASS Inside

First 20 instructions of one of our QKV-projection decode kernels, disassembled with cuobjdump:

    /*0000*/   LDC        R1,   c[0x0][0x28]         ; stack pointer
    /*0010*/   S2R        R0,   SR_TID.X                ; threadIdx.x
    /*0020*/   S2R        R3,   SR_CTAID.X              ; blockIdx.x
    /*0030*/   IMAD.WIDE  R2,   R3, c[0x0][0x0], R0   ; tid_x + bid_x * blk_x
    /*0040*/   ULDC.64    UR4,  c[0x0][0x160]        ; load weight ptr
    /*0050*/   ULDC.64    UR6,  c[0x0][0x168]        ; load input ptr
    /*0060*/   SHF.L.U32  R4,   R2, 0x5, RZ            ; row offset * 32
    /*0070*/   IMAD.WIDE  R6,   R4, 0x10, UR6          ; input + offset
    /*0080*/   LDG.E.128  R8,   [R6]                   ; load input tile
    /*0090*/   LDG.E.128  R12,  [R6+0x10]              ; load input tile +1
    /*00a0*/   CP.ASYNC.BULK.TENSOR.2D  [R10], [UR4]     ; TMA weight load
    /*00b0*/   BAR.SYNC   0x0                             ; block-wide barrier
    /*00c0*/   LDSM.16.M88.4 R16, [R24]                  ; ldmatrix for mma
    /*00d0*/   LDSM.16.M88.4 R20, [R24+0x800]             ; next tile
    /*00e0*/   WGMMA.M64N128K16.F16  R32, R16, R20, R32
    /*00f0*/   WGMMA.M64N128K16.F16  R32, R16, R20, R32
    /*0100*/   WGMMA.COMMIT_GROUP
    /*0110*/   WGMMA.WAIT_GROUP  0x0
    /*0120*/   STG.E.128  [R26], R32                     ; write output tile
    /*0130*/   EXIT

Every line is 128 bits on the wire. The low 64 bits are the opcode and operands; the high 64 bits carry the control info — reuse flags, wait-count, stall cycles, yield hints.

cuLaunchKernel

What happens when the host issues cuLaunchKernel(func, grid, block, args, shared, stream):

Launch path — CPU to first SASS instruction
1. Host thread (CPU)
  • driver assembles a launch descriptor
  • serializes args into .nv.constant0 layout
  • writes the descriptor into the stream's command buffer
  • rings the doorbell — a single MMIO write to a GPU-mapped page
C2C / PCIe ↓
2. GPU Host Interface (GHI)
  • reads the doorbell
  • DMA-pulls the command buffer entry
  • validates and queues on the GPU's Copy Engine / CE stream
↓
3. Compute Work Distributor (CWD)
  • inspects gridDim, blockDim, resource reqs (R, shmem)
  • walks the 132 SMs, finds ones with capacity
  • hands each SM a block (with its blockIdx)
↓
4. SM receives a block
  • allocates register slice, shared mem, warp slots
  • loads .nv.constant0 contents into the constant bank
  • dispatches warps to the 4 sub-partitions
  • first warp runs LDC R1, c[0x0][0x28] — execution begins

From doorbell to first instruction: typically 2-5 us on GH200. For long-running kernels the launch cost is irrelevant; for a 40 us decode kernel it is meaningful — which is why Lithos uses graph capture, persistent kernels, and co-operative launches to eliminate it where possible.

Life of a Block

A thread block is born when the CWD assigns it to an SM. It runs until every thread in it hits EXIT. It never migrates. It never pauses (beyond warp scheduling within the block). When it finishes, the SM retires its resources and asks the CWD for the next block.

Block lifetime on one SM
t=0
CWD picks SM 17 for block (5, 2).
t + ~100ns
SM 17 reserves: 128 threads × 168 regs = 21,504 reg entries, 49,152 B shared memory, 4 warp slots on sub-part 0.
t + ~200ns
First warp starts issuing. Other warps in the block start as slots free up.
t = …
block runs to completion
t = end
Last warp hits EXIT. SM retires resources.
t + ~50ns
CWD asks SM 17 for next assignment. Gets block (5, 3).

Blocks do not communicate with other blocks. Block (0, 0) and block (1, 0) might be running on the same SM, different SMs, or at different times. The kernel cannot assume any ordering. The grid is an unordered bag of independent computations.

Grid and Block Dims

gridDim is the shape of the grid — how many blocks, in up to 3 dimensions. blockDim is the shape of each block. Both are fixed at launch; kernels read them via special registers SR_CTAID and SR_TID.

KernelblockDimgridDimMapping
QKV proj (M=1, N=12288)12896Each block computes 128 output columns
FlashAttn decode25632 headsOne block per attention head
RMSNorm1024seq_lenOne block per token
MLP up/gate fused128(M/128, 1, 1)Each block computes one output row tile

Lithos picks these sizes to saturate 132 SMs without overshoot. For decode (batch=1), gridDim is usually in the range 96-256. For prefill (batch=N), gridDim scales with sequence length; that is where the GPU runs wide.
