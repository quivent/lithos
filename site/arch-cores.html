Cores

The actual compute units. What each one does, what each one is for, what it costs.

Cells in a Sub-Partition

A sub-partition is the smallest scheduling unit. Four per SM. Each contains its own warp scheduler, its own 64 KB register file slice, and a fixed set of execution units. When a warp issues an instruction, it routes to the unit that can execute that opcode.

CUDA Core

A single-lane scalar FP32 unit. One FMA per cycle: d = a * b + c. 32-bit input, 32-bit output. Takes one warp (32 threads) one cycle to fully occupy the 32 CUDA cores of a sub-partition.

Instructions that run here:

In inference: CUDA cores handle the elementwise operations — RoPE sin/cos application, residual adds, activation gates, scale factors, index arithmetic for gather/scatter. They are not the bottleneck; they are the glue.

Tensor Core

A matrix engine. The smallest operation is a 16x8x16 matrix-multiply-accumulate (the m16n8k16 shape). Takes FP16/BF16/FP8/INT8 inputs, accumulates in FP32 or FP16. One tensor core per sub-partition, four per SM.

The key instruction:

wgmma.mma_async.sync.aligned.m64n256k16.f16.f16.f16
    // 64 x 256 output tile, 16 reduction depth
    // fed from shared memory (no register stall)
    // issued asynchronously; warp continues

On Hopper, wgmma supersedes the older mma.sync. A single wgmma instruction launches a matrix multiply that takes many cycles but does not stall the warp — the warp can issue memory ops while the tensor core grinds.

Throughput: each tensor core in Hopper does ~1024 FP16 FMAs per cycle (twice what an Ampere TC did). Across 528 tensor cores on GH200 at ~1.98 GHz, that is ~990 TFLOPS FP16 dense, ~1980 sparse. This is where the FLOPS are.

In inference: Tensor cores do the heavy matmuls — QKV projection, attention output, MLP up/down, LM head. That is roughly 90% of total FLOPs in a decode step. Everything else is either feeding them or cleaning up after them.

FP64 Core

Double-precision FMA. 2 per sub-partition, 8 per SM, 1056 per GH200. Half the width of the FP32 pipe, same two-operand model.

SASSOperation
DFMA R0, R2, R4, R6FP64 fused multiply-add
DADD, DMULFP64 add, multiply

In inference: unused. LLM inference never sees FP64. These exist for HPC workloads (CFD, molecular dynamics). For Lithos they are 1056 transistors per SM that sit idle — a tax paid to the Hopper die.

SFU — Special Function Unit

Transcendentals. Reciprocal, reciprocal square root, sine, cosine, exp2, log2. 16 per SM, 4 per sub-partition. Lower throughput than CUDA cores (usually 1 SFU op per 4 cycles per warp) but they compute functions that would take dozens of FMAs to approximate.

SASSOperation
MUFU.RCP1/x
MUFU.RSQ1/sqrt(x)
MUFU.SIN, MUFU.COSsin(x), cos(x)
MUFU.EX2, MUFU.LG22^x, log2(x)
MUFU.TANHtanh(x)

In inference: softmax uses MUFU.EX2 heavily (exp is synthesized as ex2 with ln2 scaling). RMSNorm and LayerNorm use MUFU.RSQ. SwiGLU uses sigmoid — also ex2 under the hood. RoPE uses MUFU.SIN/MUFU.COS once at kernel start then reuses. SFU throughput is usually a non-issue; the work is small.

LD/ST Unit

The address generator and data mover. 16 per SM. Issues one load or store per cycle per unit. Handles global loads (LDG), shared-memory loads (LDS), ldmatrix (a wide shared-memory read that delivers a matrix tile directly to registers), cp.async (asynchronous global-to-shared copy), and all the stores.

The LD/ST unit is where the memory system intersects the SM. It enqueues a request, the request goes through L1 -> L2 -> HBM, data comes back, the warp wakes up. While waiting, the warp scheduler runs other warps. This is how latency hiding works.

In inference: LD/ST is the busiest unit in a decode step. Every weight read, every KV cache read, every activation write goes through it. Memory-bound kernels saturate LD/ST well before they saturate tensor cores.

Lithos does not write FP64 kernels. Every matmul is FP16 (or FP8 for some projections), every reduction is FP32, every elementwise op runs on CUDA cores. The tensor cores set the speed of light; the other units get out of their way.
