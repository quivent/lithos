Inference Engines

Why each is fast and where each fails.

What Makes Inference Fast

LLM inference at batch=1 is a memory bandwidth problem. The GPU must read every weight once per token. The arithmetic is trivial — a few multiply-adds per weight byte. The question is: how close to peak bandwidth can you get?

Five factors determine inference speed, in order of impact.

01 Memory Bandwidth Utilization

Are you reading weights at the physical speed limit? An H100 SXM has 3.35 TB/s HBM bandwidth. A GH200 has 4.0 TB/s. At batch=1, a 27B model in 4-bit quantization is ~14 GB of weights. At full bandwidth that is 3.5 ms per token — 285 tok/s. In practice, most engines achieve 10–30% of peak bandwidth. Every percentage point of utilization is a direct percentage point of speed.

Why is utilization so low? Global memory has ~400 cycle latency. You need hundreds of outstanding memory requests to saturate the bus. This requires careful prefetching, double-buffering, and enough concurrent warps to hide latency.

02 Kernel Fusion

Are you eliminating redundant memory traffic? Every kernel boundary means: write results to HBM, synchronize, launch next kernel, read results back. A fused kernel keeps intermediates in registers (1 cycle) or shared memory (~20 cycles) instead of round-tripping through HBM (~400 cycles). At 64 transformer layers with 20–50 kernels each, eliminating boundaries is worth 30–50% throughput.

03 Quantization Efficiency

Are you using fewer bytes per weight without destroying quality? FP16 uses 2 bytes per weight. INT4 uses 0.5 bytes. But the dequantization math must be fast: if your dequant kernel can't feed the ALUs at line rate, you lose the bandwidth savings to compute overhead. The best quantization kernels interleave dequantization with the multiply-accumulate; poor implementations dequantize first, store intermediate results, then compute.

04 Launch Overhead

Is the GPU idle between kernels? Each CUDA kernel launch costs 3–10 µs of CPU-side dispatch. CUDA graphs amortize this by recording a sequence and replaying it, but graphs are rigid — they can't handle dynamic shapes without re-recording. The real win is not launching fewer kernels faster — it is launching fewer kernels, period.

05 Batching

Are you sharing weight reads across multiple requests? At batch=1, each weight byte produces one output element. At batch=8, each weight byte produces eight output elements — arithmetic intensity increases 8x, and you shift from memory-bound to compute-bound. Continuous batching dynamically packs requests so the GPU always has a full batch; PagedAttention solves the fragmentation problem that would otherwise waste 60–80% of KV cache memory.

Engine Deep Dives

Each entry covers origin, production use, architecture, optimizations, tradeoffs, measured numbers, and a minimal code example.

01 PyTorch + torch.compile

Meta AI / PyTorch Foundation. Core: Chintala, Paszke, Gross (2016); torch.compile landed in PyTorch 2.0, March 2023. The most actively developed ML framework; thousands of contributors.

Production use

The universal baseline. Every model on the HuggingFace Hub ships PyTorch code first; conversions to other engines happen after. HuggingFace transformers, diffusers, and accelerate sit on top. Used for inference at Meta (internal serving), Databricks, most academic labs. Rarely the production inference engine at scale — serious deployments convert to vLLM, TensorRT-LLM, or similar for the hot path.

Architecture in detail

Language. Python frontend, C++ backend (libtorch, ~800 MB shared library), CUDA/cuBLAS/cuDNN/Triton under that.

The ATen dispatcher. Every tensor op (x + y, x @ w) is a dynamic dispatch through ~15 stacked keys (Autograd, AMP, Functorch, ADInplaceOrView, Python mode, CUDA, CPU, BackendSelect…). Each op walks the key stack to find a registered kernel. Per-op cost in eager mode: 5–40 µs of CPU dispatch before any GPU work starts.

Autograd in the inference path. torch.no_grad() disables gradient accumulation but the autograd dispatch key is still on the stack; every tensor still carries grad_fn slots, version counters, and view metadata. torch.inference_mode() (1.9+) skips more, but the ATen tensor still has ~224 bytes of metadata.

Kernel sources. GEMMs go to cuBLASLt. Norms/activations go to cuDNN or hand-written ATen CUDA kernels. Attention routes through F.scaled_dot_product_attention, which dispatches to FlashAttention-2, memory-efficient attention, or a math fallback.

torch.compile / Inductor / Triton. Frontend is TorchDynamo, a CPython frame evaluator hook that symbolically traces Python bytecode into an FX graph. AOT Autograd canonicalises the graph. Inductor lowers it to a scheduler IR, fuses pointwise and reduction ops, and emits Triton source; Triton compiles to PTX and caches cubins under ~/.cache/torch/inductor/. Guards are generated for every traced shape/dtype; a guard miss triggers recompilation.

Memory. CUDACachingAllocator maintains free lists of GPU blocks, splits to fit, and defragments opportunistically. Adds 10–500 µs per allocation depending on fragmentation.

Request scheduling. None. PyTorch is a library; a serving layer (TorchServe, Ray Serve, BentoML) wraps model.generate().

Every optimization

  • torch.compile fusion. Inductor emits one Triton kernel per fused region. Compile cost: 30–60 s first run, 5–10 GB CPU RAM spike.
  • CUDA graphs via torch.cuda.graph(), used internally by mode="reduce-overhead".
  • SDPA backend selection (auto-routing to FlashAttention-2 / mem-efficient / math).
  • BF16/FP16 autocast and FP8 via Transformer Engine.
  • Nested tensors for variable-length batches.
  • Inductor coordinate-descent autotuner picks tile sizes per shape.

Tradeoffs

  • Dispatcher overhead. Even with compile, Python runs between compiled regions (graph breaks from data-dependent control flow, Python builtins, unsupported ops).
  • Compile warmup. 30–60 s on first call; cache is per shape/dtype/stride.
  • Memory footprint. libtorch ~800 MB + CUDA runtime ~200 MB + cuBLAS ~800 MB + cuDNN ~300 MB + Python ~30 MB ≈ ~2 GB before the model loads.
  • GIL-serialised outer loop. Sampling, tokenisation, stopping criteria all run under the GIL between GPU calls.
  • Not a serving engine. No batching, no paged KV, no scheduler.

Real measured numbers

HuggingFace optimum-benchmark on Llama-2-7B, A100-80GB, FP16 eager: ~52 tok/s; with torch.compile: ~88 tok/s. PyTorch blog "Accelerating Generative AI with PyTorch II" (Nov 2023) reports their hand-optimised gpt-fast repo at ~107 tok/s on Llama-7B INT8 on A100 — the effective ceiling before you bring in external engines.

Code surface

# pip install torch transformers
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

tok = AutoTokenizer.from_pretrained("meta-llama/Llama-3-8B")
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3-8B",
    torch_dtype=torch.bfloat16, device_map="cuda")
model = torch.compile(model, mode="reduce-overhead")

with torch.inference_mode():
    ids = tok("Hello", return_tensors="pt").input_ids.cuda()
    out = model.generate(ids, max_new_tokens=128, do_sample=False)
print(tok.decode(out[0]))
02 vLLM

UC Berkeley Sky Computing Lab. Woosuk Kwon, Zhuohan Li et al.; SOSP '23 paper "Efficient Memory Management for Large Language Model Serving with PagedAttention." First release June 2023; incubated under the PyTorch Foundation (Oct 2024). Extremely active.

Production use

The default open-source serving engine. Runs at Anyscale, Together AI, Mistral, Databricks, DeepInfra, Modal, countless self-hosters. Most HuggingFace Inference Endpoints that aren't TGI run vLLM. Supported by Red Hat / IBM as part of their enterprise AI platform.

Architecture in detail

Language. Python scheduler and executor, PyTorch for tensor ops, custom C++/CUDA extensions for PagedAttention and sampling, FlashAttention/FlashInfer/xFormers for attention.

PagedAttention — virtual memory for the KV cache. Classical KV allocation gives each sequence one contiguous buffer sized for max_seq_len; if requests rarely reach max length, 60–80% of KV memory is wasted to internal fragmentation. PagedAttention partitions each sequence's KV into fixed-size blocks (default 16 tokens per block for Llama-class heads). A per-sequence block table maps logical block index to physical block in a global pool — exactly analogous to an OS page table. The attention kernel reads the block table, gathers K/V tiles from the non-contiguous physical blocks, and runs attention over them. Two sequences sharing a prefix can point their block tables at the same physical blocks (copy-on-write for branching beam search or shared system prompts).

Continuous (in-flight) batching. Each iteration, the scheduler adds newly arrived requests to the running batch and evicts finished ones. Prefill and decode can be co-batched (chunked prefill, PR #3130+).

Inference loop. LLMEngine.step() → scheduler returns SequenceGroupMetadataList → worker builds input + block-table tensors → model forward (PyTorch graph, possibly torch.compile'd, possibly CUDA-graphed) → sampler runs on logits → scheduler updates. One Python tick per decode step.

Kernel sources. GEMMs via cuBLASLt (FP16/BF16) or Marlin/Machete/Exllama (INT4 GPTQ/AWQ). Attention via FlashAttention-2, FlashAttention-3 on Hopper, or FlashInfer. Custom CUDA kernels for PagedAttention's block-gather attention, rotary embedding, RMSNorm, and sampling (top-k, top-p, min-p, penalties fused).

Every optimization

  • PagedAttention brings KV utilisation from ~30% to ~96% per the SOSP paper.
  • Automatic prefix caching. Hashes KV blocks; identical block-sized prefixes share physical blocks. System-prompt-heavy workloads see 2–5x throughput jumps.
  • Chunked prefill. A 16k-token prefill is split and interleaved with decode so it doesn't stall all in-flight requests.
  • CUDA graphs for decode iterations, bucketed by batch size (powers of 2 up to 256).
  • Speculative decoding. Draft-target, Medusa-style multi-head, ngram, and MTP heads.
  • Quantization kernels. Marlin (INT4 GPTQ, ~1.8x FP16 GEMM throughput on Ampere/Hopper), Machete (Hopper W4A16 successor with TMA), AWQ, FP8 E4M3/E5M2 via CUTLASS, bitsandbytes NF4.
  • FlashInfer integration for Hopper WGMMA attention paths.
  • Tensor/pipeline parallelism via Ray or Python multiprocessing + NCCL.
  • Disaggregated prefill-decode experimentally in v0.6+.

Tradeoffs

  • Python in the scheduling loop. Per-iteration overhead of 1–10 ms has been measured in vLLM's own profiling.
  • Startup time. ~30–120 s cold (imports 3–5 s, model load 3–5 s, torch.compile 30–60 s, CUDA graph capture 10–90 s, KV memory profiling 10–60 s).
  • Memory overhead. ~4 GB of framework before the model loads; that memory could hold KV cache.
  • cuBLAS dependency. 800 MB binary; cannot fuse dequant with matmul in FP16 paths.
  • Complex configuration. Dozens of flags interact non-obviously.
  • Compile/graph fragility. Unusual attention variants often fall back to eager.

Real measured numbers

Our measurement on GH200 480GB, Qwen-class 27B W4 with MTP5: 179 tok/s single stream, 13% HBM bandwidth utilization (~0.52 TB/s of 4 TB/s peak), 274K KV tokens capacity. The SOSP paper reports 2–4x throughput over baseline HF transformers on OPT/Llama at server-scale batches. vLLM v0.6 blog (Sept 2024): ~2500 tok/s aggregate Llama-8B on 1xH100 at concurrency 128.

Code surface

# pip install vllm
from vllm import LLM, SamplingParams

llm = LLM(model="meta-llama/Llama-3-8B",
          gpu_memory_utilization=0.9,
          enable_prefix_caching=True)

params = SamplingParams(temperature=0.0, max_tokens=128)
for out in llm.generate(["Hello, world!"], params):
    print(out.outputs[0].text)

# or: vllm serve meta-llama/Llama-3-8B --port 8000
03 TensorRT-LLM

NVIDIA. First public release September 2023; successor to FasterTransformer (2019–2022). Active, biweekly tags on GitHub. Maintained by NVIDIA's Deep Learning Frameworks team.

Production use

Inside NVIDIA NIM microservices, Triton Inference Server backends at Meta and Amazon Bedrock, Perplexity's serving tier, Mistral's la Plateforme, NVIDIA DGX Cloud. When NVIDIA publishes "tok/s on H100," the number almost always came out of TensorRT-LLM. All NVIDIA MLPerf Inference submissions use it.

Architecture in detail

Language split. Python build phase (model definition via a PyTorch-like module system in tensorrt_llm.models) lowers to a TensorRT INetworkDefinition, then to a serialised engine .plan file. Runtime is C++ with thin Python bindings; the hot path has no Python.

Kernel sources. A tall stack: cuBLASLt for some GEMMs, cuDNN for norm/activation, CUTLASS templates instantiated at build time for fused GEMM+epilogue, NVIDIA's XQA (eXtended Query Attention) for decode-phase attention, and a hand-written FMHA kernel derived from FlashAttention-2 but rewritten in raw CUDA/PTX for Hopper's TMA and WGMMA instructions.

Build-phase kernel selection. Given a shape and dtype, the builder's tactic selector times every candidate kernel (tile size, split-K, epilogue variant) on the actual target GPU and picks the fastest. This is why rebuilds are slow and why an engine built on H100 refuses to run on A100.

Serialised engine format. A single binary blob (.engine/.plan) containing the tactic-selected kernel catalog (embedded cubins), a frozen execution graph, tensor memory layout, and weight data. Load is mmap + pointer fixup.

Inference loop. The C++ runtime (gptManager, now executor) holds the deserialised engine, a paged KV cache manager (adopted in v0.7+), and an inflight batching scheduler. Each iteration: scheduler packs prefill + decode, fires the engine via enqueue() on a CUDA stream, drains completed tokens.

Memory. Custom pool allocated up front to a user-configured fraction of GPU memory. KV blocks (default 64 tokens each) pooled separately. Weights in a single buffer sized at build time.

Every optimization

  • XQA. Decode-phase attention specialised for batched single-token queries with long KV. Warp-specialised TMA loads on Hopper (producer warps do TMA, consumer warps do WGMMA), hiding HBM latency behind async copies.
  • FP8 E4M3 GEMMs. On Hopper, wgmma.mma_async.sync.aligned.m64n256k32.f32.e4m3.e4m3.f32 runs at ~1979 TFLOPS on H100 SXM. Per-tensor, per-channel, and per-block FP8 recipes via NVIDIA's Transformer Engine.
  • Build-time layer fusion. QKV projection + RoPE, gate + up + SiLU for SwiGLU, RMSNorm + residual-add — fused via CUTLASS epilogues.
  • In-flight continuous batching in C++ with zero Python on the scheduling loop.
  • Speculative decoding. Medusa, EAGLE, ReDrafter, Lookahead, and vanilla draft-target, with verification fused into the main forward pass.
  • Multi-GPU. Tensor parallel via NCCL (SHARP-aware on NVSwitch), pipeline parallel with 1F1B micro-batching, expert parallel for MoE.
  • CUDA graphs for decode-phase iterations keyed on batch size.

Tradeoffs

  • 5–30 minute build phase, per model, per GPU arch, per dtype, per max batch/seq combination.
  • Closed-source runtime. C++ runtime ships as a static library; you can read the Python builder but not the executor. No way to patch a bug without NVIDIA's next release.
  • Shape rigidity. Max batch and max sequence length compiled in. Small requests pad up.
  • Version matrix fragility. Exact TensorRT/CUDA/driver combos required.
  • NVIDIA-only. By design.

Real measured numbers

NVIDIA MLPerf Inference v4.1 (Aug 2024, Llama2-70B Server, 1xH100 SXM, FP8): ~24,500 tok/s aggregate at high concurrency. Single-stream Llama3-8B on H100 FP8: ~280 tok/s. Llama3-70B single-user on H100 SXM FP8: ~95 tok/s per NVIDIA's developer blog. (mlcommons.org; NVIDIA Developer Blog, 2024.)

Code surface

# Runtime

print(tokenizer.decode(out[0]))
04 llama.cpp

Georgi Gerganov, March 2023. Started as a weekend port of Llama to C++ with no dependencies; grew into the canonical on-device LLM runtime. Maintainer-led, 800+ contributors. Tied to the ggml tensor library.

Production use

Runs inside Ollama, LM Studio, Jan, GPT4All, Faraday, and the llama-cpp-python bindings that power a huge share of edge deployments. Obsidian plugins, VS Code extensions, local coding assistants. Production use at cloud scale is rare; production use on one machine is enormous.

Architecture in detail

Language. Pure C/C++17 core (ggml.c, llama.cpp), zero external dependencies in the base build. Optional CUDA / HIP / Metal / Vulkan / SYCL / OpenCL backends compiled in.

GGUF format. Single-file container holding model metadata (as a key-value blob) and tensor data laid out for direct memory-mapped consumption. Tensors are stored in the exact quantisation block layout the kernel consumes — no runtime re-packing. Supersedes GGML and GGJT (May 2023).

Quantisation formats. Q8_0 (8-bit, scale per 32-element block), Q4_0/Q4_1 (legacy 4-bit), and the K-quant family: Q2_K, Q3_K_S/M/L, Q4_K_S/M, Q5_K_S/M, Q6_K. K-quants use a two-level scale structure: 16-element sub-blocks each with their own 6-bit scale, grouped into 256-element super-blocks with a shared FP16 scale. Q4_K_M averages ~4.85 bits/weight and is the common sweet spot. IQ-quants use k-means codebooks for sub-3-bit formats.

mmap-based loading. mmap the GGUF file and hand the kernels pointers into it. Zero-copy, near-instant startup; the OS pages weights in lazily on first access. Arguably llama.cpp's most copied idea — Lithos, MLX, and others all do variants of it.

Inference loop. llama_decode() takes a batch of tokens, runs the ggml graph (topologically sorted op list), dispatches each op to the active backend. CUDA backend has hand-written kernels for mul_mat_q (quantised GEMM), rope, rms_norm, soft_max, and fused attention.

Memory. Custom bump allocator (ggml_allocator) for activations; KV cache a single preallocated tensor of shape [n_ctx, n_layer, 2, n_head, d_head]. Paged KV added 2024 but less sophisticated than vLLM's.

Scheduling. Multi-sequence batching via llama_batch; the server example implements continuous batching but is less mature than vLLM.

Every optimization

  • Fused dequant-GEMM CUDA kernels (mul_mat_q*.cu) — one per quant format, dequant interleaved with FMA. No intermediate FP16 buffer.
  • MMQ kernels on Turing+ use tensor cores for Q4_0/Q8_0 by dequantising into shared memory as INT8 tiles and running mma.sync. K-quants largely miss tensor cores.
  • Flash Attention port (Gaessler) activated via -fa; custom CUDA with online softmax.
  • Speculative decoding via --draft.
  • GPU layer offload (-ngl): run N layers on GPU, rest on CPU.
  • BLAS fallback via cuBLAS for large FP16 GEMMs at long prompts.
  • Metal backend (pure MSL) on every Mac; Vulkan for AMD/Intel; SYCL for Intel GPUs.
  • Grammar / JSON-schema constrained decoding built in.

Tradeoffs

  • Tensor cores underused for K-quant formats — peak FLOPS left on the table on Hopper/Ampere.
  • Generic tile sizes. Same kernel shapes run on RTX 3060 and H100; no per-GPU tactic search.
  • Single-maintainer bandwidth. Kernels that exist are excellent; kernels that don't (Hopper WGMMA paths) haven't been prioritised.
  • Server mode less mature than vLLM / TensorRT-LLM for high-concurrency serving.
  • Quality drift in aggressive K-quants. Q2_K and IQ2 noticeably degrade model quality.

Real measured numbers

Gerganov's H100 PCIe benchmarks (llama.cpp discussion #4167): Llama-7B Q4_0 ~150 tok/s, Q4_K_M ~130 tok/s. RTX 4090 Llama-7B Q4_K_M: ~135 tok/s. M2 Ultra (Metal) Llama-70B Q4_K_M: ~11 tok/s.

Code surface

# Build: make GGML_CUDA=1

    -p "Hello, world!" -n 128 -ngl 99

# Python bindings:

05 MLX

Apple Machine Learning Research. First released December 2023. Awni Hannun, Jagrit Digani, Angelos Katharopoulos et al. Actively developed; the closest architectural peer to Lithos.

Production use

Every "run LLMs on Mac" app worth its salt uses MLX or llama.cpp. mlx-lm is the reference path for Apple Silicon. LM Studio's Apple Silicon backend, Ollama's Metal path (partly), and a wave of on-device apps for journaling, coding, summarisation run MLX models. Apple uses it internally for on-device model R&D.

Architecture in detail

Language. C++ core, Python and Swift frontends. The Python API is NumPy-shaped with a PyTorch-ish nn.Module layer.

Unified memory. Apple Silicon's unified memory architecture means CPU and GPU share the same physical RAM over a shared memory controller. MLX arrays live in this shared pool; there is no .to("mps") equivalent — the array is accessible from both.

Lazy evaluation. Every op (mx.matmul(a, b)) returns a placeholder that records the computation without executing. Evaluation happens at mx.eval(), at array.item(), or implicitly when you print. The graph accumulated in between is the fusion opportunity: MLX's compiler walks it and fuses pointwise chains before dispatching.

mx.compile. Takes a Python function returning MLX arrays and produces a compiled function: the traced graph is cached, shape-specialised, and the generated Metal shaders are reused across calls. This is MLX's answer to torch.compile, native rather than bolt-on.

Kernel source. Metal Shading Language (MSL) shaders, JIT-compiled by Apple's Metal compiler. GEMMs go to MLX's own steel kernels (template-metaprogrammed MSL using simdgroup_matrix on M3+ for tensor-core-equivalent throughput). Attention has a custom fused SDPA kernel.

Inference loop. For mlx-lm: tokenise → run the model function (lazy graph) → mx.eval on the logits → sample → loop. KV cache is a list of mx.arrays updated in-place.

Every optimization

  • Zero-copy weight loading. Safetensors mmapped; mx.array points into the mapping directly. Model "load" is milliseconds.
  • Lazy + compile fusion. Pointwise chains fuse into one dispatch by construction.
  • simdgroup_matrix on M3/M4 for 16x16 half-precision MMA at tensor-core-equivalent throughput.
  • Quantisation kernels for 2/3/4/6/8-bit group-quantised weights; dequant fused into matmul.
  • Distributed via MPI for multi-Mac inference (Llama-3-405B across three M2 Ultras, 2024).
  • Graph-level CSE and constant folding during the compile pass.

Tradeoffs

  • Apple-only. No path to NVIDIA / AMD / Intel GPUs.
  • Bandwidth ceiling. M4 Max ~546 GB/s, M2 Ultra 800 GB/s, M3 Ultra 819 GB/s — vs 4 TB/s on GH200.
  • Opaque Metal compiler. No SASS-equivalent disassembly; you cannot inspect the generated GPU ISA.
  • No instruction-level control. No inline assembly equivalent of PTX; simdgroup_matrix is the floor.
  • Smaller ecosystem. Fewer quantisation formats than llama.cpp, no vLLM-class server, limited speculative decoding.

Real measured numbers

Published mlx-lm benchmarks: Llama-3-8B Q4 on M2 Ultra ~69 tok/s; M3 Max ~85 tok/s; M4 Max ~95 tok/s. Our Qwen-27B Q4 with adaptive MTP chain on M4 Max 128 GB: 51.1 tok/s. Llama-70B Q4 on M2 Ultra: ~12 tok/s.

How it compares to Lithos

Architecturally the closest peer: both reject framework overhead, both build on unified memory, both are language + engine in one. Diverges on (1) hardware — Apple Silicon vs. Hopper; (2) compiler transparency — opaque Metal vs. inspectable SASS; (3) instruction control — simdgroup_matrix floor for MLX vs. Lithos emitting SASS directly; (4) scope — MLX includes autograd and training, Lithos is inference-only.

Code surface

text = generate(model, tokenizer,
                prompt="Hello, world!",
                max_tokens=128, verbose=True)
print(text)
06 SGLang

UC Berkeley, Stanford, CMU, xAI collaboration. Lianmin Zheng et al.; NeurIPS '24 paper "SGLang: Efficient Execution of Structured Language Model Programs." First release January 2024. Very active; runs production at xAI for Grok serving.

Production use

xAI serves Grok on SGLang. ByteDance, LMSYS, and several Chinese clouds run it. HuggingFace TGI has adopted chunks of its scheduler ideas. Growing fast in 2024–2026 as the serious alternative to vLLM for high-throughput serving.

Architecture in detail

Language. Python runtime, PyTorch, FlashInfer for attention, Triton for several custom kernels. Same foundation as vLLM — differentiation is above the kernel layer.

RadixAttention — the key idea. Instead of vLLM's hash-keyed block cache, SGLang maintains the KV cache as a radix tree keyed on token prefixes. Each node represents a token sequence; edges share common prefixes. When a new request arrives, the scheduler walks the tree from root matching the prompt tokens; every matched node's KV blocks are reused. Branches (beam search, parallel sampling, multi-turn chat rewrites) all live in the tree naturally. LRU eviction at leaves keeps the tree within memory budget. Result: the common-prefix case ("you are a helpful assistant") has 100% reuse without any explicit toggle.

Structured generation via compressed FSM. For JSON-schema or regex-constrained output, SGLang compiles the grammar to a finite-state machine, then compresses runs of tokens that can only transition one way into a single jump — so "generate a 16-character UUID" emits the fixed prefix in one forward pass rather than token-by-token. Outlines-style mask-based constrained decoding is also supported but the compressed-FSM path is the novel contribution.

Zero-overhead scheduler. v0.3+ batches the scheduler's own Python work so one forward-pass tick overlaps with CPU scheduling for the next tick — effectively hiding the Python cost.

Inference loop. TokenizerManager → Scheduler (radix tree, batch assembly) → TpModelWorker (PyTorch forward, FlashInfer attention) → DetokenizerManager. Prefill and decode mix in one batch.

Every optimization

  • RadixAttention prefix reuse without configuration.
  • Compressed FSM for structured generation.
  • Overlap scheduler (zero Python overhead in steady state).
  • FlashInfer attention — variable-length batches, GQA-aware, paged KV, Hopper FA-3 paths.
  • Continuous batching with chunked prefill.
  • Speculative decoding (EAGLE, Medusa) first-class.
  • Quantisation via Marlin / AWQ / FP8 shared with the ecosystem.
  • Multi-LoRA with batched LoRA matmuls.
  • Tensor / data / expert parallelism across GPUs.

Tradeoffs

  • Same PyTorch/cuBLAS floor as vLLM.
  • Startup time similar to vLLM (~30–120 s cold).
  • Radix-tree metadata overhead small but non-zero; adversarial prompt patterns can fragment.
  • Thinner docs than vLLM; fewer quantisation backends out of the box.
  • DSL incomplete. Original SGLang frontend is still evolving; most users just use the OpenAI-compatible server.

Real measured numbers

LMSYS benchmarks (Sept 2024): Llama-3-70B on 8xH100 SXM at concurrency 128 — SGLang v0.3 ~5,200 tok/s vs. vLLM v0.5.4 ~3,900 tok/s, a 30–40% edge. Single-stream decode comparable: ~170–190 tok/s on H100 for 27B quantised.

Code surface

import sglang as sgl
sgl.set_default_backend(sgl.RuntimeEndpoint("http://localhost:30000"))

@sgl.function

07 ExLlamaV2

turboderp. ExLlama v1 mid-2023; ExLlamaV2 with the EXL2 format landed September 2023. A one-person project with periodic contributors.

Production use

Default backend in Oobabooga's text-generation-webui for GPTQ/EXL2 weights; widely used by self-hosters on single-GPU consumer hardware (RTX 3090, 4090, 5090). TabbyAPI wraps it with an OpenAI-compatible server. Not used at data-centre scale.

Architecture in detail

Language. Python frontend, custom CUDA C++ extensions for every performance-critical kernel. Compiled against PyTorch for tensor interop but bypasses cuBLAS entirely.

EXL2 quantization format — per-layer bit allocation. GPTQ/AWQ quantise every weight to the same bit depth. EXL2 instead runs a calibration pass, measures per-layer / per-row reconstruction error at 2/3/4/5/6/8-bit candidates, and solves an integer program: "minimise total bits subject to a target perplexity increase." The result is a mixed-precision file where sensitive layers (first and last MLPs, certain attention heads) carry 6 or 8 bits while tolerant layers go down to 2.5 or 3. Typical average 3.5–4.5 bits/weight for quality comparable to uniform 4-bit.

Kernel sources. Entirely custom. q_gemm.cu, q_attn.cu, q_mlp.cu: hand-written CUDA for mixed-precision dequant + GEMM with the EXL2 header-per-row layout. Attention uses FlashAttention-2 via upstream's C++ interface.

Inference loop. ExLlamaV2DynamicGenerator manages a paged KV cache (2024) and runs continuous batching in Python; the hot path is a single custom forward() calling the CUDA extensions.

Every optimization

  • Per-layer bit allocation (above).
  • Hand-written mixed-precision GEMM fused with dequant; shared-memory double-buffering; tensor cores via mma.sync.
  • Custom rotary, RMSNorm, SwiGLU kernels — no cuBLAS in the hot path.
  • Speculative decoding with draft models.
  • Dynamic batching via the generator class.
  • CFG and advanced sampling (min-P, quadratic) built in.
  • Q4 KV cache for long-context memory savings.

Tradeoffs

  • Single-GPU only. No tensor parallel.
  • EXL2 format is niche. Cannot load GGUF, AWQ, or standard GPTQ weights directly.
  • Small maintainer bandwidth. New architectures land weeks after vLLM.
  • Batching is afterthought. Single-user-first.
  • No FP8 / no Hopper-native kernels — tuned for Ampere/Ada.

Real measured numbers

turboderp's published benchmarks: Llama-3-70B EXL2 4.0bpw on RTX 4090: ~35 tok/s. Llama-3-8B EXL2 6.0bpw on RTX 4090: ~135 tok/s. RTX 3090 8B: ~100 tok/s.

08 MLC-LLM / Apache TVM

Apache TVM founded by Tianqi Chen (SAMPL, U. Washington, 2017). MLC-LLM built on TVM Unity, released mid-2023, led by CMU Catalyst and OctoAI (now part of NVIDIA). Active but slower cadence than vLLM/SGLang.

Production use

The go-to path for weird targets: WebLLM (Llama running in a browser tab via WebGPU, used by Chatbot Arena's in-browser demos), Android and iOS (MLC Chat app), AMD RX 7900 XTX (ROCm), Intel Arc (Vulkan). Rarely chosen on pure NVIDIA where vLLM / TensorRT-LLM dominate.

Architecture in detail

Language. Python frontend (model definition via tvm.relax or PyTorch import), TVM's Relax IR, TIR (tensor IR), and the MetaSchedule auto-scheduler. Emits target-specific code: PTX (via nvcc), SPIR-V (Vulkan/WebGPU), Metal (MSL), OpenCL, HIP.

Compilation pipeline. Relax graph → operator-level fusion passes → TIR lowering → MetaSchedule auto-tuning (search over tile sizes, loop orderings, vectorisation, software-pipelining decisions with an XGBoost-based cost model) → target code generation → runtime library.

Target-agnostic auto-scheduling. The scheduler profiles candidates on the actual target device; for CUDA this means nvcc + ptxas + measurement loops. Full tuning for one model can take hours.

Runtime. Minimal C++ (tvm_runtime) that loads a compiled module (.so/.wasm/.metallib) and executes fused kernel calls. WebLLM ships this runtime as WebAssembly alongside WebGPU shaders.

Every optimization

  • MetaSchedule auto-tuning for per-shape kernel specialisation.
  • Fusion passes at Relax and TIR levels.
  • Quantisation (AWQ, GPTQ, group-quant) lowered to target-specific kernels.
  • Cross-platform fast attention on every backend.
  • Speculative decoding in MLC-LLM proper.
  • WebGPU code generation — LLMs in a browser are a real feature, not a demo.

Tradeoffs

  • Auto-scheduling rarely beats hand-written kernels on flagship NVIDIA. CUTLASS / FlashAttention authors hand-tune with knowledge the cost model lacks.
  • Long compile times. Minutes to hours for full tuning.
  • Toolchain complexity. TVM, LLVM, target compilers, Python bindings — version skew breaks builds.
  • Thinner serving story than vLLM/SGLang.
  • Breadth vs. depth. Reasonable performance everywhere rather than peak performance somewhere.

Real measured numbers

MLC-LLM blog (Oct 2023) on RTX 4090 Llama-2-7B Q4: ~165 tok/s (tied with ExLlama at the time). WebLLM Llama-3-8B Q4 in Chrome on M3 Max: ~25–30 tok/s. AMD RX 7900 XTX Llama-3-8B Q4: ~60–75 tok/s — significantly faster than llama.cpp's ROCm path.

10 NVIDIA Triton Inference Server

NVIDIA, 2018. A generic inference server that hosts backends rather than implementing inference itself. Not to be confused with OpenAI's Triton DSL.

Production use

Core of NVIDIA NIM, used at Meta, Amazon, Microsoft, and essentially every NVIDIA-stack enterprise deployment. Not a competitor to vLLM/TensorRT-LLM — it hosts them as backends, adding HTTP/gRPC APIs, dynamic batching across models, multi-model ensembles, metrics, health checks.

Architecture in brief

C++ server with pluggable backends: tensorrtllm_backend, vllm_backend, python_backend, ONNX Runtime, PyTorch, TensorFlow. Dynamic batcher, sequence batcher, and request scheduler run in-process. Model ensembles compose multiple backends into a DAG (useful for RAG + reranker + generator pipelines).

Complex for one-model serving; its power is in heterogeneous or high-scale deployments. Not an engine itself — included here only so you don't mistake it for one.

11 Lithos

This project. A Forth-derived language that happens to host an inference engine. Framework and engine, like MLX.

Lithos is a language first. core.fs (255 lines) and patterns.fs (1,204 lines) define a Forth vocabulary whose words emit SASS. The Python in src/ is a temporary host runtime until the language self-hosts.

Why it will be fast

  • Direct SASS emission. No compiler chain between the programmer and the GPU. Forth words emit binary SASS instructions directly into cubin ELF files. Every scheduling decision is explicit.
  • Fused mega-kernels. One kernel per transformer layer. RMSNorm, attention, GEMV, activation, residual — all fused into a single launch. No HBM round-trips between ops.
  • mmap weight loading on unified memory (GH200). mmap sets up virtual address mappings in ~9 ms with zero weight bytes in memory. PyTorch's 3–5 s "load" copies all 18 GB into RAM/HBM before returning. On GH200's unified memory, the GPU reads resident pages directly at 900 GB/s (NVLink-C2C) or 4 TB/s (HBM3e) after hardware page migration.
  • No framework, no Python in the inference path. Runtime is Rust calling the CUDA driver API. Launch overhead: 2.3 µs (Lithos) vs 4.48 µs (cuBLAS).
  • DeltaNet-native kernels. Optimised for DeltaNet's access patterns from the start.
  • 32 KB of cubins replacing 800 MB of cuBLAS — 25,000x reduction. Cubin load: 0.092 ms for all 7 kernels.

Architecture

Two layers: language and engine. The language is Forth: core.fs + patterns.fs (1,459 lines) define words that emit SASS via emit-sass.fs. The engine is the host runtime that invokes the Forth, builds the cubin ELF, loads it via cuModuleLoadData, and dispatches with cuLaunchKernel. No ptxas in the pipeline.

Tradeoffs

  • Not yet running full inference. Proof of life passing; kernels compile; engine skeleton works. End-to-end model inference not yet demonstrated.
  • Extreme programming model. Writing GPU code in a Forth dialect is not accessible to most developers.
  • Single-hardware target initially. Optimised for GH200 (SM 9.0).

Targets

>400 tok/s single stream (vs vLLM's measured 179 tok/s — 2.2x). <2 s startup. >70% bandwidth utilization at batch=1, with measured 3.59 TB/s (90%) achievable on HBM3e.

Component Libraries (Not Engines)

These are building blocks, not engines. An inference engine orchestrates model loading, memory management, request scheduling, and token generation. These libraries provide individual kernels that engines call. Confusing the two leads to category errors — FlashAttention does not "compete with" vLLM; vLLM uses FlashAttention.

FlashAttention / FlashAttention-2/3 / FlashInfer

Tri Dao (Princeton, later TogetherAI). FlashAttention: NeurIPS 2022. FlashAttention-2: July 2023. FlashAttention-3: July 2024 (with Colfax Research and NVIDIA). FlashInfer: Zihao Ye et al., Sept 2024.

The algorithm — tiled attention with online softmax

Naive attention materialises an N×N matrix S = QKT/sqrt(d), softmaxes it, then multiplies by V. At N=16k that matrix is 256M entries × 4 bytes = 1 GB, written to HBM and read back — disastrous for bandwidth.

FlashAttention avoids materialising S by tiling. Load a block of Q rows into SRAM (shared memory). Stream blocks of K and V through SRAM one tile at a time; for each tile, compute partial QKT, apply an online softmax update that maintains running max and denominator, and accumulate weighted V into the output tile. The output row is complete when all K/V tiles have streamed through. Result: O(N) HBM reads/writes instead of O(N²); the attention matrix never leaves shared memory.

Online softmax is the key math trick. For two partial sums s1 = Σ exp(xi - m1) and s2 = Σ exp(xj - m2) with maxima m1, m2, the combined sum with new max m = max(m1, m2) is em1-m·s1 + em2-m·s2. This lets you process the sequence in blocks and combine results without ever holding the full S matrix.

Versions and what changed

  • FlashAttention (2022). Original tiled algorithm; CUDA/Turing/Ampere.
  • FlashAttention-2 (2023). Rewrites the kernel to parallelise across sequence length, reduces non-matmul FLOPs, better work partitioning. ~2x faster than v1 on A100.
  • FlashAttention-3 (2024). Hopper-specific. Uses TMA (Tensor Memory Accelerator) for async bulk loads, warp-specialisation (producer warps do TMA, consumer warps do WGMMA), and FP8. Claims 1.5–2.0x over FA-2 on H100.
  • FlashInfer (2024). Not a new algorithm; a kernel suite with variable-length batching, paged KV, GQA-awareness, append-only prefill, and prefix-sharing prefill — the operations modern servers actually need. Used by SGLang and optionally by vLLM.

Build fragility

FlashAttention kernels are hand-written CUDA with heavy template instantiation. Compilation takes 10–30 minutes and several GB of RAM; the prebuilt wheels on PyPI match only specific CUDA + PyTorch + Python + arch combinations (e.g. flash-attn-2.5.8+cu122torch2.3cxx11abiFALSE-cp311-cp311-linux_x86_64.whl). A mismatched wheel silently imports but crashes at first forward with an opaque symbol error.

Used by

vLLM (default), SGLang (via FlashInfer), TensorRT-LLM (FA-2-derived FMHA), PyTorch's scaled_dot_product_attention, HuggingFace TGI, llama.cpp (Gaessler port). Not an engine — no model loading, no scheduling, no serving. Just the attention kernel.

cuBLAS / cuBLASLt

NVIDIA. cuBLAS since CUDA 1.0 (2007); cuBLASLt since CUDA 10.1 (2019). Closed source, ships with the CUDA Toolkit.

The atlas of pre-compiled kernels

cuBLAS is a library of dense matrix-multiply kernels. The ~800 MB shared object (libcublas.so + libcublasLt.so) contains thousands of pre-compiled SASS variants covering the cross product of: GPU architecture (SM 7.0/7.5/8.0/8.6/8.9/9.0), input dtype (FP32, FP16, BF16, FP8 E4M3/E5M2, INT8), output dtype, accumulator dtype, tile shape (BM × BN × BK, dozens of choices), split-K or not, stride patterns, alpha/beta configurations. At call time, a heuristic (or cuBLASLt's tactic search if you opt in) picks a kernel from this atlas.

What cuBLAS cannot do

  • No INT4 / no sub-byte dtypes. You can do INT8 GEMM; for W4A16 you need Marlin, Machete, ExLlama kernels, or TensorRT-LLM.
  • No custom epilogue fusion. cuBLASLt supports a fixed menu of epilogues (RELU, GELU, bias-add, bias-add+RELU, dRELU); arbitrary pointwise fusion is not expressible.
  • Closed source. You cannot fix a bug, change a tile shape, or learn from the implementation.
  • Per-call overhead. ~4.5 µs per GEMM launch (heuristic pick + library state + kernel launch).

Used by

Essentially every engine except llama.cpp and Lithos. PyTorch, TensorRT-LLM, vLLM, SGLang, MLC-LLM, ExLlamaV2 (FP16 only). The 800 MB payload is the single largest contributor to framework-engine memory footprint and startup time.

cuDNN

NVIDIA. cuDNN v1: 2014. Current v9.x. Closed source. ~400 MB.

Deep learning primitives

Kernels for convolution (forward/backward/dgrad/wgrad), pooling, batch-norm / layer-norm / RMSNorm, activation (ReLU, GELU, SiLU…), RNN cells, and fused attention as of cuDNN 9 (Feb 2024 — a FlashAttention-equivalent kernel via the cuDNN graph API). cuDNN 9's frontend API lets you build an operation graph (matmul + bias + activation + cast) and cuDNN compiles it to a single fused kernel, competing with hand-written Triton.

Used by

PyTorch (primary consumer; every nn.Conv2d, many norms, some attention paths), TensorRT, TensorRT-LLM. Transformer-dominated workloads lean on it less than CNN workloads did, but it is still loaded because PyTorch imports it.

Tradeoffs

Same shape as cuBLAS: large closed binary, no sub-byte quantisation, rigid epilogue menu. The cuDNN 9 graph API improves supported patterns but fusion is limited to what NVIDIA has pre-canonicalised.

Triton (OpenAI)

Philippe Tillet (OpenAI), MAPL 2019 paper; OpenAI Triton project 2021+. Open source. Not the same as NVIDIA Triton Inference Server.

A GPU DSL embedded in Python

Triton is a Python-embedded DSL and compiler for writing GPU kernels. You write at block granularity rather than per-thread: tl.load(ptr + offsets) loads a tile, tl.dot(a, b) is a block matmul, tl.store writes back. The compiler handles: scheduling threads within the block, vectorising memory accesses, picking tile layouts for tensor cores, shared-memory allocation and bank-conflict avoidance, software pipelining of loads vs. computation, and lowering to PTX.

Pipeline: Triton IR (your Python traced) → Triton-GPU IR → LLVM IR → PTX → ptxas → cubin. Tile shapes, num_warps, num_stages tunable via @triton.autotune.

How torch.compile uses it

TorchInductor emits Triton source strings for fused pointwise/reduction regions. The first call to a compiled function triggers Triton compilation (seconds per kernel); cubins cache under ~/.cache/torch/inductor/. For any non-GEMM work in torch.compile, you are running Triton kernels.

Other users

FlashAttention has reference Triton implementations (flash_attn.triton), as do many community fused kernels (RMSNorm+residual, fused RoPE, liger-kernel's entropy loss). vLLM and SGLang ship several Triton kernels for sampling, custom attention variants, and quant ops.

The compile-time tax

  • Per-kernel JIT cost. First call compiles; 1–10 s per kernel, minutes cumulatively for a full model.
  • Autotune cost. triton.autotune runs every candidate config on warmup — tens of seconds.
  • Cache fragility. Keyed on source hash + shapes + dtypes; driver or Triton upgrades invalidate.
  • Not a full kernel suite. Triton generates kernels; cuBLAS still handles most GEMMs in PyTorch.

Code shape

import triton
import triton.language as tl

@triton.jit
def rmsnorm_kernel(x_ptr, w_ptr, y_ptr, N,
                   BLOCK: tl.constexpr, eps: tl.constexpr):
    row = tl.program_id(0)
    cols = tl.arange(0, BLOCK)
    mask = cols < N
    x = tl.load(x_ptr + row*N + cols, mask=mask, other=0.0)
    rms = tl.sqrt(tl.sum(x*x, axis=0) / N + eps)
    w = tl.load(w_ptr + cols, mask=mask, other=0.0)
    tl.store(y_ptr + row*N + cols, (x / rms) * w, mask=mask)

The pattern: engines use these libraries to build their compute path. Lithos bypasses all of them — its ~32 KB of cubins replace cuBLAS's 800 MB, its fused mega-kernels replace the attention library, and its SASS-direct emission replaces the Triton compiler. The tradeoff is that the optimisation breadth of NVIDIA's thousands of cuBLAS variants must be replaced by a small number of hand-tuned kernels for the specific shapes Lithos cares about.

What the table reveals

The engines cluster into four tiers by architectural approach.

The baseline (PyTorch + torch.compile) is what every model ships with. Ceiling: ~150 tok/s at batch=1 on H100. Every other tier exists because this ceiling is too low for production.

Framework engines (vLLM, SGLang, DeepSpeed) build on PyTorch and cuBLAS. They optimise at the scheduling layer — batching, caching, paged KV, radix trees — but inherit the kernel granularity and dispatch overhead of their foundation. Ceiling: ~200 tok/s at batch=1.

Native engines (llama.cpp, ExLlamaV2, TensorRT-LLM, MLX) write their own kernels. They control the compute path but still launch many kernels per layer. Ceiling: ~300 tok/s at batch=1.

Direct-to-silicon (Lithos) eliminates the kernel boundary entirely. One fused kernel per layer, with the programmer controlling instruction scheduling. The only approach that can theoretically reach >70% bandwidth utilisation at batch=1.

Where Lithos stands today (measured, April 2025)

First end-to-end run: ~2 tok/s on GH200, 90× slower than vLLM's 179 tok/s. Output is correct — "Paris" rank 1, logit 18.057, all 64 layers verified layer-by-layer (cosine 1.0). The gap is entirely kernel optimization: our projection kernel does not yet use tensor cores for the batch=1 GEMV. vLLM's cuBLAS runs at 3460 GB/s (86% peak); our kernel does not come close. The architecture is proven. The kernels need tensor core tiling, WGMMA instructions, and TMA async loads. This is the next phase of work.
