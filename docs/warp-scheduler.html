Warp Scheduler

How the GPU picks which warp to run each cycle — and how Lithos influences the decision

What the Warp Scheduler Does

Each Streaming Multiprocessor (SM) on Hopper has 4 warp schedulers, one per sub-partition. Every clock cycle — one tick of the 1.83 GHz SM clock, about 0.55 nanoseconds — each scheduler picks one ready warp from its assigned set and issues one instruction from it.

A warp is "ready" when two conditions are met: all source operands for its next instruction are available (no pending writes), and the functional unit the instruction needs (FMA, load/store, special function) is free.

With 4 schedulers issuing independently, an SM can issue up to 4 instructions per cycle — one from each sub-partition. The schedulers do not coordinate with each other. Each operates on its own pool of resident warps.

How It Decides

Scoreboard Tracking

Each register has a pending-write counter maintained by the hardware scoreboard. When an instruction writes to a register, the counter increments. When the write completes, it decrements. An instruction is eligible for issue only when all of its source registers have a pending-write count of zero.

Warp Selection Policy

Among eligible warps, the scheduler uses a selection policy to pick one. NVIDIA does not publish the exact algorithm. It is believed to be a form of greedy-then-round-robin: prefer warps that can issue to an idle functional unit, break ties with round-robin to ensure fairness.

The Failure Case: Stall

If no warp in a sub-partition is eligible — every warp is waiting on a memory load, a barrier, or a long-latency operation — the scheduler stalls. No instruction issues that cycle. This is the latency hiding failure case. It means there aren't enough independent warps to cover the latency of in-flight operations.

What Lithos Can Influence

1. Occupancy (Number of Resident Warps)

More resident warps give each scheduler more choices, improving the odds that at least one warp is ready every cycle. Occupancy is determined by register count per thread: fewer registers per thread means more threads (and therefore more warps) fit on the SM simultaneously. Lithos controls register count directly via .maxnreg declarations in PTX, rather than relying on compiler heuristics.

2. Instruction Mix

Interleaving memory loads with arithmetic gives the scheduler something to issue while loads are in flight. If a kernel issues 10 loads in a row, all warps quickly become blocked waiting on memory. Lithos patterns explicitly alternate ld.global with fma sequences — load a value, compute on a previous value, load the next, compute again. This keeps the scheduler fed.

3. Register Reuse Flags (SASS-level)

In SASS, bits [63:58] of the control word encode register reuse flags. These hint to the operand collector that a source register value will be reused by the next instruction, so it should be kept in the collector's local storage rather than re-read from the register file. This avoids register bank conflicts where two instructions need to read from the same bank simultaneously. Lithos SASS emitter sets these flags directly based on the kernel's register access pattern.

4. Stall Counts (SASS-level)

Bits [57:53] of the control word encode a minimum stall count: how many cycles the scheduler must wait before issuing the next instruction from this warp. Values range from 0 to 15 (5 bits, though practical values rarely exceed 15). ptxas sets these based on its dependency analysis. Lithos SASS emitter can override them — for example, inserting a stall of 0 when Lithos knows the next instruction has no true dependency, even if ptxas conservatively inserted a stall.

5. Yield Hints

Bit [52] of the control word is the yield hint. When set, it suggests the scheduler should prefer other warps over this one for subsequent issue slots. This is useful when a warp has just issued a long-latency global memory load — there's no point selecting it again until the load completes (typically 200-400 GPU cycles on HBM, or ~110-220 ns end-to-end from ld.global issue to data in a register). Setting yield lets other warps use those cycles productively — up to 400 other instructions per sub-partition during a single HBM miss.

6. Barrier Synchronization

bar.sync forces all warps in a thread block to reach a synchronization point before any can proceed past it. This is necessary for shared memory patterns (write shared, sync, read shared) but expensive: the last warp to arrive unblocks all others, so any imbalance in arrival time is wasted cycles. Lithos controls barrier placement to minimize the synchronization surface — using barriers only where shared memory coherence requires it, not where the compiler defensively inserts them.

7. Warp-level Divergence

When threads in a warp take different branches, the scheduler must execute both paths sequentially, with inactive threads masked off. A warp that diverges into two paths takes twice as many cycles. Lithos patterns avoid divergence entirely by using predicated execution (@p instruction guards) instead of branches. Every thread executes every instruction; the predicate simply controls whether the result is written. No divergence, no wasted cycles.

8. Memory Access Patterns

Coalesced memory accesses — where all 32 threads in a warp access a contiguous 128-byte region — complete in a single memory transaction. Uncoalesced accesses split into multiple transactions, each with its own latency. Faster completion means the warp becomes eligible sooner, giving the scheduler more ready warps. Lithos patterns guarantee coalescing by construction: thread i always accesses element base + i, never base + stride*i with stride > 1.

The SASS Control Word — The Direct Interface

Every SASS instruction on Hopper is 16 bytes: an 8-byte instruction word and an 8-byte control word. The control word is the scheduler's per-instruction configuration. Here is the bit layout, derived from systematic probing documented in sass/ENCODING.md:

Control Word (64 bits) Bits [63:58] Register reuse cache flags (6 bits) Which source operand values to cache for reuse by the next instruction. Bit 58 = src1 reuse. Higher bits for src2/src3. Bits [57:53] Stall count (5 bits, 0-31 cycles) Minimum cycles to wait before issuing the next instruction from this warp. Bit [52] Yield hint Suggest scheduler should prefer other warps after this instruction. Bits [51:49] Write barrier index (0-5, 7=none) Scoreboard slot for tracking when this instruction's write completes. Bits [48:46] Read barrier index (0-5, 7=none) Scoreboard slot for tracking when this instruction's read completes. Bits [45:42] Wait barrier mask Which scoreboard barriers must complete before this instruction issues. Bits [41:0] Instruction-specific modifiers Varies by opcode: rounding mode, negation, abs, saturation, etc.

This is the mechanism. Every instruction carries its own scheduling hints, embedded in the binary. There is no separate scheduling pass or runtime decision for these parameters — they are baked into the cubin at compile time.

ptxas fills in the control word automatically based on its dependency analysis and heuristics. It does a reasonable job for general-purpose code. But it optimizes for the average case, not for a specific kernel's known memory access pattern and compute pipeline.

Lithos SASS emitter fills them in deliberately. When Lithos knows a kernel streams sequentially through HBM with no reuse, it can set stall counts to exactly match the load-to-use latency. When it knows register allocation avoids bank conflicts, it can clear reuse flags that ptxas conservatively sets. When it knows a load will take 300+ cycles (~165+ ns), it sets the yield hint so the scheduler doesn't waste cycles polling the warp.

Why This Matters for Inference

At batch=1, the GPU is memory-bound. Every token generation reads the full model weights from HBM once. The compute (matrix-vector multiplies) takes far fewer cycles than the memory transfer. The scheduler's job reduces to one thing: keep warps cycling between "issue load" and "issue compute" fast enough that HBM bandwidth is saturated.

If the scheduler makes bad choices — picks a warp that's still waiting on a load, issues to a functional unit that's busy, stalls because no warp is eligible — bandwidth utilization drops. Each wasted cycle is a cycle where HBM could have been streaming data but wasn't.

The 13% HBM utilization measured in vLLM means the scheduler is failing 87% of the time. Either there aren't enough eligible warps (occupancy too low, or all warps blocked on the same long-latency event), or the kernel structure doesn't give the scheduler good options (loads bunched together, no compute to interleave, unnecessary barriers forcing serialization).

Lithos attacks this from both directions. The kernel patterns are structured to maximize scheduler choice (interleaved loads and compute, high occupancy, no divergence, guaranteed coalescing). And the SASS emitter provides precise scheduling hints (stall counts, yield, reuse flags) tuned to the specific access pattern of each kernel, rather than relying on ptxas heuristics that don't account for the batch=1 streaming workload.

The theoretical ceiling is 4 TB/s on GH200. Lithos has measured 3.59 TB/s (89.7%) on bandwidth benchmarks. The gap between 13% and 89.7% is the gap between generic scheduling and deliberate scheduling.
