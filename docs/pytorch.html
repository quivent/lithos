Lithos vs PyTorch

Why the inference stack doesn't need a training framework.

What PyTorch Is

PyTorch is a tensor math library with automatic differentiation, built for training neural networks. Installed size: 2+ GB. It loads cuBLAS (800 MB), cuDNN (hundreds of MB), the Python runtime, and dozens of C++ dispatch layers.

It is an excellent training framework. It is the standard for research. It is also what most of the industry uses for inference, because it was already there when inference became a product.

That does not make it the right tool.

What PyTorch Does During Inference

The parts that matter.

  • Loads weight tensors from .safetensors files
  • Dispatches matrix multiplications to cuBLAS
  • Dispatches attention, norm, activation to cuDNN or custom CUDA kernels
  • Manages GPU memory allocation via CUDACachingAllocator
  • Manages tensor metadata — shapes, strides, dtypes

Five things. Everything else PyTorch does during inference is overhead from its training heritage.

What PyTorch Does During Inference That Is Waste

  • Autograd engine — gradient computation. Disabled for inference but still present in the codepath, checked on every operation.
  • Computation graph recording — unnecessary for inference. The model is frozen.
  • Python GIL — serializes all CPU-side dispatch. One thread at a time, always.
  • Dispatcher overhead — 1-3µs per operation through a virtual dispatch chain (Autograd, BackendSelect, CUDA, etc.).
  • Dynamic shape handling — reshaping, broadcasting rules checked at runtime when shapes are fixed.
  • Memory allocator complexity — pool management, fragmentation handling, when mmap would suffice on GH200.
  • Error checking and validation on every tensor operation.
  • Module system — nn.Module, parameters, buffers, hooks. Training abstractions carried into inference.

Every one of these exists because PyTorch must support training: dynamic shapes, arbitrary graphs, gradient tracking, flexible memory. Inference needs none of it.

The PyTorch Inference Path — Step by Step

One matrix multiplication. The actual call chain:

Python: model.linear(x)                            ~1µs
  → torch.nn.functional.linear(x, weight, bias)   ~0.5µs
  → torch._C._TensorBase.mm(x, weight)            ~0.5µs  Python → C++ boundary
  → PyTorch C++ dispatcher                         ~2µs   5+ dispatch keys checked
  → ATen::mm implementation                        ~1µs
  → validates shapes, dtypes, contiguity           ~1µs
  → allocates output tensor via CUDACachingAllocator ~2µs
  → cuBLAS workspace setup                         ~1µs
  → cublasGemmEx()                                 ~1µs   selects kernel from atlas
  → CUDA Runtime: cudaLaunchKernel                 ~0.5µs
  → CUDA Driver: cuLaunchKernel                    ~0.5µs
  → GPU executes

Total overhead before the GPU does anything: 10-40µs. Per operation. Thousands of operations per token.

The Lithos Inference Path — Same Operation

Lithos: PROJECTION                                   ~1µs
  → cuLaunchKernel (pre-loaded cubin, pre-packed args) ~1-2µs
  → GPU executes

Total overhead: 2-5µs.

No Python. No dispatcher. No shape validation. No memory allocator. No cuBLAS atlas lookup. The kernel is already compiled. The arguments are already packed. One driver call.

What Lithos Keeps

libcuda.so — the driver. Approximately 30 MB. This is the only interface to the GPU hardware. It translates API calls into command buffers the GPU understands. Cannot be eliminated without reverse-engineering the GPU's command buffer protocol.

The weight files — .safetensors is just a flat array of numbers with a JSON header. Lithos reads it directly with mmap. No PyTorch deserialization. No pickle. No torch.load(). The file maps into the unified address space and the GPU reads it in place.

The Point

PyTorch is a training framework pressed into service for inference. Every abstraction it provides — autograd, dynamic graphs, the module system, the dispatcher — is designed to make training flexible. Inference is not flexible. The model is frozen. The shapes are fixed. The operations are known.

Lithos is built for this: a fixed model, fixed shapes, known operations, maximum throughput. No training abstractions. No dispatcher. No Python. Just the math, on the metal.

Lithos — from lithography. Writing on stone. The process that puts circuits on silicon. Lithos writes on the GPU.
