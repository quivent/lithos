GH200

The whole die. 132 SMs, 50 MB of L2, six HBM3e stacks, one NVLink-C2C to Grace.

The Chip

GH200 is two chips fused together. A Hopper GPU (GH100-class silicon) and a Grace CPU, connected by NVLink-C2C. The GPU is what Lithos writes on. The CPU is an assistant: fast path for long KV, slow path for everything that isn't a tensor.

132 SMs

Each Streaming Multiprocessor is the unit of execution. 132 of them on the die, arranged in GPCs (GPU Processing Clusters) and TPCs (Texture Processing Clusters), though from a kernel's point of view that hierarchy is invisible. What matters is: 132 independent engines, each running up to 64 warps concurrently.

Every SM is identical. The compiler does not care which SM a block lands on — the hardware's Compute Work Distributor picks. Lithos does not target SMs by index; it targets the count.

50 MB L2

The L2 cache sits between all SMs and HBM. Every byte read from HBM passes through it; every byte written ends up there first. It is split into two partitions, each serving one half of the die, but the GPU presents it as a single coherent cache. Hit latency is ~260 cycles. Miss latency goes out to HBM — ~500 ns.

50 MB is large enough to hold the full KV cache for a short context, or all the weights of a small model, or one activation tensor for a 70B forward pass. Lithos uses it as the primary working set: if a tensor fits in L2 and gets reused across kernels, it never touches HBM twice.

6 HBM3e Stacks

Six stacks of High Bandwidth Memory 3e, each 16 GB, each delivering about 670 GB/s. Aggregate 96 GB at 4 TB/s. They sit on the edges of the die on a silicon interposer; signal distance is sub-millimeter. This is where model weights live after mmap. This is where the KV cache lives when it's hot.

HBM bandwidth is the hard ceiling on memory-bound kernels. A decode step reading 20 GB of weights can finish no faster than 20 / 4000 = 5 ms. Everything in Lithos downstream of that arithmetic is about hitting that ceiling, not beating it.

NVLink-C2C

900 GB/s of coherent bandwidth between Grace and Hopper. Cache-coherent. Unified virtual address space. A pointer allocated on the CPU is dereferenceable from the GPU and vice versa; the hardware moves the page where it needs to go.

This is what makes the 480 GB of LPDDR5X usable as GPU memory. A pointer into LPDDR5X dereferenced from an SM triggers a page fault (first touch) or a remote load over C2C (subsequent). At 512 GB/s from the LPDDR5X side, it is 8x slower than HBM — but 300 GB/s remains 30x faster than an SSD page-in. The cold KV cache lives here.

Lithos places tensors so that most accesses stop at L2 or L1. Weights read once from HBM; activations stay in registers; KV cache streams from HBM or LPDDR5X but never copies.

What Lithos Sees

The pieces above are not abstractions. They are what the code targets:

  • gridDim is chosen so that blocks saturate 132 SMs.
  • Block size is chosen so that registers_per_thread * threads <= 65536 per SM.
  • Tile size is chosen so that input + output + weights fit in 228 KB shared memory.
  • Access pattern is chosen so that 128-byte L2 lines are fully consumed.
  • KV cache is placed in HBM or LPDDR5X based on age; both are pointers.

There is no cudaMalloc, no cudaMemcpy, no stream synchronization except at the token boundary. The GPU is one large memory system with compute bolted on.
