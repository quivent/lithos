Bandwidth Utilization

HBM utilization as the governing constraint at batch=1.

Claim

At batch=1 decode, a transformer reads every weight in the model to produce one token. Compute is not the bottleneck; tensor cores are underutilized. Throughput is set by HBM bandwidth — the fraction of the 4 TB/s peak consumed by useful weight traffic.

vLLM on this GH200 at batch=1 sustains 13% of peak (effective ~520 GB/s). Lithos targets >70% (>2.8 TB/s). A device-to-device copy benchmark on the same hardware measured 3.59 TB/s, 90% of the 4 TB/s theoretical peak — establishing the hardware ceiling.

The gap between sustained bandwidth during a kernel and time-averaged bandwidth across a forward pass is the inter-kernel gap. The HBM controller is idle for the remaining 87% while CPU-side dispatch proceeds.

Defining Bandwidth Utilization Precisely

"Bandwidth utilization" covers two distinct measurements. Both are reported; Lithos targets the second.

Sustained utilization (during active reads)

While a kernel is executing and its warps are issuing LDG instructions, sustained utilization measures the fraction of peak the HBM bus delivers. Well-tuned kernels — vLLM's individual kernels, cuBLAS, FlashAttention — reach 80%+.

Time-averaged utilization (over wall time)

Total bytes transferred divided by total elapsed time, including gaps between kernels. This number sets tok/s. vLLM lands at 13% here.

A kernel can sustain HBM at 80% while the program averages 13%: 80% × 16% duty cycle ≈ 13%.

Reconciling 13% vs 35.8%

Two measurements of the same run yield different numbers because they measure different things.

vLLM on this GH200 achieves 179 tok/s at batch=1.

Wall time per token: 1 / 179 = 5.58 ms
Bytes moved per token: 8 GB
 
Effective bandwidth (avg):
  8 GB / 5.58 ms = 1.43 TB/s
  1.43 / 3.59 = 39.8% of measured peak
  1.43 / 4.00 = 35.8% of theoretical peak

The 13% figure is the profiler estimate. The GPU profiler samples memory bandwidth across wall time, counting only cycles during which HBM controllers are actively servicing requests. Between kernels — dispatch gap, scheduler idle, scoreboard drain, CUDA graph node boundary — the controller reports zero.

The profiler sees:

  • Kernel executing, reading hard: ~80% of peak sustained, for ~13% of wall time.
  • Between kernels, GPU idle: 0% of peak, for ~87% of wall time.
  • Time-weighted average: 80% × 0.13 ≈ 10–13%.

The bytes/time estimator gives 35.8%, which dictates tok/s. The profiler's number reports the duty cycle. The bytes/time number reports distance from the physical ceiling. Lithos targets the bytes/time number at >70%.

vLLM's individual kernels reach ~80% sustained bandwidth. The 87% idle time between kernels is the dominant cost.

How Lithos Closes the Gap

The path to >70% utilization is mechanical: remove overhead at each layer of the stack.

Four decisions

1. Fused kernels: ~47x fewer launches

3000 launches become 64 layer-kernels, or fewer with cross-layer fusion. Launch overhead collapses from 30+ ms scheduled to <1 ms. See the Fusion section on the Performance page for the mechanism.

2. No Python in the path

The inference loop is Forth-compiled native code calling cuLaunchKernel directly. Dispatcher overhead: zero. No tensor objects, no autograd guards, no backend selection.

3. No CUDA graph capture needed

CUDA graphs exist to amortize the cost of launching many small kernels. The fused Lithos kernels are the graph. Nothing to capture.

4. Pre-computed launch parameters

Grid dims, block dims, argument pointers, shared memory sizing — all baked at kernel generation time. At dispatch there's no arithmetic, just a pointer to a pre-assembled cuLaunchKernel argument block.

The >70% target leaves margin below the ~90% hardware ceiling. HBM row activation, refresh, and bank conflicts account for the gap between 90% and 100%.

The Single-Launch Double Win

Lithos takes kernel fusion to its logical endpoint: one kernel launch for the entire forward pass. One function, one cuLaunchKernel call, all 64 layers executed sequentially within that function. This eliminates two independent sources of idle HBM time.

Win 1: Zero launch overhead

Each kernel launch has a fixed cost: CPU writes the launch descriptor, GPU reads it, the CWD dispatches work to SMs. Measured cost: 2–5 µs per launch.

A 64-layer forward pass has ~448 operation boundaries (64 layers × ~7 operations per layer). At ~3 µs average, that is ~1.3 ms of pure overhead per token — time when the HBM controller is ready to deliver weights but no warp has issued an LDG. One launch replaces all of it with a single 3 µs cost.

Win 2: Zero intermediate HBM traffic

Between separate launches, intermediate values must be written to HBM — the only memory visible to both launches. Within a single launch, intermediates stay in registers (1 cycle) or shared memory (23 cycles).

Boundaries per forward pass: 64 layers × ~7 ops = ~448
Traffic per boundary: 20 KB × 2 (write + read) = 40 KB
Total redundant traffic: 448 × 40 KB = ~17 MB
 
At 4 TB/s, 17 MB takes ~4 µs of bus time.
But latency dominates: ~800 cycles per HBM round-trip (write 400 + read 400)
  448 × 800 = 358,400 cycles = ~196 µs at 1.83 GHz

One launch eliminates 17 MB of redundant HBM traffic and 196 µs of round-trip latency per token. All intermediates stay in registers or shared memory.

Combined savings

Launch overhead eliminated: ~1.3 ms
HBM latency eliminated: ~0.2 ms
Total savings: ~1.5 ms per token
 
Current per-token time (vLLM): 5.58 ms (179 tok/s)
Reduction: 1.5 / 5.58 = ~27%
Projected rate from this alone: ~245 tok/s

Single-launch architecture eliminates both launch overhead and intermediate memory traffic. The combined savings are ~1.5 ms per token on GH200, representing ~27% of the current per-token time measured with vLLM. This is a design target, not a measurement.

Engineering constraint: registers and code size

One function doing 64 layers does not need 64× the registers. Operations execute sequentially — layer 0 finishes before layer 1 starts. The compiler reuses registers. Peak register usage equals the most register-hungry single operation (the tensor core tile), not the sum of all operations.

Compiled code size does grow (more instructions = larger cubin), but instructions stream from L2 cache to the SM's instruction cache on demand. This is identical to how a CPU handles large functions — the instruction cache holds the hot working set, not the entire binary.

The copy benchmark at 3.59 TB/s shows the hardware delivers 90% of its rated peak when no framework, dispatcher, or per-op kernel layer is involved. The gap between 13% and 90% is software overhead.

Lithos's 70% target sits below the 90% copy ceiling to account for decode work that is not pure weight streaming: norms, rotary embeddings, softmax, and gate/up/down sequencing.

Summary

Batch=1 inference is memory-bandwidth bound. Each token reads every weight once and performs a small amount of compute per weight. Tensor core throughput, clock speed, and occupancy are not the constraint.

Reducing wall-time per token requires reducing GPU idle time between kernels: PyTorch dispatcher resolution, Python interpreter steps, CUDA runtime argument validation, and CUDA graph node boundaries all contribute idle HBM cycles.

Each layer of abstraction between the model and the silicon contributes idle HBM cycles. Lithos's approach is to remove those layers.

At 70% sustained bandwidth, Lithos would run at ~2.3x vLLM's measured throughput on the same hardware. The delta comes from reduced idle time between kernels, not from kernel-level optimization.
