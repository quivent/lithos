Kernels

Beyond the vanilla transformer.

Why Custom Kernels Matter

The vanilla transformer has 7 kernel types. Projection, norm, activation, residual, attention scoring, rotary embedding, sampling. Every inference engine in existence is built around these seven operations. cuBLAS handles projection. cuDNN handles attention. FlashAttention handles both. The ecosystem is mature.

Real models are not vanilla transformers.

Qwen 3.5-27B is a hybrid. 48 of its 64 layers are DeltaNet — linear recurrent attention with a learned state matrix. The remaining 16 layers are full attention, placed every 4th layer in a strict 3:1 pattern. Three-quarters of this model runs on an operation that cuBLAS has no kernel for. cuDNN has no kernel for it. FlashAttention has no kernel for it.

Every existing inference engine handles this one of two ways: ignore the recurrent structure and run it as dense attention (wrong, and slow), or use slow generic fallbacks that treat the recurrence as a sequence of matmuls with Python-level loop overhead. Neither option is acceptable.

A purpose-built DeltaNet kernel is the single largest optimization opportunity for Qwen 3.5-27B. It is not an incremental improvement. It is the difference between the GPU doing useful work and the GPU waiting.

The Qwen 3.5-27B Architecture Map

64 layers. Not all the same. The pattern is absolute:

Layer 0: DeltaNet (linear attention, recurrent state) Layer 1: DeltaNet Layer 2: DeltaNet Layer 3: Full Attention ← every 4th layer Layer 4: DeltaNet Layer 5: DeltaNet Layer 6: DeltaNet Layer 7: Full Attention Layer 8: DeltaNet Layer 9: DeltaNet Layer 10: DeltaNet Layer 11: Full Attention ...repeats 16 times... Layer 60: DeltaNet Layer 61: DeltaNet Layer 62: DeltaNet Layer 63: Full Attention

48 DeltaNet layers. 16 full attention layers. 75/25 split.

DeltaNet Layers

Each DeltaNet layer maintains a compressed state matrix S of shape [num_heads x key_dim x value_dim]. For Qwen 3.5: [16 x 128 x 128] = 256 KB per layer at FP16. Processing a token costs O(1) — one state update, one query against the state. No KV cache. No attention over the full sequence. The cost per token is constant regardless of context length.

The tradeoff: the state is lossy. Information from early tokens gets overwritten by later tokens. The state drifts. After three consecutive DeltaNet layers, the accumulated drift is measurable.

Full Attention Layers

Every 4th layer runs full quadratic attention. It reads the entire KV cache — cost O(seq_len) per token. This is expensive, but it corrects the drift accumulated by the preceding DeltaNet layers. Full attention sees every prior token directly, not through a compressed state.

The 3:1 Ratio

Three tokens of cheap recurrence, one token of expensive correction. The architecture gets 75% of its layers at O(1) cost and uses the remaining 25% to keep the representation accurate. This is not arbitrary — it is a design decision that trades a small amount of per-layer accuracy for a large reduction in compute. The full attention layers act as periodic error correction for the DeltaNet state drift.

Lithos Kernel Types for Qwen 3.5

Not 7 kernel types. 12. The vanilla seven, plus five that exist because this model is a hybrid.

Standard — Shared with Vanilla Transformers

01 PROJECTION

Matrix multiply. Weight matrix times activation vector. The workhorse — 448 per forward pass across all layers. Every QKV projection, every gate/up/down in the MLP, every output projection. This is what cuBLAS does well. Lithos does it with the exact tile sizes for Qwen's dimensions, inlined constants, no dispatch overhead.

02 NORM

RMSNorm. Square, reduce, reciprocal square root, scale, multiply. 128 per forward pass. Elementwise, memory-bound, trivially parallel. Always fused with RESIDUAL in practice.

03 ACTIVATE

SiLU activation — x * sigmoid(x). 64 per forward pass (once per MLP layer). Five instructions. Entirely compute-bound. Fused into the MLP kernel.

04 RESIDUAL

Skip connection addition. 128 per forward pass. Always fused with NORM — one read, one write for both operations. Never launched as a standalone kernel.

05 SAMPLE

Logit to token. 1 per forward pass. Temperature scaling, top-k/top-p filtering, categorical sampling. Runs on CPU (72 Grace cores). The only operation in the forward pass that is not on the GPU.

06 EMBED

Token to embedding vector. 1 per forward pass. A table lookup. Trivial.

Attention-Specific — 16 of 64 Layers

07 ATTENTION-SCORE

Flash attention for the full attention layers only. QK^T scoring, softmax, value aggregation. Reads the KV cache. Cost: O(seq_len) per token. Only runs on layers 3, 7, 11, 15, ..., 63 — the 16 full attention layers. The other 48 layers never touch this kernel.

08 ROTATE

Rotary position embedding (RoPE). Applies rotation matrices to query and key vectors to encode position information. Only needed for full attention layers — DeltaNet layers encode position implicitly through the sequential state update.

DeltaNet-Specific — 48 of 64 Layers — New Territory

This is where every existing engine stops. No library provides these. No framework fuses them. Lithos builds them from PTX.

09 RECURRENCE

The core DeltaNet state update. The operation that defines the entire recurrent layer:

S_{t+1} = g_t * S_t + beta_t * k_t * (v_t - k_t^T @ S_t)
  • State matrix S: [num_heads x key_dim x value_dim] = [16 x 128 x 128] = 256 KB per layer at FP16
  • g_t: gating scalar (per head) — controls how much old state to retain
  • beta_t: learning rate scalar (per head) — controls how much new information to write
  • k_t: key vector, v_t: value vector — the new information
  • k_t^T @ S_t: retrieval from current state — makes the update state-dependent

The k_t^T @ S_t retrieval is what makes this non-trivial. The update depends on the current state. This means the recurrence is non-invertible — you cannot reconstruct the previous state from the current state. This has direct consequences for speculative decoding, which needs to roll back state on rejection.

10 CONV1D

Causal 1D convolution with kernel size 4 in Qwen 3.5. Applied before the DeltaNet recurrence. Maintains a small rolling buffer of the 4 most recent input vectors. A 4-tap FIR filter, trivially parallel across channels. This gives the recurrence a short-range local context before it updates the global state.

11 DELTANET-OUTPUT

Compute the output from the recurrent state. The query vector multiplied against the state matrix:

output = q_t^T @ S_t

Different from attention output: no softmax, no KV cache scan. One matrix-vector multiply against the state. Cost: O(key_dim x value_dim) = O(16384) FLOPs per head. Constant regardless of sequence length.

12 STATE-CHECKPOINT

Save and restore DeltaNet state for speculative decoding. Clone the state matrix before verification. Restore on rejection. 48 layers x 256 KB = 12.3 MB per checkpoint.

Measured: 0.85 ms for rollback via state restore vs 8.7 ms for recomputation. A 10.2x speedup. This kernel exists because the DeltaNet recurrence is non-invertible — you cannot undo a state update, you can only restore from a saved copy.

DeltaNet Optimization Opportunities

Kernel Fusion Specific to DeltaNet

FUSED-DELTANET-LAYER

The entire DeltaNet layer as one kernel: NORM + CONV1D + PROJECTION(Q,K,V,gate,beta) + RECURRENCE + DELTANET-OUTPUT + RESIDUAL. One kernel launch. No intermediate writes to HBM. No kernel boundary gaps.

This is simpler than fused attention because there is no KV cache to manage. The only persistent state is the state matrix. At 256 KB per layer, the state matrix almost fits in shared memory (228 KB available per SM on Hopper). With careful tiling — partitioning across heads, keeping one or two heads resident at a time — it fits. The entire recurrence stays in shared memory. No HBM round-trips for the state.

State Precision Optimization

This was proven empirically on Qwen 3.5-27B. The results are unambiguous:

State PrecisionAccuracyNotes
FP32100%Exact match with reference implementation
BF16~50%Drift accumulates through state-dependent feedback
FP16~15%Catastrophic drift — the model produces garbage

The recurrence accumulates errors because it is state-dependent. The k_t^T @ S_t retrieval feeds back into the update. Small precision errors in the state corrupt the retrieval, which corrupts the update, which corrupts the state further. The feedback loop amplifies rounding errors exponentially.

The Lithos RECURRENCE kernel maintains FP32 state with FP16/BF16 compute — mixed precision within a single kernel. The state matrix is accumulated in FP32. The projections, gating, and output query run in FP16 or BF16 for throughput. This is the same principle behind vLLM's --mamba-ssm-cache-dtype float32 flag, but built into the kernel rather than bolted on as a runtime option.

Modal MTP — Self-Speculative Decoding

The key insight: skip all 16 attention layers for draft tokens. Run only the 48 DeltaNet layers. The DeltaNet-only forward pass is O(1) per token with no KV cache reads. Draft at ~3x speed, then verify with the full model.

The acceptance rate depends entirely on state precision:

State PrecisionDraft Acceptance RateEffective Speedup
FP32100%Full benefit of speculative decoding
BF16~50%Marginal — half the drafts are rejected
FP16~15%Worse than no speculation — verification overhead exceeds savings

The kernel precision directly determines whether speculative decoding works. FP32 state is not optional. Lithos can have a DRAFT-FORWARD kernel that only runs the 48 DeltaNet layers — skipping attention, skipping KV cache, skipping RoPE. Pure recurrence. This is the model speculating against itself, using its own recurrent layers as the draft model.

Recurrent Rollback Optimization

Split-recurrence rollback: save state references at each speculative position. On rejection, restore the correct state with one memcpy per layer — O(1) vs O(K) recomputation where K is the number of speculative positions.

The numbers:

MethodRollback CostScaling
Recomputation8.7 msO(K) — rerun K tokens through 48 layers
State checkpoint + restore0.85 msO(1) — one memcpy per layer, constant regardless of K

This is unique to Lithos. No other engine has STATE-CHECKPOINT as a native kernel type. Other engines either do not support DeltaNet at all, or they recompute on rejection because they have no mechanism to snapshot and restore recurrent state. The 12.3 MB checkpoint cost is trivial — it fits in L2 cache (50 MB on GH200).

GDN shadow state is the alternative approach found in vLLM patches: maintain a parallel "shadow" copy of the DeltaNet state that tracks the last verified position. On rejection, swap the shadow into the active slot. Same O(1) rollback, but doubles the memory footprint for state. Lithos uses explicit checkpointing instead — snapshot only at speculation boundaries, not continuously.

When to Create a Custom Kernel

Create a Kernel When

  • Unique memory access pattern. DeltaNet: state-dependent read-modify-write. The state is both read (for retrieval) and written (for update) in the same operation. This pattern does not exist in matmul or attention.
  • No existing library supports it. cuBLAS has no recurrence kernel. cuDNN has no recurrence kernel. There is no library to call. You write it or it does not exist.
  • Hot path dominance. 48 of 64 layers = 75% of the model. An operation that runs in 75% of layers is not a corner case. It is the model.
  • Cross-boundary fusion opportunity. CONV1D + RECURRENCE + OUTPUT in one kernel eliminates two HBM round-trips per layer. At 48 layers, that is 96 eliminated round-trips per forward pass.
  • The operation does not map to existing types. Recurrence is not a matrix multiply (it has state). It is not attention (no softmax, no KV cache). It is its own thing.

Do Not Create a Kernel When

  • It is a different-sized version of an existing type. A 4096x4096 matmul and a 2048x2048 matmul are the same PROJECTION kernel with different constants. Do not create a second kernel type.
  • It is already fused into another kernel. RESIDUAL is always inside NORM. It does not need its own launch. It is not a kernel type in practice — it is part of the NORM kernel.
  • The operation is trivially mapped. If cuBLAS already achieves >90% of peak for your operation at your sizes, the engineering cost of a custom kernel exceeds the performance gain.

Ideas for Future Custom Kernels

Not needed for Qwen 3.5-27B. Listed for when the model changes or the target expands.

F1 MIXTURE-OF-EXPERTS-ROUTING

Top-k expert selection, token scatter/gather. The routing decision determines which expert weights to load — a memory access pattern that changes per token. Requires dynamic dispatch within a single kernel launch.

F2 SPECULATIVE-VERIFY

Parallel verification of K draft tokens against the full model. Run K forward passes simultaneously, compare output distributions, accept/reject in one kernel. Avoids K sequential verification steps.

F3 KV-CACHE-COMPACT

Compress old KV cache entries for long context. Evict low-attention-score entries, quantize old entries to lower precision. Keeps the 16 full attention layers viable at very long sequence lengths without unbounded memory growth.

F4 CROSS-ATTENTION

For encoder-decoder models. Different from self-attention — queries come from the decoder, keys and values from the encoder. Different access pattern, different fusion opportunities.

F5 SPARSE-PROJECTION

For pruned or sparse weight matrices. Structured sparsity (2:4 on Hopper) gets hardware support via tensor cores. Unstructured sparsity requires custom index lookup and scattered memory access.

F6 GROUPED-GEMV

Multiple small GEMVs batched together. For GQA with many query heads sharing few KV heads — batch the per-head matrix-vector multiplies into one kernel launch instead of looping.

Building Lithos Around Qwen 3.5

Lithos is not a generic framework. It is built for this model on this hardware.

The kernel vocabulary reflects this directly:

  • 48 DeltaNet layers get first-class RECURRENCE / CONV1D / DELTANET-OUTPUT kernels
  • 16 attention layers get ATTENTION-SCORE / ROTATE kernels
  • Speculative decoding is native: DRAFT-FORWARD + STATE-CHECKPOINT + SPECULATIVE-VERIFY
  • Mixed precision (FP32 state, FP16/BF16 compute) is built into the RECURRENCE kernel, not bolted on as a runtime flag
  • The 3:1 hybrid pattern is encoded in the engine loop, not discovered at runtime

The Fused Kernel Target

In production, the 12 kernel types collapse into 5 fused kernels:

5 kernel types in production. Not 7. Not thousands. Five.

Every kernel boundary that still exists is between operations that must synchronize: the DeltaNet layers need their state before the attention layer can correct drift. The attention layer needs the KV cache written before it can score. The sample step needs all 64 layers done before it picks a token. These are the only boundaries. Everything else is fused.

This is what it means to build an engine for a specific model. Not an engine that can run Qwen 3.5. An engine that is Qwen 3.5, compiled to silicon.
