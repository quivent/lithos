The Kernel Recipe

How config.json becomes cubins.

The Recipe Book Is config.json

Every model ships with a config.json. Lithos reads it and derives exactly which kernels are needed.

Qwen 3.5-27B:

{
  "num_hidden_layers": 64,
  "hidden_size": 5120,
  "num_attention_heads": 24,
  "num_key_value_heads": 4,
  "head_dim": 256,
  "intermediate_size": 17408,
  "vocab_size": 248320,
  "layer_types": [
    "linear_attention", "linear_attention", "linear_attention", "full_attention",
    ... (repeats 16 times)
  ],
  "quantization_config": {
    "quant_method": "gptq",
    "bits": 4,
    "group_size": 128
  }
}

Every field is a compile-time constant in some Lithos kernel — baked into the SASS as immediates. No runtime dispatch reads these values.

The Derivation Rules

Four rules. Applied mechanically. No heuristics, no tuning knobs.

RULE 01 Every model needs three kernels regardless of architecture.
  • EMBED — sized (vocab_size, hidden_size) = (248320, 5120)
  • NORM — sized (hidden_size) = (5120)
  • SAMPLE — sized (vocab_size) = (248320)
RULE 02 Projections are derived from the model's dimensions.

For each unique (input_dim, output_dim) pair that appears in the model, emit one PROJECTION kernel. Shapes that repeat — like K and V — share a single cubin.

If any layer is full_attention:

  • Q projection: (hidden, num_heads × head_dim) = (5120, 6144)
  • K projection: (hidden, num_kv_heads × head_dim) = (5120, 1024)
  • V projection: (hidden, num_kv_heads × head_dim) = (5120, 1024) — same shape as K, shares kernel
  • O projection: (num_heads × head_dim, hidden) = (6144, 5120)

Every layer has MLP:

  • Gate: (hidden, intermediate) = (5120, 17408)
  • Up: (hidden, intermediate) = (5120, 17408) — same shape as gate, shares kernel
  • Down: (intermediate, hidden) = (17408, 5120)

Unique projection shapes for Qwen: (5120, 1024), (5120, 6144), (5120, 17408), (6144, 5120), (17408, 5120) — 5 projection variants.

RULE 03 Architecture-specific kernels.

If any layer is full_attention:

  • ATTENTION-SCORE — (head_dim, num_kv_heads, GQA ratio)
  • ROTATE — (head_dim, partial_rotary_factor)

If any layer is linear_attention (DeltaNet):

  • RECURRENCE — (num_kv_heads, key_dim, value_dim)
  • CONV1D — (conv_kernel_dim, channels)
  • STATE-CHECKPOINT — (state size per layer)
RULE 04 Quantization fuses.

If quantization_config.quant_method is gptq or awq:

  • DEQUANTIZE is fused into PROJECTION — no separate kernel, just a different PROJECTION variant.
  • Parameters: bits = 4, group_size = 128.

When Does This Happen?

At startup, once per model:

  1. Load config.json — ~1 ms.
  2. Derive the kernel list from the four rules above — instant.
  3. Check the cache. Cache key = hash of the config.json subset that affects kernel code.
  4. If cached: load cubins from disk — 0.6 ms for all 11.
  5. If not cached: emit SASS cubins directly — microseconds. No ptxas in the pipeline.

SASS-Direct Compilation

Lithos emits SASS binary directly into cubin ELF files — no ptxas JIT step.

  1. Lithos compiler emits SASS bytes — 16 bytes per instruction (8-byte opcode + 8-byte control word).
  2. Wrap SASS in a cubin ELF with the required NVIDIA metadata sections — .nv.info, .nv.constant0, etc.
  3. Write cubin to disk or pass directly to cuModuleLoadData — the driver loads it as-is, no JIT step.

Why SASS-direct

  • Instant recompilation — change a constant, regenerate a kernel, load in microseconds.
  • Full scheduling control — set stall counts and register reuse flags deliberately instead of relying on ptxas heuristics.
  • No dependency on ptxas — the compiler that lives inside the NVIDIA driver.

Current status

  • 47 opcodes mapped via probing. Vector-add cubin runs correctly on GH200.
  • Memory-op control words still being characterized — Hopper pipeline depths, dual-issue rules, and register bank layout are reverse-engineered but not exhaustive.
  • Focus areas: PROJECTION tensor-core loop, DELTANET recurrence.
At inference time: no recipe work. Kernels match the model's dimensions after startup. Every token runs the same set. The per-matmul dispatch path present in other engines is absent here.

Different Model → Different Recipe → Different Cubins

The cache is keyed by the subset of config that affects kernel code — dimensions, layer_types, quantization. Everything else is irrelevant to the cubin.

  • Change the prompt? Same cubins.
  • Change the model weights (same architecture)? Same cubins.
  • Switch architectures (Qwen → LLaMA)? New cubins.
  • Change quantization (FP16 → GPTQ-4bit)? New cubins — the PROJECTION kernels now dequantize inline.

The hash is deterministic. Two machines reading the same config.json derive the same cache key and produce bit-identical cubins. The recipe is the config; the config is the recipe.

Compiling for a specific model moves the dimensions from runtime data into the kernel code itself.
