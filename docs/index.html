Lithos

Writing on silicon.

What Lithos Is

Lithos is a Forth-derived language that writes computation directly onto GPU silicon. It emits PTX (and eventually SASS) for NVIDIA GPUs, dispatched from ARM64 CPUs. No C++, no nvcc, no framework.

The Problems Lithos Solves

01 Kernel Granularity

Current inference stacks launch 20-50 kernels per transformer layer. Each kernel boundary forces hardware synchronization: drain warps, flush pipeline, re-schedule. Inter-kernel gaps: 1-5µs each. At 64 layers, thousands of gaps.

Measured on GH200: 13% bandwidth utilization at batch=1. The GPU is idle 87% of the time.

Lithos eliminates kernel boundaries by fusing entire layers — or entire models — into single kernels. Fused RMSNorm+residual, fused attention+softmax, fused GEMV+activation. No inter-kernel gaps. No re-scheduling.

02 Compiler Quality

nvcc passes through 6-7 intermediate representations. ptxas makes generic scheduling decisions. NVIDIA's own cuBLAS uses hand-written SASS for peak GEMM performance because the compiler can't match it.

Lithos emits PTX directly. The programmer controls instruction interleaving, register allocation, shared memory access patterns, and async copy pipelines. Every pattern in the vocabulary encodes a known-good scheduling decision.

03 Memory Management on Unified Memory

cudaMalloc/cudaMemcpy exist for discrete GPUs with separate memory spaces. GH200 has hardware-coherent unified memory. The entire CUDA memory management layer is unnecessary overhead.

Lithos on GH200: allocate with mmap, pass the pointer to the kernel. The hardware handles coherence. No cudaMalloc, no memory pools, no transfer scheduling.

04 Dispatch Overhead

Current path: Python → PyTorch → ATen → cuBLAS → CUDA Runtime → Driver → GPU. Measured: 10-40µs per operation. With CUDA graphs: 0.3-1µs amortized.

Lithos: Forth → driver API (cuLaunchKernel) → GPU. One call. For the fused mega-kernel approach, one launch per forward pass.

05 Compilation Latency

nvcc compiles a FlashAttention kernel in 10-60 seconds. CUTLASS template instantiation takes hours. TensorRT model compilation takes minutes.

Lithos compilation is string concatenation. Vocabulary words append PTX text to a buffer. A full inference kernel compiles in microseconds. Specialize kernels at runtime for exact model config, sequence length, batch size. Recompile instantly when conditions change.

Design Rules for Lithos Patterns

Constraints applied to every Lithos pattern. Violations leave bandwidth or compute unused.

1. Coalesce at 32 bytes. Every global memory access from a warp must hit contiguous 32-byte sectors. Misaligned access wastes bandwidth 1:1 with the misalignment.
2. Avoid shared memory bank conflicts. 32 banks, 4 bytes each. If two threads in a warp access the same bank (different addresses), one stalls.
3. Fill warps for latency hiding. 6-8 warps per sub-partition to hide shared memory latency. 12-16 to hide global memory latency.
4. Match tensor core shapes. m16n8k16 for FP16/BF16. m16n8k32 for FP8/INT8. Tiles that don't match waste tensor core cycles.
5. Use async copy. cp.async and TMA bypass registers. Global memory flows directly to shared memory. Don't waste register file on data staging.
6. Minimize register pressure. ≤128 registers/thread for memory-bound, ≤255 for compute-bound. More registers = fewer warps = less latency hiding.
7. Fuse to eliminate global memory round-trips. Every kernel boundary writes to HBM and reads back. Fused kernels keep intermediates in registers or shared memory. Savings: 10-100x per intermediate value.

Architecture

Current Path: Lithos → PTX → SASS

Lithos Source Forth words that emit PTX | | microseconds string concatenation v PTX Text Buffer valid PTX, ready to load | | cuModuleLoadData driver JITs to SASS v SASS on GPU cached after first load | | cuLaunchKernel one call v GPU Execution

Future Path: Lithos → SASS Direct

Lithos Source | | emit-sass.fs raw 16-byte SASS instructions v SASS Code Buffer 65536 bytes, binary | | build-cubin wrap in ELF, NVIDIA sections v cubin ELF .text.<kernel> + .nv.info + .nv.constant0 | | cuModuleLoadData no JIT, direct load v GPU Execution skip ptxas entirely

The GPU never runs Forth. The GPU runs whatever Lithos compiled for it. The CPU orchestrates: token input, KV cache management, sampling, speculative decode logic. On GH200, both share the same memory. The orchestration is pointer passing, not data copying.

Vocabulary

Lithos words are organized in layers. Lower layers emit individual PTX instructions. Higher layers compose them into complete kernels. Each word takes register IDs from the Forth stack, allocates new registers, emits PTX, and returns result register IDs.

L0 PTX Primitives

Direct 1:1 mapping to PTX instructions. ~65 instruction words, ~20 type modifiers, ~6 memory space qualifiers.

add.f32 — Float add

( fa fb -- fc )
: add.f32
  f32+
  >r >r >r
  tab s" add.f32 " ptx+ r> %f ,ptx r> %f ,ptx r> %f ;ptx
  n-f32 @ 1 - ;

Emits: add.f32 %f2, %f0, %f1;

fma.rn.f32 — Fused multiply-add

( fa fb fc -- fd )
: fma.rn.f32
  f32+
  >r >r >r >r
  tab s" fma.rn.f32 " ptx+ r> %f ,ptx r> %f ,ptx r> %f ,ptx r> %f ;ptx
  n-f32 @ 1 - ;

Emits: fma.rn.f32 %f3, %f0, %f1, %f2;

rsqrt.f32 — Reciprocal square root

( fa -- fb )
: rsqrt.f32
  f32+
  swap >r >r
  tab s" rsqrt.approx.ftz.f32 " ptx+ r> %f ,ptx r> %f ;ptx
  n-f32 @ 1 - ;

Emits: rsqrt.approx.ftz.f32 %f1, %f0;

ld-global-f32 — Coalesced global load

( rd-base rtid -- fval )
: ld-global-f32
  r64+ f32+
  >r >r >r >r
  tab s" cvt.u64.u32 " ptx+ r> %rd ,ptx r> %r ;ptx
  drop
  tab s" mul.wide.u32 " ptx+ r@ %rd ,ptx r@ %rd s" , 4" ptx+ ;ptx
  tab s" add.u64 " ptx+ r@ %rd ,ptx r@ %rd ,ptx r> %rd ;ptx
  tab s" ld.global.f32 " ptx+ r> %f s" , [" ptx+ n-r64 @ 1 - %rd s" ]" ptx+ ;ptx ;
L1 Access Patterns

Cached optimizations for common GPU memory and compute patterns. ~30 pattern words.

gtid.x — Global thread ID

( -- rtid )
: gtid.x
  r32+ r32+ r32+ r32+   \ r0=tid.x r1=ctaid.x r2=ntid.x r3=result
  dup 3 - >r
  tab s" mov.u32 " ptx+ r@ 3 - %r s" , %tid.x" ptx+ ;ptx
  tab s" mov.u32 " ptx+ r@ 2 - %r s" , %ctaid.x" ptx+ ;ptx
  tab s" mov.u32 " ptx+ r@ 1 - %r s" , %ntid.x" ptx+ ;ptx
  tab s" mad.lo.s32 " ptx+ r@ %r ,ptx r@ 2 - %r ,ptx r@ 1 - %r ,ptx r@ 3 - %r ;ptx
  r> ;

Emits PTX for blockIdx.x * blockDim.x + threadIdx.x — the most common pattern in every kernel.

warp-reduce-sum.f32 — Butterfly reduction

( fin -- fout )
: warp-reduce-sum.f32   \ 5 rounds: offsets 16, 8, 4, 2, 1
  f32+
  dup >r swap >r
  tab s" shfl.sync.bfly.b32 " ptx+ r@ %f ,ptx r> %f
      s" , 16, 0x1f, 0xffffffff" ptx+ ;ptx
  tab s" add.f32 " ptx+ ... ,ptx r@ %f ;ptx
  \ ... repeat for 8, 4, 2, 1
  r> ;

Full warp-level sum reduction using butterfly shuffles. 10 instructions: 5 shuffles + 5 adds. Returns sum in all lanes.

cp-async-128 — Async global-to-shared copy

( rd-smem rd-gmem -- )
: cp-async-128
  tab s" cp.async.ca.shared.global [" ptx+ swap %rd
      s" ], [" ptx+ %rd s" ], 128" ptx+ ;ptx ;

Emits: cp.async.ca.shared.global [%rd0], [%rd1], 128; — bypasses registers, uses TMA on Hopper.

L2 Fused Operations

Multiple PTX instructions that always appear together in inference kernels. Fusion eliminates intermediate memory traffic. ~30 composite words.

silu-elem.f32 — SiLU activation

( fin -- fout )    \ x * sigmoid(x) = x / (1 + exp(-x))
: silu-elem.f32
  f32+ f32+ f32+
  dup 3 - >r swap >r
  tab s" neg.f32 " ptx+ r@ %f ,ptx r> %f ;ptx
  tab s" mul.f32 " ptx+ r@ %f ,ptx r@ %f s" , 0f3FB8AA3B" ptx+ ;ptx  \ log2(e)
  tab s" ex2.approx.ftz.f32 " ptx+ r@ %f ,ptx r@ %f ;ptx
  tab s" add.f32 " ptx+ r@ %f ,ptx r@ %f s" , 0f3F800000" ptx+ ;ptx  \ + 1.0
  tab s" rcp.approx.ftz.f32 " ptx+ r@ %f ,ptx r@ %f ;ptx
  tab s" mul.f32 " ptx+ r@ 1+ %f ,ptx n-f32 @ 4 - %f ,ptx r> %f ;ptx
  n-f32 @ 3 - ;

6 instructions. No intermediate memory access. Computes x * sigmoid(x) entirely in registers via ex2.approx.

rmsnorm-elem — RMSNorm

( fx fweight feps rlane rwarp -- fout )
: rmsnorm-elem   \ x * rsqrt(mean(x^2) + eps) * weight
  \ x^2
  f32+ tab s" mul.f32 " ptx+ ...
  \ warp-reduce sum of x^2
  warp-reduce-sum.f32
  \ broadcast from lane 0
  warp-bcast.f32
  \ add eps, rsqrt
  rsqrt.f32
  \ x * rsqrt * weight
  ... ;

Fuses square, warp-cooperative reduce, rsqrt, and scale into a single pattern. No shared memory needed for the reduction — uses butterfly shuffles.

mma-f16-f32 — Tensor core MMA

( rA0 rA1 rA2 rA3 rB0 rB1 fC0 fC1 fC2 fC3 -- fD0 fD1 fD2 fD3 )
: mma-f16-f32   \ m16n8k16, f16 inputs, f32 accumulators
  f32+ f32+ f32+ f32+
  tab s" mma.sync.aligned.m16n8k16.row.col.f32.f16.f16.f32" ptx+
  tab tab s" {" ptx+ ... %f s" }," ptx+    \ D accumulators
  tab tab s" {" ptx+ ... %r s" }," ptx+    \ A fragments
  tab tab s" {" ptx+ ... %r s" }," ptx+    \ B fragments
  tab tab s" {" ptx+ ... %f s" }" ptx+ ;ptx  \ C accumulators
  ;

Register Encoding

Destination register: bits [23:16] (tentative). Source registers: various positions depending on instruction format. RZ (zero register) = 0xFF. Uniform registers (UR) use a different encoding space. Predication (@P0) modifies the instruction word, not the control word.

Inference Engine

The inference engine's job: take model weights and a token sequence, produce the next token. Repeat.

The Work Per Token

StepOperationNotes
1Read weights from memory~8 GB for 27B W4. Every token.
2Matrix-vector multiplyActivation × weights. 64 layers.
3AttentionScore against all previous tokens' KV cache. Read/write KV entries.
4NormalizeRMSNorm before and after attention + MLP. Elementwise.
5ActivateSiLU in the MLP. Elementwise.
6Residual addSkip connections. Elementwise.
7SampleLogits to token. Once per forward pass.

Steps 1-6 repeat per layer. Step 7 once at the end.

Engine Decisions

DecisionStrategy
Weight placementHBM for hot layers, CPU memory for cold. Unified memory makes this a soft boundary.
KV cache placementStarts in HBM, spills to CPU memory at NVLink speed. Not a cliff — a gradient.
CPU vs GPU dispatchMatmuls: GPU. Sampling: CPU (72 Grace cores). KV cache management: CPU.
OverlapNext layer's weight prefetch overlaps current compute. CPU sampling overlaps GPU prefill.
BatchingMultiple requests share weight reads. Linear scaling until compute-bound. Batch=1: 13% BW. Batch=8: approaching ceiling.

Kernel Guide — Why These Kernels, Why Not Others

Why 7 Kernels

Each kernel matches a distinct parallelism strategy — PROJECTION tiles matrices across warps, NORM reduces a single vector. Seven operations, everything else is repetition with different weight pointers.

The Kernels

PROJECTION — GEMM

Token-vector × weight matrix. 448 times per token (7 projections × 64 layers). 90% of all work. Tile into shared-memory blocks, feed tensor cores, accumulate.

ATTENTION-SCORE

Streams the KV cache rather than reading a fixed matrix — the cache grows per token. Different access pattern from PROJECTION.

NORM

RMS reduction + broadcast across the hidden vector. Reduction-then-broadcast pattern, distinct from GEMM or attention.

ACTIVATE

SiLU, elementwise. Embarrassingly parallel — no inter-element dependency.

ROTATE

RoPE. 2D rotations on pairs of elements, angle depending on position.

RESIDUAL

Elementwise add. Always fused with NORM — never launched alone.

DEQUANTIZE

4-bit → FP16 unpack. Fused inside PROJECTION. Format-specific logic (W4, W8, GPTQ, AWQ).

SAMPLE

Argmax/top-k over the 248,320-wide logit vector. Runs on CPU — small enough for cache.

EMBED

Token ID → 5,120-dim vector. Table lookup.

Why Not cuBLAS

cuBLAS ships 800 MB of pre-compiled kernels for every supported matrix size, data type, and GPU. Lithos generates the kernel for the model's specific dimensions. cuBLAS does not fuse PROJECTION with DEQUANTIZE or NORM with RESIDUAL because it operates at the GEMM level and has no view of surrounding operations.

Fusion

Separate kernels write intermediates to HBM and read them back. Launch overhead and write-then-read dominate at a few KB. Fusion keeps intermediates in registers (1 cycle) vs HBM (400 cycles).

The 3-Kernel Target

Target configuration: 3 fused kernels — ATTENTION (norm + rotate + projection×4 + attention-score + residual), MLP (norm + projection×3 + activate + residual), and SAMPLE. One kernel per major block. No intermediate HBM traffic within a block.

Code Examples

1. Defining a Kernel

A Lithos kernel definition using the vocabulary from core.fs:

\ Vector add kernel: output[i] = input_a[i] + input_b[i]

ptx-reset ptx-header

s" vec_add" kernel{
  s" ptr_a"   param-u64 ,param
  s" ptr_b"   param-u64 ,param
  s" ptr_out" param-u64 ,param
  s" n"       param-u32
  )params

  \ Load parameters
  s" ptr_a"   ld-param-u64    \ -- rd0
  s" ptr_b"   ld-param-u64    \ -- rd0 rd1
  s" ptr_out" ld-param-u64    \ -- rd0 rd1 rd2

  \ Compute global thread ID
  gtid.x                        \ -- rd0 rd1 rd2 rtid

  \ Bounds check
  \ bounds-exit

  \ Load, compute, store
  2 pick over ld-global-f32     \ -- ... fa
  3 pick over ld-global-f32     \ -- ... fa fb
  add.f32                       \ -- ... fc
  2 pick over st-global-f32     \ store result
}kernel

2. The PTX It Produces

.version 8.0
.target sm_90
.address_size 64

.visible .entry vec_add (
  .param .u64 ptr_a,
  .param .u64 ptr_b,
  .param .u64 ptr_out,
  .param .u32 n
)
{
  .reg .b32  %r<8>;
  .reg .b64  %rd<6>;
  .reg .f32  %f<4>;

  // Load parameters
  ld.param.u64    %rd0, [ptr_a];
  ld.param.u64    %rd1, [ptr_b];
  ld.param.u64    %rd2, [ptr_out];

  // Global thread ID
  mov.u32         %r0, %tid.x;
  mov.u32         %r1, %ctaid.x;
  mov.u32         %r2, %ntid.x;
  mad.lo.s32      %r3, %r1, %r2, %r0;

  // Coalesced load: addr + tid*4
  cvt.u64.u32     %rd3, %r3;
  mul.wide.u32    %rd3, %rd3, 4;
  add.u64         %rd3, %rd3, %rd0;
  ld.global.f32   %f0, [%rd3];

  cvt.u64.u32     %rd4, %r3;
  mul.wide.u32    %rd4, %rd4, 4;
  add.u64         %rd4, %rd4, %rd1;
  ld.global.f32   %f1, [%rd4];

  // Add
  add.f32         %f2, %f0, %f1;

  // Coalesced store
  cvt.u64.u32     %rd5, %r3;
  mul.wide.u32    %rd5, %rd5, 4;
  add.u64         %rd5, %rd5, %rd2;
  st.global.f32   [%rd5], %f2;

  exit;
}

3. How It Launches

\ Compile PTX and launch on GPU
ptx$                             \ -- addr u    (PTX text buffer)

\ Load PTX module via CUDA driver API
cuModuleLoadData                 \ -- module
s" vec_add" cuModuleGetFunction  \ -- func

\ Set up launch parameters
256 1 1                          \ block: 256 threads
n 255 + 256 / 1 1                \ grid: ceil(n/256) blocks

\ Launch — one call, one kernel, the GPU executes
cuLaunchKernel                   \ GPU runs vec_add

Forth words concatenate PTX strings. ptx$ returns the buffer. The driver JITs it to SASS. One cuLaunchKernel call. No framework, template instantiation, or intermediate representations between Forth and PTX.

Because compilation is microseconds, kernels specialize at runtime: generate a kernel tuned for the exact hidden_dim, num_heads, head_dim. Regenerate when batch size changes. Inline constants. Unroll loops to exact layer count.

Lithos — from lithography. Writing on stone. The process that puts circuits on silicon. Lithos writes on the GPU.
