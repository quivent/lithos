The Three Resources That Determine Performance

GPU performance comes down to three physical resources. Every kernel is a bet on how to balance them. Understanding these resources is understanding why Lithos exists.

Registers

Each SM has 65,536 registers — 256 KB of the fastest storage on the chip. Each thread can claim up to 255 of them. A register access takes 1 cycle.

When a thread needs more data than its registers can hold, the excess spills to memory. The nearest fallback is shared memory at 23 cycles. The real fallback is HBM at 400 cycles. That is a 400x penalty for running out of register space.

But here is the tradeoff: more registers per thread means fewer threads can run simultaneously on the SM. Fewer threads means the warp scheduler has less work to choose from, which means it cannot hide memory latency by switching to another warp while one waits. This is THE fundamental tradeoff in GPU kernel optimization.

cuBLAS picks a generic balance — one that works reasonably well across thousands of possible problem sizes. Lithos picks the exact balance for each specific kernel, because it knows the model dimensions at compile time.

Memory Bandwidth

HBM3e delivers 4 TB/s. But you only get that number if every warp's memory access is coalesced: 32 threads reading 32 consecutive bytes, aligned to a 32-byte sector boundary. Misaligned access wastes bandwidth in direct proportion to the misalignment. A half-misaligned access pattern gets half the bandwidth.

Between HBM and registers sits shared memory: 256 KB per SM, accessible in ~23 cycles. It is the staging area. Data arrives from HBM into shared memory, then threads pull it into registers for computation. But shared memory has its own pitfall: 32 banks, 4 bytes each. If two threads in a warp hit the same bank at different addresses, one of them stalls. This is a bank conflict, and it serializes what should be parallel.

Every Lithos access pattern is designed for conflict-free shared memory access and fully coalesced global memory access. Not as a best practice. As a guarantee.

Compute Units

Each SM has 128 FP32 CUDA cores and 4 tensor cores. The tensor cores are the heavy lifters: mma.sync.m16n8k16 at FP16 delivers 1,024 FLOPs/clock/SM.

At batch=1, compute is almost irrelevant. The GPU finishes the arithmetic before memory can deliver the next chunk of data. The kernel is memory-bound — the bottleneck is how fast you can read weights from HBM, not how fast you can multiply them.

At batch=8 and above, the arithmetic starts to matter. Multiple requests share the same weight read, so you do more math per byte fetched. Tensor core utilization becomes the constraint. The kernel's job shifts to keeping data flowing to tensor cores without stalls — every gap between mma instructions is wasted throughput.

What Every Existing Engine Does — And Where It Falls Short

Every major inference engine makes real optimizations. None of them go far enough. Here is what each one does well and where it stops.

vLLM
Python + PyTorch + cuBLAS + FlashAttention

Optimizations

  • PagedAttention — virtual memory management for KV cache, eliminates fragmentation
  • Continuous batching — new requests join a running batch without waiting
  • CUDA graphs — record and replay kernel launch sequences to cut dispatch overhead
  • Prefix caching — reuse KV cache across requests with shared prompts
  • Speculative decoding — draft tokens with a smaller model, verify in parallel

Falls Short

  • Python dispatch overhead on every operation, even with CUDA graphs
  • torch.compile JIT warmup adds seconds before first token
  • Cannot fuse across cuBLAS calls — each GEMM is an island
  • cuBLAS selects kernels from a generic atlas — no model-specific specialization
  • PyTorch memory allocator (caching allocator, fragmentation, OOM handling) adds overhead
  • Measured on GH200: 13% bandwidth utilization at batch=1, 179 tok/s single stream
TensorRT-LLM
C++ + cuBLAS + custom kernels

Optimizations

  • Layer fusion — attention + norm combined into single kernels
  • INT8/FP8 quantization with calibrated scaling
  • In-flight batching for continuous request serving
  • Multi-GPU tensor parallelism across NVLink
  • Compiled execution plan — fixed graph, no runtime decisions

Falls Short

  • Minutes-long build phase before first inference — recompile for different batch sizes
  • Fixed execution plan — rebuild the entire engine when batch size changes
  • Still uses the cuBLAS atlas for GEMMs — no model-specific GEMM kernels
  • No custom SASS — relies on ptxas for final instruction scheduling
llama.cpp
C++ + custom CUDA kernels

Optimizations

  • Quantization-aware kernels (Q4_K, Q5_K, etc.) with custom dequantization
  • Fused operations where hand-written
  • Minimal dependencies — single-file CUDA kernels, no framework

Falls Short

  • Kernels written in CUDA C++ — nvcc compiles them through 6-7 IR stages
  • Generic tile sizes that do not adapt to specific model dimensions
  • No tensor core usage for some quantization formats
  • No SASS-level control over scheduling or register assignment
MLX
Apple Metal + Swift

Optimizations

  • Unified memory — no CPU-GPU transfers on Apple Silicon
  • Lazy evaluation — operations fuse before execution
  • Fused Metal kernels with compile-time specialization

Falls Short

  • Metal-only — no NVIDIA GPU support
  • Apple controls the Metal compiler — no SASS equivalent, no ISA-level access
  • simdgroup_matrix is opaque — cannot control the hardware matrix units directly
FlashAttention
Tri Dao — CUDA C++ + inline PTX

Optimizations

  • Tiled attention that never materializes the full N×N attention matrix
  • O(N) memory instead of O(N²) — the key insight that makes long contexts feasible
  • Inline PTX for wgmma and cp.async — reaches past CUDA C++ for critical paths

Falls Short

  • Written in CUDA C++ with inline PTX — not full PTX control, still goes through nvcc
  • One kernel for attention only — does not fuse with MLP or normalization
  • Compile time: 10-60 seconds per kernel variant

Lithos Optimizations — What We Can Do

Each technique below is something Lithos can do that existing engines cannot, or cannot do as well. The common thread: Lithos controls the full path from model definition to GPU instruction.

01 Register Allocation Control
What

Choose exactly how many registers each thread uses. More registers means more data stays on-chip. Fewer registers means more concurrent threads for latency hiding.

How

Lithos emits PTX with explicit register declarations — .reg .b32 %r<N> with N chosen per kernel. Eventually, the SASS emitter will assign physical registers directly, bypassing ptxas allocation entirely.

Impact

Memory-bound kernels (projection at batch=1): maximize registers to 255, minimize threads. Compute-bound kernels (projection at batch=8): balance for occupancy with ~128 registers per thread. The right choice depends on the workload, and Lithos makes a different choice for each kernel.

cuBLAS cannot do this — it uses one-size-fits-all register allocation per kernel variant from the atlas.

02 Kernel Fusion
What

Combine multiple operations into one kernel launch. Intermediate values stay in registers instead of round-tripping through HBM.

How

Lithos vocabulary words compose. fused-add-rmsnorm emits one kernel that does residual add + RMSNorm in a single pass. The 3-kernel target (ATTENTION, MLP, SAMPLE) eliminates all intermediate memory traffic within a layer.

Impact

Each eliminated kernel boundary saves one HBM write + one HBM read of the full activation vector. At hidden_dim=5120 and FP16: 10 KB per boundary. With 40+ boundaries per forward pass in a conventional stack, that is 400+ KB of eliminated memory traffic per token — and more importantly, 40+ fewer pipeline drains and kernel launch gaps.

cuBLAS cannot do this — it only does GEMM. It has no concept of the operations before or after.

03 Constant Specialization
What

Bake model dimensions into the kernel as compile-time constants instead of runtime parameters.

How

Lithos generates kernels for specific hidden_dim, head_dim, num_heads. Loop bounds become known constants. Branch conditions resolve at compile time. Dead code disappears. Because Lithos compilation takes microseconds, re-specializing for a different model is instant.

Impact

Eliminates parameter loads, enables full loop unrolling, and lets the compiler (ptxas or the future Lithos SASS emitter) schedule optimally with no runtime ambiguity about trip counts or branch targets.

cuBLAS cannot do this — its kernels must handle arbitrary dimensions, so loop bounds and tile sizes remain runtime variables.

04 Memory Access Pattern Control
What

Guarantee that every global memory load is coalesced (32 consecutive bytes per warp, aligned to sector boundaries) and that every shared memory access is bank-conflict-free.

How

Lithos patterns encode the correct access patterns structurally. coalesced-f16@ guarantees alignment by construction. Shared memory swizzling is built into the tile load patterns — not bolted on as an optimization pass.

Impact

Full 4 TB/s bandwidth utilization, versus the 13% measured in vLLM at batch=1. Conflict-free shared memory means 32 threads access 32 banks simultaneously every cycle, with no serialization.

05 Async Copy Pipelining
What

While tensor cores compute on the current tile, the next tile is being loaded from HBM to shared memory asynchronously. Double or triple buffering keeps the pipeline full.

How

cp.async and TMA instructions in the Lithos patterns. The software pipeline uses multiple shared memory buffers: one being consumed by compute, one being filled by async copy, one waiting. LDGSTS at the SASS level (opcode 0xfae) flows data directly from global to shared memory without touching the register file.

Impact

Hides memory latency entirely when compute is long enough to cover the load. Eliminates stalls between tiles. The GPU never waits for data — the next tile is always ready.

06 Warp-Level Programming
What

Use warp shuffle instructions instead of shared memory for small reductions and broadcasts. 1 cycle versus 23 cycles.

How

warp-reduce-add and warp-broadcast patterns emit shfl.sync.bfly chains. At the SASS level, this is the SHFL instruction (opcode 0xf89 for immediate lane source) with butterfly mode in bits [59:58]. Five shuffles reduce 32 values. No shared memory needed.

Impact

RMSNorm, softmax, and other reductions become 5x faster within a warp. For operations that are dominated by reduction (normalization, softmax denominator), this is the difference between compute-bound and memory-bound.

07 Tensor Core Utilization
What

Keep the tensor cores fed continuously. mma.sync.m16n8k16 at FP16 delivers 1,024 FLOPs/clock/SM. If there is any stall between mma instructions, throughput drops proportionally.

How

Lithos interleaves mma with async loads in the software pipeline. The schedule ensures the next operands are in shared memory (via cp.async) and loaded into registers (via ldmatrix / LDSM) before the current mma completes. At the SASS level, HMMA.16816.F32 (opcode 0x23c) is the instruction that must never stall.

Impact

Sustained tensor core throughput instead of burst-and-stall. At batch=8+, this is the difference between 50% and 90% of peak compute utilization.

08 GH200-Specific: Unified Memory
What

On the GH200, CPU and GPU share the same physical memory with hardware-level coherence via NVLink-C2C at 900 GB/s. No cudaMalloc. No cudaMemcpy. No staging buffers.

How

Lithos uses mmap to allocate memory. Pointers are passed directly to GPU kernels. The hardware maintains coherence between CPU and GPU address spaces at page granularity. No driver calls for memory management.

Impact

Eliminates memory management overhead entirely. KV cache can seamlessly spill from HBM (96 GB at 4 TB/s) to LPDDR5X (480 GB at 500 GB/s) without explicit transfers. It is not a cliff — it is a gradient.

09 CPU Offload (72 Grace Cores)
What

Run sampling, KV cache management, and speculative decode verification on the Grace CPU while the GPU computes the next matmul.

How

CPU and GPU share memory. The CPU reads GPU output directly — coherent, no copy. No synchronization needed beyond the kernel completing. The CPU starts sampling the moment the GPU writes the logits. While the CPU picks a token, the GPU begins the next token's first layer.

Impact

Overlap CPU work with GPU work. Sampling time is hidden behind GPU compute. For speculative decoding, the CPU verifies draft tokens while the GPU generates the next draft batch.

10 Direct SASS Emission (future)
What

Skip ptxas entirely. Emit SASS binary directly into cubin ELF files. Control instruction scheduling, register bank assignment, and reuse flags at the hardware level.

How

47 SASS opcodes mapped and verified through systematic probing (see the SASS Encoding section). Every instruction is 16 bytes: 8-byte instruction word (opcode, registers, immediates) + 8-byte control word (stall counts, barriers, reuse flags). The Lithos SASS emitter writes cubin ELF directly.

Impact

Eliminates ptxas JIT time (100ms–2s per kernel). Potentially beats ptxas scheduling for memory-bound workloads where manual interleaving of loads and computes — with precise stall counts in the control word — outperforms the generic scheduler.

What Lithos Cannot Do (Yet)

  • Training. No autograd, no optimizer, no backward pass. Lithos is inference-only. Training requires gradient tracking through every operation — a fundamentally different architecture.
  • Multi-GPU tensor parallelism. Currently targets a single GH200. The 96 GB HBM + 480 GB LPDDR5X is enough for most models up to ~70B parameters at W4 quantization. Multi-GPU support is a future goal.
  • Dynamic model architectures. Lithos specializes kernels at compile time for specific model dimensions. Mixture-of-experts routing or dynamic sparse architectures would require runtime kernel selection — possible but not yet implemented.
  • Non-NVIDIA GPUs. PTX and SASS are NVIDIA-specific instruction sets. AMD (RDNA/CDNA), Intel (Xe), and Apple (Metal) each have their own ISA. Lithos would need a separate backend for each.
