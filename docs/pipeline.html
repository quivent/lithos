Inference Stack

Source to silicon. Every layer of the Lithos pipeline, with no abstraction layers hidden.

The Complete Pipeline

Lithos is a direct path: source files and safetensors compile to GPU machine code, packaged in ELF, loaded by a minimal kernel module, and launched by an ARM64 process via a single memory write to the GPFIFO ring. No CUDA. No Python. No intermediate runtimes. Weights are code — Q4 weight values are encoded as instruction immediates in the ELF .text section at compile time. There is no runtime weight loading from HBM.

Standard ELF binary containing sm90a GPU machine code. No NVIDIA-specific sections required. Kernel Module Layer
lithos.ko  —  /dev/lithos0
Replaces nvidia.ko. Owns the GPU hardware. Boots the GSP (RISC-V coprocessor on die) via firmware before any GPFIFO channel can be created. Exposes exactly 4 ioctls to userspace. Maps USERD (User Space Engine RegiDo) so the ARM64 launcher can ring the GPU doorbell without a syscall.

Dispatch Layer
GPFIFO ring + USERD doorbell
ARM64 launcher writes GPU commands directly into the GPFIFO ring via memory-mapped I/O. Zero syscalls at inference time. After writing, it rings the doorbell by writing the new GPPut pointer to the USERD register page.
Write GPPut → USERD + 0x8c (new tail pointer) Ring doorbell → USERD + 0x90 (kick GPU to fetch new work) For the fused megakernel: one GPFIFO entry per forward pass. No kernel re-launch, no synchronization barriers between layers.

Hardware Layer
GH200 SM array — 132 SMs
NVIDIA GH200 Hopper GPU. 132 streaming multiprocessors, each with 4 warp schedulers. Runs the two cooperative-grid megakernels: forward pass and recurrence state update. BAR4 gives 128 GB coherent HBM via NVLink-C2C — CPU physical address equals GPU virtual address. No DMA mapping, no copies, no page faults.
Per SM: 4 warp schedulers, 256 KB register file, 228 KB shared mem GPU: 132 active SMs, ~1830 MHz boost Memory: 128 GB HBM3, BAR4 coherent (CPU PA == GPU VA) Target: two cooperative-grid megakernels per model
No CUDA anywhere
No libcuda.so. No PTX. No nvcc. No ptxas. No CUDA Runtime. No cuBLAS. No cuDNN. No framework. The pipeline goes: Lithos source + safetensors → SASS machine code → ELF → lithos.ko → GPU. Every layer is owned by Lithos. No weight loading — weights are instruction immediates compiled into the ELF .text section.
BAR4 coherent memory
GH200 NVLink-C2C: 128 GB HBM is coherent between ARM64 Grace CPU and Hopper GPU. CPU physical address == GPU virtual address. No cudaMemcpy, no DMA mapping, no transfer scheduling. Pass a pointer; the hardware handles coherence.
GSP requirement
The RISC-V GSP coprocessor on-die must boot via firmware before any GPFIFO channel is functional. lithos.ko handles GSP boot during module init.

16 bytes / instruction
Every sm90a instruction is exactly 16 bytes: an 8-byte 64-bit instruction word (opcode + operands) paired with an 8-byte control word (stall counts, yield bit, read/write barrier flags). Lithos emits both words directly — no assembler pass.

Two Cooperative-Grid Megakernels

Lithos compiles entire transformer forward passes into exactly two cooperative-grid kernels per model. No inter-kernel synchronization barriers. No kernel re-launch between layers. One GPFIFO entry per token at inference time.

Forward Pass Kernel

  • RMSNorm + residual add (fused)
  • Weight-immediate projection streams (Q, K, V)
  • Attention + softmax (fused)
  • Weight-immediate output projection
  • RMSNorm (FFN) + residual
  • Weight-immediate gate + up projection
  • SiLU activation
  • Weight-immediate down projection
  • Sampling (argmax / temperature)

Recurrence Update Kernel

  • DeltaNet state update (delta rule)
  • KV cache write (attend.li)
  • Recurrent hidden state (recur.li)
  • Elementwise gating (elementwise.li)
  • Cross-layer state threading
  • Cooperative grid sync between SMs

Cooperative grid launch means all SMs in both kernels can synchronize via a device-wide barrier (bar.sync / cooperative_groups::sync) without returning to the CPU. The GPU itself orchestrates the inter-SM handoff between attention and FFN, and between layers.

12 Primitive Families

The entire Lithos vocabulary is 12 primitive families. Each family maps to a known-good binary pattern that the compiler (gpu/emit.fs) emits directly. The dimensional notation encodes the shape of the data flowing through each operation.

Each primitive emits a fixed binary pattern: the same inputs always produce the same bytes. The compiler is a lookup table, not an analysis engine. This is a deliberate design choice — predictability over generality.

Weights as Code

In the weights-as-code architecture, the compiler reads model.safetensors at compile time. Each Q4 weight value is packed into the immediate field of an FMUL-IMM instruction. The resulting ELF .text section is the model weights — there is no separate weight tensor in HBM.

How it works

At compile time, the Lithos compiler opens the safetensors file, reads each layer's weight matrix, quantizes to Q4 if not already quantized, and emits a stream of 64-bit instructions where the immediate fields encode the packed 4-bit weight values. Each row of the weight matrix becomes a sequence of FMUL-IMM instructions that multiply the activation register by the weight immediate. There is no GEMV loop, no weight pointer parameter, no LDG instruction fetching weights from HBM.

Memory hierarchy inversion

Conventional Weights-as-Code
Weights are data in HBM, fetched via LDG Weights are instruction immediates in .text, fetched via L1I cache
Weight bandwidth limited by HBM (4 TB/s) Weight bandwidth limited by instruction fetch (L1I cache hierarchy)
HBM stores weights + activations + KV cache HBM stores only activations + state matrices + KV cache
Kernel takes weight pointer as parameter No weight parameter — weights are compiled into the instruction stream
cubin/ELF loads weights at runtime The ELF .text section IS the weights

The instruction cache becomes the weight cache

The L1 instruction cache (L1I) on each SM fetches instruction words from L2 cache, which fetches from HBM. Weights flow through this path instead of the data path. The instruction cache hierarchy (L1I → L2 → HBM) replaces the data cache hierarchy for weight access. Activations remain runtime data in registers and shared memory.
