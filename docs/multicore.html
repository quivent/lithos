Multicore

The 72 Grace cores are part of the machine.

The Premise

The GH200 is not a GPU with a CPU bolted on for I/O. It is a coherent compute package: one Hopper die with 132 SMs, and one Grace die with 72 ARM Neoverse V2 cores, connected by NVLink-C2C at 900 GB/s bidirectional in a single cache-coherent address space.

A single-threaded orchestrator wastes 71 cores. Every idle cycle is work the GPU could be fed, a request tokenized, a sample taken, a KV slot rotated. Single-threaded design is legacy thinking — an accident of CPython's GIL, not a property of the hardware.

The GPU is one compute unit. The CPU is 72 compute units. Lithos treats them as peers sharing memory, not as a master-slave relationship where the CPU exists only to launch kernels.

The Grace CPU on GH200 is not a throttled companion processor. It is a full server-class ARM64 CPU on the same package as the GPU.

Memory Coherence: NVLink-C2C Makes Them Peers

On discrete GPUs, CPU and GPU memory are separate pools. Data moves through cudaMemcpy, DMA, and PCIe. CPU pointers are meaningless to the GPU and vice versa.

GH200 extends the GPU's cache coherence protocol across the chiplet boundary via NVLink-C2C. A CPU mmap pointer is valid on the GPU; a GPU kernel's write is immediately visible to any Grace core. One address space, hardware coherence.

Lithos treats them as peers. Tokenizer writes into the buffer the GPU reads; GPU writes logits into the buffer the sampler reads. Orchestration is pointer passing.

The Pipeline Model

The GPU is never waiting for the CPU. The CPU is never waiting for the GPU. Per-token wall time approaches the GPU's forward-pass time plus the NVLink-C2C latency for the handoff (nanoseconds), not the sum of every serial stage.

Rule of thumb. GPU for matmuls and parallel reductions; CPU for anything sequential, scheduling, I/O, sampling, tokenization, and state management. The boundary is not CPU=slow vs GPU=fast — both are fast at different shapes of work.

Request-Level Parallelism

Each request gets its own CPU orchestrator thread owning token stream, KV cache slots, sampling params, streaming connection. With 72 cores, tens of in-flight requests each get a dedicated core with cycles to spare.

All threads share the cubins (132 KB) and weights (mmapped, read-only). Sharing is free on the coherent fabric — no marshalling, no copies.

Requests serialize at the GPU via stream ordering, but with in-flight batching, multiple requests share the forward pass; between passes, their CPU work runs in parallel.

Speculative Decoding: CPU Drafts, GPU Verifies

Speculative decoding drafts tokens with a small model and verifies them in a single parallel pass of the big model. The traditional split puts both on the GPU, competing for tensor cores.

On GH200, the draft belongs on the CPU. Draft is sequential by nature (token N+1 depends on N) and small enough that LPDDR5X bandwidth suffices. The big model verifies k draft tokens as a batch in one forward pass.

Handoff is a pointer on the coherent fabric. Draft writes k token IDs into a shared buffer; GPU reads without copy.

Why Frameworks Like vLLM Can't Easily Do This

Python's GIL serializes bytecode process-wide. Tokenization, scheduling, sampling, streaming all contend for one lock. 72 cores, one running Python at a time.

Workarounds, none clean:

  • Multiple processes. Per-process GIL at the cost of the shared address space — marshalling or duplication. Coherent memory defeated.
  • C extensions. nogil C code releases the GIL for one call, but the scheduler and orchestrator stay in Python.
  • asyncio. Single-threaded cooperative concurrency. Any CPU work blocks all coroutines.
  • Python 3.13 free-threaded. Immature for production, significant single-thread penalty.

Result: vLLM, TensorRT-LLM's Python bindings, and every Python-hosted engine runs single-threaded host. 71 Grace cores are wallpaper.

Lithos is Forth/Rust. No GIL, real threads on every core.

What This Buys in Practice

  • Startup time is max, not sum. 7 kernels that each take 40–92 ms to compile finish in 92 ms wall-clock, not ~500 ms. The 4 shard files of an 18 GB model mmap and parse concurrently in a single slow-shard's time.
  • GPU utilization rises. The GPU is not waiting on tokenization, sampling, or KV bookkeeping — those are happening on other cores in parallel with the next forward pass. Batch=1 bandwidth utilization can climb toward 70%+ because the gap between kernels is filled by overlapped CPU work, not by an idle CPU running Python.
  • Request throughput scales with cores, not with orchestrator rate. Ten concurrent requests means ten orchestrator threads, each fully parallel with the others until they meet at the GPU.
  • Speculative decoding becomes cheap. The draft runs on cores the GPU wasn't using anyway. Acceptance checks happen in the shared buffer.

Closing

The GH200 is a two-compute-unit machine where one unit happens to be 132 SMs and the other happens to be 72 ARM cores. They share memory, share cache coherence, and share a thermal envelope. Treating the CPU as a scheduler that exists to launch kernels leaves most of the machine unused.

Lithos is designed for what the hardware actually is: a peer-to-peer system with a fast coherent link between a wide parallel accelerator and a wide multicore host. The language compiles kernels that run on the GPU; the host fans out across every Grace core it can reach. Single-threaded design is the exception, not the default.

Lithos — from lithography. Writing on stone. The process that puts circuits on silicon. Lithos writes on the GPU.
