Registers

The fastest memory in the system. Where the work actually happens.

The File

256 KB of SRAM per SM, partitioned four ways — one 64 KB slice per sub-partition. Each slice holds 16,384 entries, each 32 bits wide. A register is named by index: R0, R1, ..., R254. RZ is the zero register.

On Hopper the register file is notionally 4 banks wide on each sub-partition. Two operands can be read in one cycle (one tick of the 1.83 GHz SM clock, ~0.55 ns) without conflict if they land in different banks. Three-source instructions (FFMA, IMAD) that pick all three operands from the same bank incur a bank conflict — one extra cycle per conflict.

Register Allocation

A warp owns 32 register indices — one per lane. When SASS writes R5, it is really 32 physical entries (one per lane in the warp), at index 5 within that warp's register window.

The maximum indices per thread is 255 (the sixth through eighth bits of a source operand select special registers). When the compiler says "this kernel uses 80 registers per thread", it means each warp consumes 32 * 80 = 2,560 entries out of the 16,384 available in the sub-partition.

Bank Conflicts

SASS instructions encode up to 4 source operands. The scoreboard and operand-collector logic tries to read them in one cycle. If two sources map to the same bank with different addresses, one stalls — adding a cycle.

Example:

FFMA R8, R0, R4, R12  // R0 bank 0, R4 bank 0, R12 bank 0  -> 2 conflicts
FFMA R9, R1, R4, R13  // R1 bank 1, R4 bank 0, R13 bank 1  -> 0 conflicts

The compiler handles this by assigning destinations to interleave banks, but at high register pressure it runs out of options. Hopper's SASS also carries a reuse flag:

FFMA R8, R0.reuse, R4, R12

The .reuse flag tells the operand collector to cache R0 in an 8-entry operand reuse buffer, so the next instruction reading R0 skips the register file entirely — no bank access, no conflict. ptxas emits .reuse whenever a register is read by the next few instructions.

Occupancy Tradeoff

The sub-partition has 16,384 registers. Each resident warp consumes 32 * registers_per_thread of them. Max 16 warps per sub-partition. So the register count per thread is a hard cap on how many warps can live on an SM at once.

Occupancy sounds like it should always be high, but it is a means not an end. What matters is whether the warp scheduler has a ready warp every cycle. For compute-bound kernels (tensor-core heavy), even 25% occupancy can saturate. For memory-bound kernels, low occupancy leaves the pipe unfed.

Fewer Threads, More Registers

Counter to intuition: for some memory-bound kernels, running fewer threads with more registers each is faster.

Why? Two reasons:

  1. Larger tiles per thread. If each thread holds a 16x16 output tile in registers (256 registers), it performs 256 multiplies per memory load. Doubling threads while halving tile size doubles loads but quarters the compute-per-load ratio.
  2. Lower memory traffic. Weights loaded into registers are used many times. A smaller number of threads with bigger accumulator tiles re-reads fewer weights.

CUTLASS GEMMs routinely ship with 128-160 registers per thread. CUDA cores sit at ~25% occupancy. Tensor cores stay saturated because each warp has enough accumulator registers to issue wgmma back-to-back without waiting.

Lithos kernels follow this: decode matmuls are tuned for register tile size first, thread count second. A Qwen 27B QKV projection uses 168 registers per thread at 8 warps per sub-partition — well below max occupancy, well above peak throughput.

Spilling

If the compiler cannot fit a kernel's live values in 255 registers, it spills to local memory — which is physically L1 but is per-thread private. Spills are death. Every spill is a load from L1 on the next read, ~30 cycles (~16 ns) minimum versus a 1-cycle register read, often contending with shared memory access. If the spill misses L1 and falls through to HBM, each reload costs ~400 cycles (~220 ns at 1.83 GHz).

ptxas prints a warning when it spills:

ptxas info    : 128 bytes stack frame, 96 bytes spill stores, 112 bytes spill loads

Lithos treats any spill as a bug — fix the kernel, reduce tile size, split the kernel in two, or use __launch_bounds__ to give ptxas a tighter budget.

What Lithos Controls

  • __launch_bounds__(128, 2) — max 128 threads, min 2 blocks per SM. Forces ptxas to stay under 128 registers/thread.
  • Tile size choice in the kernel source — determines live register count.
  • Loop unroll depth — more unroll = more live values = more registers.
  • Explicit .reuse hints via inline SASS for the tightest inner loops.

The register file is where the speed comes from. Lithos writes kernels to keep the hot operands there, stream the rest through L1, and never let ptxas spill.
