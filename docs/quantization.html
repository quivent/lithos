Hardware-Matched Quantization

Designing the quantization scheme from the silicon up: fit one layer on-chip, utilize every compute unit.

This page describes a research direction, not an implementation. The approach outlined here is a design space that Lithos is uniquely positioned to explore, because Lithos compiles kernels on the fly. Existing engines cannot try new quantization formats without months of kernel engineering. Lithos generates the dequantization kernel from a format specification — change the spec, recompile in microseconds, measure quality and speed.

The Premise

Every existing quantization method — GPTQ, AWQ, AQLM, QuIP#, EXL2, GGUF — is designed for quality preservation and storage efficiency. Some also target compatibility with existing CUDA kernels. None are designed for two properties that matter at the hardware level:

  1. Fitting exactly in the on-chip memory of a specific GPU.
  2. Balancing dequantization work across all compute unit types on the SM.

The Lithos approach: the quantization format is a function of the target silicon, not a property of the model alone.

Part 1: Fitting a Layer On-Chip

At batch=1, inference is bandwidth-bound. The GPU reads ~92 MB of weights per layer from HBM at 400 cycles per access. But the SM has on-chip memory that is drastically faster:

On-chip resource Per SM 132 SMs total
Register file 256 KB 33 MB
L1 / shared memory 256 KB 33 MB
Total on-chip 66 MB

If one layer's quantized weights fit in 66 MB, they load from HBM once, sit on-chip, and every operation within that layer reads from on-chip memory (1–23 cycles instead of 400 cycles). Inference flips from bandwidth-bound to compute-bound.

The math

Qwen 3.5-27B per layer (approximate):

  • MLP: 3 × hidden_dim × intermediate = 3 × 5120 × 17408 = 267M weights
  • Attention / DeltaNet projections: ~4 × 5120 × 5120 = 105M weights (varies by layer type)
  • Total per layer: ~180M weight values

At different quantization levels:

Bits/weight Bytes/weight Layer size Fits in 66 MB?
16 (FP16) 2.0 360 MB No
8 (FP8/INT8) 1.0 180 MB No
4 (W4) 0.5 90 MB No
3 (W3) 0.375 67.5 MB Nearly
2.93 0.366 66 MB Exact fit
2 (W2) 0.25 45 MB Yes, room to spare

Target: ~2.93 bits per weight average across a layer. This is achievable with existing research:

  • AQLM achieves 2-bit quantization with learned codebooks.
  • QuIP# achieves 2-bit with lattice quantization.
  • EXL2 does mixed precision per layer, targeting any average.

Per-layer adaptive precision

Not all layers need the same number of bits. Early layers (more important for output quality) get 3–4 bits. Late layers (more redundant) get 2 bits. The average hits 2.93. Each layer's total fits in 66 MB. Quality is preserved where it matters most.

What changes

Instead of streaming weights from HBM for every matmul, you load one layer's weights into distributed shared memory across 132 SMs. All operations within that layer — norm, projections, attention, MLP — read from on-chip. Only between layers does the memory bus activate to swap in the next layer's weights.

The result

The memory bus is used 64 times per token (once per layer swap) instead of ~448 times (once per operation). Each use transfers 66 MB. Total memory traffic per token: 64 × 66 = ~4.2 GB. Compare to current: ~8 GB at W4. Half the traffic and no latency stalls within a layer.

Part 2: Utilizing Every Compute Unit

The SM has four types of compute units. Current inference kernels use almost exclusively tensor cores (for matmul) and a few FP32 cores (for elementwise ops). Three-quarters of the silicon sits idle during most operations.

Hopper SM compute units (per SM)

Unit type Count per SM Peak ops/clock
FP32 cores 128 128 FLOPs
FP64 cores 64 64 FLOPs
INT32 cores 64 64 ops
Tensor cores 4 1,024 FP16 FLOPs / 2,048 FP8 ops

The insight

Design the quantization format so that dequantization work maps to the idle units, pipelined with tensor core computation. INT32 cores extract packed integers. FP32 cores apply scale factors. Tensor cores consume the dequantized tiles. FP64 cores optionally accumulate results at high precision.

Pipelined execution within one SM

Clock N: INT32 cores: extract 4-bit nibbles from packed weights for tile T+2 FP32 cores: multiply extracted values by scale factors for tile T+1 Tensor cores: matrix-multiply dequantized tile T against activations FP64 cores: high-precision accumulation for tile T-1 (reduces rounding error)

Four pipeline stages, four unit types, all running simultaneously on different tiles. The output of INT32 feeds FP32, which feeds tensor cores, which feeds FP64 accumulation.

Designing the format for pipeline balance

The quantization format should produce work that balances across unit types according to their throughput ratios.

For one tile (64×16 weights = 1,024 values at 2.93 bits):

  • INT32 work: Extract ~3,000 bits of packed data into integer values. 64 INT32 cores at 1 op/clock: ~47 cycles of extraction work.
  • FP32 work: Apply scale factors to 1,024 values. 128 FP32 cores at 1 op/clock: ~8 cycles of scaling work.
  • Tensor core work: 64×16 × input_dim matmul. 4 tensor cores at 1,024 FLOPs/clock: depends on tile shape.

The format design goal: INT32 extraction time approximately equals FP32 scaling time approximately equals tensor core consumption time. If INT32 takes too long, simplify the packing. If FP32 is too fast, add more scale factors (finer granularity). Tune the format until the pipeline is balanced.

A possible format

A hardware-matched quantization format might look like:

  • Weights packed as 3-bit values with a 1-bit residual in a separate stream.
  • Per-16-element scale factors (finer than typical per-128 grouping — gives FP32 cores more work).
  • INT32 extraction uses shift+mask (native INT32 ops).
  • FP32 converts and scales (native FP32 ops).
  • Tensor cores consume the result as FP16 tiles.
  • FP64 cores accumulate the tile result for numerical precision (or unused if not needed).

What Has Not Been Tried

Every existing quantization method (GPTQ, AWQ, AQLM, QuIP#, EXL2, GGUF) is designed for:

  1. Quality preservation (minimize perplexity loss).
  2. Storage efficiency (fewer bits).
  3. Possibly: compatibility with existing CUDA kernels.

No existing method is designed for:

  • Fitting exactly in on-chip memory of a specific GPU.
  • Balancing work across all compute unit types.
  • Pipeline-stage alignment between INT32/FP32/TC/FP64.

Feasibility

Sub-3-bit quantization is proven feasible. AQLM and QuIP# achieve 2-bit quantization with acceptable quality for many tasks. The question is whether the specific bit layout — the way values are packed and scales are arranged — can be tuned for pipeline balance without sacrificing quality.

The quality comes from the calibration method (how you choose which values to round up vs. down). The layout is independent of quality — it determines how quantized values are arranged in memory, not what those values are.

The approach separates these concerns:

  • Calibrate with AQLM/QuIP# methods for quality.
  • Pack the calibrated values in a layout designed for INT32/FP32/TC pipeline balance.

Same quality, different bytes-on-disk arrangement. The model does not change. The bits-per-weight does not change. Only the packing order and scale granularity change.

Why Lithos can explore this. Existing inference engines (vLLM, TensorRT-LLM, llama.cpp) cannot try new quantization formats without months of kernel engineering. Each format requires hand-written CUDA kernels for dequantization, tiling, and matmul integration. Lithos generates the dequantization kernel from a format specification — change the spec, recompile in microseconds, measure quality and speed. The search over format parameters (group size, bit split, packing order) becomes an empirical optimization loop rather than an engineering project.
