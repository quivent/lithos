Architecture Roadmap

Six work items. Two parallel tracks. Zero abstraction layers when complete.

Overview

Status: the self-hosting compiler is DONE. All six compiler components are written in Lithos (.ls), totalling 6,125 lines across lexer, parser, GPU emitter, ARM64 emitter, ELF wrapper, and safetensors reader. The compiler compiles itself. Remaining work: model-to-binary templating (item 2/3), megakernel linking (item 4), and the hardware track (items 5–6).

Lithos reaches its endgame through six architecture work items on two independent tracks. The compiler track (items 1–4) transforms a safetensors model file into two cooperative megakernel ELF binaries. The hardware track (items 5–6) eliminates C and libcuda.so by booting the GPU and submitting work directly from the self-hosted compiler binary.

When both tracks complete, the entire system is: a native ARM64 binary (itself compiled from Lithos .ls sources) that reads a safetensors file, compiles it into two megakernels, boots the GPU via BAR0/BAR4 register writes, and runs inference by writing GPFIFO entries and ringing a doorbell. No Python. No C. No CUDA. No kernel module. No runtime weight loading.

Dependency Graph

1. Safetensors Parser
5. GSP Boot from Lithos
2. Template Language (.ls v2)
6. GPFIFO + Doorbell
3. Model-to-Binary Compiler
4. Megakernel Linker

Compiler track: 1 → 2 → 3 → 4 (strictly serial). Hardware track: 5 → 6 (serial within track, parallel with compiler track). The two tracks converge when the linker output (item 4) is launched via the doorbell (item 6).

1. Safetensors Parser Compiler Track Done

Status: DONE. Implemented as compiler/lithos-safetensors.ls (617 lines). The compiler reads model weight files at compile time via the self-hosted Lithos safetensors reader.

The compiler must read model weight files at compile time. The safetensors format is straightforward binary: an 8-byte little-endian header length, a JSON header mapping tensor names to metadata, and contiguous raw tensor data. The parser runs in the Forth bootstrap interpreter during compilation.

Format Structure

Byte 0..7      8-byte LE integer: header_size
Byte 8..8+N-1  JSON header (UTF-8, N = header_size)
Byte 8+N..EOF  Raw tensor data (contiguous, no padding)

JSON header entry:
  "model.layers.0.self_attn.q_proj.qweight": {
      "dtype": "I32",
      "shape": [5120, 640],
      "data_offsets": [0, 13107200]
  }

Implementation

The parser mmaps the safetensors file, reads the 8-byte header length, scans the JSON header to build a dictionary of tensor names to (offset, nbytes, dtype, shape), and exposes a lookup word. For any tensor name, it returns the base pointer plus offset into the mmap region.

Key Details

  • Binary format — no external dependencies
  • JSON header is simple key-value (no nesting beyond one level)
  • mmap via Forth file words or thin SYS_mmap syscall wrapper
  • Output: ( tensor-name -- base-addr offset nbytes dtype shape )
  • Q4 GPTQ tensors have .qweight, .qzeros, .scales sub-tensors

Dependencies

  • Forth interpreter with file I/O (exists)
  • SYS_mmap word (syscall 222 on aarch64, trivial to add)
  • Minimal JSON tokenizer (parse { } : , "string" number)

Creates

  • compiler/safetensors.fs — parser and tensor lookup words
  • compiler/json.fs — minimal JSON tokenizer
  • mmap Forth word wrapping the syscall

Deletes

  • Runtime weight loading from engine.py
  • LoadedModel.weight_offsets dictionary
  • Any Python safetensors dependency
\ Forth usage at compile time:
s" model.safetensors" open-safetensors
s" model.layers.0.self_attn.q_proj.qweight" tensor-lookup
\ ( -- base-addr offset nbytes dtype shape-addr )

2. Template Language (.ls v2) Compiler Track In Progress

Status: lexer (818 lines) and parser (1,931 lines) are done in compiler/lithos-lexer.ls and compiler/lithos-parser.ls. Template keywords (weight, layer, bind, project, runtime) are the remaining piece.

The .ls file format evolves from a standalone kernel description into a parameterized template. The compiler instantiates the template once per layer, binding concrete weight tensor addresses from safetensors. The same template with different weight bindings produces 48 different instruction streams — identical structure, different addresses and immediates.

New Syntax Elements

\ Weight declarations (compile-time constants from safetensors)
weight W_q     : [5120, 5120]  q4     \ Q projection, quantized 4-bit
weight w_norm1 : [5120]        f32    \ RMSNorm weight (broadcast scalar)
weight A_log   : [48]          f32    \ decay log-parameter (per-head)

\ Layer parameterization
layer L : 0..47                        \ stamp out 48 times

\ Binding (safetensors tensor paths)
bind L
    W_q     <- model.layers.{L}.self_attn.q_proj.qweight
    w_norm1 <- model.layers.{L}.input_layernorm.weight

\ Projection directive (compiler emits weight-address-hardcoded LDG stream)
project W_q x -> q

\ Runtime data declaration (activations, not weights)
runtime x : [5120] f32                 \ input hidden state

The SIMT Constraint

All 32 threads in a warp execute the same instruction with the same immediate. This means matrix weights cannot be literal instruction immediates — they must be LDG-loaded from HBM, because different threads in the warp need different weight values for different output rows. However, all weight addresses, strides, dequantization scale factors, zero points, and broadcast scalars (norm weights, decay, beta) are compiled in. No runtime decisions, no pointer parameters, no cbuf0 parameter passing.

Compile-Time Constant Folding

When a computation involves only compile-time weights, the compiler evaluates it and emits the result as a single immediate. Examples:

ExpressionRuntime InstructionsCompiler Action
sigmoid(beta) (steps 23–26) 0 Pre-computed at compile time, emitted as MOV-IMM per head
-exp(A_log) (steps 31–32) 0 Pre-computed, becomes one FMUL-IMM immediate
a_proj + dt_bias (step 27) 1 FADD-IMM dt_bias is compile-time, a_proj is runtime (depends on input)
x * w_norm (step 6) 1 FMUL-IMM per element Broadcast scalar — same value for all 32 warp lanes

Key Insight

  • Matrix weights: LDG with hardcoded addresses (SIMT constraint)
  • Broadcast scalars: FMUL-IMM / FADD-IMM immediates
  • Weight-only expressions: constant-fold to zero runtime instructions
  • Every value is either weight (compile-time) or runtime (activation)

Dependencies

  • Item 1 (safetensors parser) — must be able to read tensors
  • parser.fs — extended with new keywords

Creates

  • Extended parser.fs with weight, layer, bind, project, runtime
  • Template instantiation loop in compiler
  • Compile-time constant folding for weight expressions

Deletes

  • param W_packed ptr syntax
  • Indexed weight loads (W_packed [ idx ])
  • Runtime dequantization in .ls source
  • The for k_p ... endfor weight reduction loop

3. Model-to-Binary Compiler Compiler Track In Progress

Status: GPU emitter (compiler/emit-gpu.ls, 1,016 lines) and ARM64 emitter (compiler/emit-arm64.ls, 1,067 lines) are done. The emit-project weight-immediate specialization is the remaining piece.

The project primitive's emit word changes completely. Instead of emitting a parameterized GEMV loop, it reads the Q4 weight tensor from safetensors at compile time and emits a straight-line instruction stream with all addresses hardcoded.

What the project Emit Word Does

: emit-project  ( weight-tensor activation-reg output-reg -- )
    \ 1. Read Q4 weight tensor from safetensors (at compile time)
    \ 2. For each output row assigned to this thread block:
    \    a. Emit LDG instructions with hardcoded .rodata addresses
    \       for the packed Q4 weight words of that row
    \    b. Emit dequant sequences with per-group scale factors
    \       and zero points as FMUL-IMM / FADD-IMM immediates
    \    c. Emit FFMA accumulation against activation registers
    \    d. Emit shuffle-tree reduction (unchanged from current)
    \ 3. No loops over weight groups at runtime
    \    The compile-time loop is unrolled with concrete addresses
;

Compilation Model

The compiler becomes a model-to-binary translator. It reads safetensors once, instantiates the .ls template once per layer (binding different weight tensors each time), and outputs per-layer instruction streams. Each stream is a self-contained ELF section.

safetensors + .ls template
      |
      v
  lithos-compile  (Forth bootstrap)
      |
      +-- layer_00.text  (all projections + elementwise + reduction for layer 0)
      +-- layer_01.text
      +-- ...
      +-- layer_47.text
      +-- layer_00.rodata  (Q4 packed weight data with known offsets)
      +-- ...

The resulting binary can only run one specific model. The model IS the program. A different model requires a different compilation.

Key Details

  • Weight addresses hardcoded as LDG operands — no pointer parameters
  • Per-group scale factors embedded as FMUL-IMM immediates
  • Zero points embedded as FADD-IMM immediates
  • Dequantization happens at compile time for broadcast scalars, at runtime (with hardcoded constants) for matrix weights
  • The compile-time loop over weight groups is unrolled into a straight-line instruction stream

Dependencies

  • Item 1 (safetensors parser) — tensor data access
  • Item 2 (template language) — project directive parsing
  • gpu/emit.fs — existing instruction emitter
  • gpu/opcodes-sm90.fs — existing opcode definitions

Creates

  • New emit-project word in gpu/emit.fs
  • Per-layer ELF .text and .rodata sections
  • Compile-time dequant pipeline for Q4/Q8 formats

Deletes

  • emit-gemv.fs parameterized GEMV loop emitter
  • cbuf0 parameter passing for weight pointers
  • Runtime scale/zero-point buffer setup
  • LoadedKernels dispatch table

4. Megakernel Linker Compiler Track In Progress

Status: ELF wrapping is done in compiler/lithos-elf.ls (676 lines). Per-layer concatenation and grid-sync insertion remain.

The per-layer instruction streams from item 3 must be linked into two cooperative megakernels. The megakernel IS the model. One QMD dispatch, one doorbell write per token.

Two Megakernels

MegakernelContentsDispatch
Forward pass All 64 layers (48 DeltaNet + 16 full-attention): projections + elementwise + reductions + attention + delta rule One QMD per token
Recurrence update State matrix updates across all 48 DeltaNet layers One QMD per token

Linking Process

layer_00.text + layer_01.text + ... + layer_47.text
      |
      v
  megakernel-link
      |
      +-- forward_megakernel.elf
      |     .text = concatenation of all layer .text sections
      |     .rodata = concatenation of all layer weight data
      |     grid-sync (ATOMG + BAR) between each layer
      |
      +-- recurrence_megakernel.elf
            .text = state update code for all layers
            grid-sync between layers

Grid-sync instructions (ATOMG + BAR) are emitted between layers. The linker concatenates per-layer code sections, resolves cross-layer address references (next layer's activation buffer = this layer's output buffer), and produces two final ELF binaries.

Key Details

  • Concatenation, not compilation — per-layer code is already final SASS
  • Grid-sync barriers (ATOMG + BAR) inserted between layers
  • Cross-layer references: activation buffer addresses resolved at link time
  • 528-byte QMD descriptor built for each megakernel

Dependencies

  • Item 3 (model-to-binary compiler) — per-layer ELF sections
  • compiler/elf-wrap.fs — ELF packaging (exists)

Creates

  • compiler/linker.fs — megakernel linker
  • Two final ELF binaries per model
  • Pre-built QMD descriptors embedded in the ELF

Deletes

  • Per-kernel .cubin files (inference/*.cubin)
  • Per-kernel launch overhead (QMD construction per kernel)
  • Inter-kernel synchronization via CPU
  • The kernel dispatch table concept entirely

5. GSP Boot from Lithos Hardware Track

Status: not started. Currently lithos.ko (C kernel module) handles GSP boot; this item replaces that path with Lithos code running entirely in userspace over vfio-pci.

The GH200's BAR0 (16 MB GPU registers) and BAR4 (128 GB coherent HBM) can both be mmap'd from userspace via vfio-pci — a stock Linux kernel module, zero custom code. The entire GSP boot sequence reduces to register writes via BAR0 and structure population in BAR4.

The BAR Mapping

BARSizeContentsAccess Method
BAR0 16 MB GPU MMIO registers (PMC, Falcon, FSP, USERD) vfio-pci mmap or sysfs resource0
BAR4 128 GB Coherent HBM (PA == GPU VA, hardware cache coherent via C2C/ATS) vfio-pci mmap or sysfs resource4

Boot Sequence (All Userspace)

Phase 0:  mmap BAR0 + BAR4 via vfio-pci (3 syscalls)
Phase 1:  Read PMC_BOOT_0 at BAR0+0x000000, verify Hopper (1 load)
Phase 2:  Falcon reset: BAR0+0x1103C0 assert/deassert + poll (~4 stores)
Phase 3:  Populate FMC params, WPR metadata in BAR4 (bump-allocate)
Phase 4:  Load GSP firmware ELF from /lib/firmware/ into BAR4
Phase 5:  Program RISC-V BCR registers (6 BAR0 register pairs + start)
Phase 6:  FSP boot commands via BAR0+0x8F2000 EMEMC/EMEMD (~2000 lines)
Phase 7:  Poll BAR0+0x1100F4 bit 13 until PRIV_LOCKDOWN releases

BAR4 replaces dma_alloc_coherent() entirely. On GH200, CPU physical addresses equal GPU virtual addresses for BAR4 mappings. CPU stores to BAR4 are GPU-visible without explicit cache flushes. The kernel module's DMA allocation was unnecessary overhead.

Key Details

  • BAR4 IS coherent memory: PA == GPU VA, no IOMMU needed
  • All DMA buffers bump-allocated from BAR4 (same as lithos_mem.c)
  • FSP communication (~2000 lines) is pure MMIO writes to BAR0
  • No interrupts needed: polling is optimal for single-tenant inference
  • Entire sequence is ~500 ARM64 instructions or ~800 lines of Forth

Dependencies

  • vfio-pci kernel module (ships with Linux, zero custom code)
  • GPU unbound from nvidia.ko (already required)
  • Forth interpreter with SYS_mmap and file I/O

Creates

  • gpu/bar.fs — BAR0/BAR4 mmap and register access words
  • gpu/gsp-boot.fs — full GSP boot sequence in Forth
  • gpu/fsp.fs — FSP communication protocol
  • gpu/hbm-alloc.fs — bump allocator over BAR4

Deletes

  • kernel/lithos_main.c — PCI driver, char device, ioctl
  • kernel/lithos_gsp.c — GSP boot (ported to Forth)
  • kernel/lithos_mem.c — allocator (simplified to Forth)
  • kernel/lithos_init.c — init wrapper
  • All kernel build artifacts (Makefile, .ko, .o, Module.symvers)
  • All linux/ kernel headers

6. GPFIFO + Doorbell from Lithos Hardware Track

Status: not started. Depends on item 5.

After GSP boots and the RISC-V coprocessor is running, the Forth bootstrap creates a GPFIFO channel via GSP RPC, builds the QMD pushbuffer, and runs inference with zero syscalls per token.

Channel Creation

1. Send ALLOC_ROOT / ALLOC_DEVICE / ALLOC_CHANNEL RPCs via GSP message queue
2. GSP allocates channel resources on the GPU side
3. Bump-allocate GPFIFO ring buffer from BAR4 (8 KB)
4. Map USERD page (BAR0 + 0xFC0000 + chid*0x200, already in BAR0 mmap)

Work Submission (Per Token)

1. Update activation buffer    CPU write to BAR4 (hidden state)
2. Build 528-byte QMD          block/grid dims, entry_pc, register count, smem size
3. Write QMD to GPFIFO ring    CPU store to BAR4
4. Write GPPut                 store to USERD+0x8C (advance ring pointer)
5. Ring doorbell               one 32-bit store to USERD+0x90 (GPU starts)
6. Poll completion             read fence value from BAR4 (GPU writes when done)
7. Read output token           read BAR4 (argmax result)

Zero syscalls at inference time. The token loop is three memory writes (activation + GPFIFO entry + doorbell) and two memory reads (fence + output). Total latency is dominated by GPU execution, not submission overhead.

Key Details

  • 528-byte QMD: block dims, grid dims, entry_pc, register count, shared memory
  • GPFIFO ring: circular buffer of 8-byte entries pointing to QMD pushbuffers
  • Doorbell: single 32-bit store at USERD+0x90 triggers GPU fetch
  • Completion: poll a BAR4 fence value (GPU writes when megakernel finishes)
  • No context switching, no preemption, no multi-tenant scheduling

Dependencies

  • Item 5 (GSP boot) — GPU must be initialized
  • GSP RPC protocol for channel allocation
  • BAR0 mmap for USERD access
  • BAR4 mmap for GPFIFO ring and activation buffers

Creates

  • gpu/channel.fs — channel creation via GSP RPC
  • gpu/gpfifo.fs — ring buffer management + QMD construction
  • gpu/doorbell.fs — submission and completion polling

Deletes

  • kernel/lithos_channel.c — kernel-mode channel creation
  • src/cuda_driver.py — Python CUDA driver wrapper
  • All ioctl-based GPU communication
  • libcuda.so dependency entirely

Execution Timeline

Ordering by dependency, not calendar. The two tracks are independent and can proceed in parallel.

Phase 1: Safetensors Parser
Add SYS_mmap to Forth. Parse safetensors header. Look up any tensor by name. Prove: read a weight matrix from Forth.
Phase 1: BAR Access
Unbind nvidia.ko, bind vfio-pci. mmap BAR0 from Forth. Read PMC_BOOT_0. Prove: Forth can read GPU registers.
Phase 2: Template Language
Extend parser.fs with weight, layer, bind, project, runtime keywords. Parse a full DeltaNet .ls v2 template.
Phase 2: GSP Boot
Map BAR4. Port Falcon reset, FMC bootstrap, FSP communication, message queues. Prove: GSP reaches ready state from Forth.
Phase 3: Model-to-Binary Compiler
Implement emit-project: read Q4 tensor, emit LDG with hardcoded addresses, embed scales as immediates. Compile one layer.
Phase 3: Channel + Doorbell
Create GPFIFO channel via GSP RPC. Build QMD. Write GPFIFO entry. Ring doorbell. Prove: launch a test kernel from Forth.
Phase 4: Megakernel Linker
Concatenate 48 layer .text sections. Insert grid-sync barriers. Resolve cross-layer addresses. Produce two final ELF binaries.
Convergence: First Token
Megakernel ELF (compiler track) launched via doorbell (hardware track). First token generated with zero Python, zero C, zero CUDA.
