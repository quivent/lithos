Memory

Why mmap replaces cudaMalloc on GH200.

The Old Model (Every Other Engine)

CUDA assumes separate memory spaces. cudaMalloc for the GPU. malloc for the CPU. cudaMemcpy to move data between them. This model was designed for discrete GPUs connected over PCIe, where CPU and GPU memory are physically separate chips on separate boards.

PyTorch's CUDACachingAllocator

PyTorch wraps cudaMalloc in a caching allocator that manages memory pools, handles fragmentation, runs garbage collection, and provides memory profiling. Thousands of lines of C++ to manage memory that the GPU never directly controls.

vLLM's KV Cache

Pre-allocated as one contiguous cudaMalloc block. Size determined by profiling free memory at startup. Fixed for the lifetime of the server. Want to resize? Restart. That takes ~2 minutes.

What cudaMalloc Actually Does

Internally, cudaMalloc issues an ioctl to /dev/nvidiaX. The kernel driver manipulates GPU page tables. Each call costs 10–500µs depending on allocation size and fragmentation state. The driver maintains separate page tables for a memory separation that, on GH200, does not physically exist.

The Result

Complex memory management code. Fragmentation problems. OOM errors on machines with plenty of physical memory. Restart-to-resize. All to maintain the fiction of separate address spaces on hardware that shares one.

The GH200 Reality

Hardware-coherent unified memory via NVLink-C2C at 900 GB/s.

CPU and GPU share the same physical address space. Any pointer that the CPU can read, the GPU can read. The hardware handles coherence — no software intervention, no explicit transfers, no synchronization primitives for memory consistency.

cudaMalloc still works. But it is unnecessary overhead. It maintains separate page tables for a separation that doesn't physically exist. It issues kernel driver ioctls for an allocation that mmap can do in userspace.

mmap returns a pointer. Both processors can use it. That is the entire memory management story on GH200.

Lithos Memory Strategy

Weights: mmap (lazy)

fd = open("model.safetensors")
base = mmap(fd, size, PROT_READ, MAP_PRIVATE)
// No data loaded yet. Just a virtual mapping.
// First GPU access faults in pages from SSD on demand.
// After first forward pass: all pages resident.
// Pointer arithmetic gives direct access to any tensor.

Why mmap for weights: 18 GB from SSD. Don't wait for all of it. Let the first inference request start immediately. Pages load as the GPU touches them. After warmup, everything is resident at full speed.

KV Cache: Pre-allocate (eager)

kv_size = layers × kv_heads × head_dim × 2 × max_tokens × sizeof(f16)
ptr = mmap(NULL, kv_size, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANON, -1, 0)
memset(ptr, 0, kv_size)  // touch every page, force physical allocation
// ptr is fully backed. First access is full speed. No faults during inference.

Why pre-allocate for KV cache: this is hot memory hit every token. Cannot afford page faults during generation. Know the size from arithmetic: 64 layers × 4 heads × 256 dim × 2 bytes × 2 (K+V) = 262 KB per token. Want 300K tokens? Pre-allocate 77 GB.

Activations: Pre-allocate (eager)

act_size = hidden_dim × sizeof(f16) × num_buffers
ptr = mmap(NULL, act_size, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANON, -1, 0)
memset(ptr, 0, act_size)

Fixed size. Known at model load. Pre-allocate. Zero runtime allocation cost.

Why mmap + Pre-allocate Together

LAZY mmap for things loaded from disk (weights)

Large and loaded from SSD. Lazy is fine — startup is fast. Pages fault in on first access. After one forward pass, fully resident.

EAGER Pre-allocate for things written during inference (KV cache, activations)

Cannot tolerate page faults during token generation. Touch every page at allocation time. First inference access is full speed.

Both use mmap as the underlying mechanism. One maps a file. The other maps anonymous memory. No cudaMalloc anywhere. No CUDACachingAllocator. No memory pools. No fragmentation management. No garbage collection.

Why No One Else Does This

The CUDA programming model assumes separate memory spaces. It was designed for discrete GPUs where CPU and GPU memory are physically different chips. Every abstraction in the ecosystem — cudaMalloc, cudaMemcpy, unified memory with managed hints — exists to bridge that gap.

PyTorch, TensorRT, vLLM: all built on cudaMalloc because it works on every NVIDIA GPU from the GTX 1080 to the H100. Correctness across hardware generations is the priority.

GH200 broke the fundamental assumption — memory is no longer separate — but the frameworks have not caught up. They cannot special-case for GH200 without maintaining two complete codepaths: one for discrete GPUs, one for unified memory. The engineering cost is prohibitive when GH200 is one SKU among dozens.

Everyone builds on PyTorch. So everyone inherits PyTorch's memory model. Even vLLM, which rewrites the serving layer, still uses PyTorch tensors backed by cudaMalloc under the hood.

Lithos has no legacy. No backward compatibility requirement. No discrete GPU support to maintain. Built for this machine from scratch. The memory model follows from the hardware, not from twenty years of CUDA convention.

The KV Cache Problem Solved

vLLM's KV cache is fixed at startup because cudaMalloc cannot resize. The block is contiguous. Growing it means allocating a new, larger block, copying, and freeing the old one. In practice, you restart the server.

Lithos KV cache can grow: mmap more anonymous memory. The OS finds physical pages. No copy. No restart. Milliseconds.

The GH200 Advantage

On GH200, the KV cache can seamlessly span both memory tiers:

HBM3e: 96 GB @ 4 TB/s ← hot KV cache (recent tokens) LPDDR5X: 480 GB @ 500 GB/s ← cold KV cache (old context) Total addressable: ~480 GB = ~1.8 million tokens of context

Hot pages — recent tokens accessed every generation step — stay in HBM. Cold pages — old context rarely touched — drift to LPDDR5X. The hardware manages this automatically. No code needed. No "offloading strategy." No tiered caching logic.

Total addressable KV cache: ~480 GB. At 262 KB per token, that is ~1.8 million tokens of context. Without a single line of memory management code.

First access to a page: the OS places it. GPU access favors HBM. CPU access favors LPDDR5X. Subsequent accesses can trigger hardware migration — pages move to whichever tier is accessing them most frequently.

cudaMemAdvise can hint placement for known access patterns. Or let the hardware migrate automatically. Either way: no explicit transfers. Just pointers.
