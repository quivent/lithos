The Lithos Compiler

Primitives to GPU machine code. A macro assembler, not an optimizer.

The Decision

The Lithos language consists of .ls source files. The Lithos compiler is implemented in bootstrap Forth (.fs files) — this is an implementation detail, not part of the language surface.

The compiler is a macro assembler, not a traditional optimizer. Each primitive family maps to a fixed, known-good binary pattern. There is no optimization pass, no register allocator guessing, no IR lowering. The encoding table is empirically verified against live GH200 hardware via ptxas + nvdisasm probing. The compiler runs on forth-bootstrap — an ARM64 Linux assembly Forth interpreter, 4,831 lines of ARM64 assembly. When Eighth self-hosts, the compiler compiles itself to a native ARM64 binary.

The choice is ideological and practical.

Ideological

Lithos is Forth-derived. The compiler should be Forth. Writing it in any other language breaks the stack-machine thesis the language is built on. If the compiled surface is concatenative stack code, the compiler that emits it has no reason to be anything else.

Practical

Forth has no runtime, no garbage collector, no framework. A Forth compiler binary is ~60 KB. Startup is instant. Memory footprint is kilobytes. There is no interpreter warm-up, no import graph, no dependency tree, no venv, no Cargo.lock.

Future

Eventually self-hosting — the Lithos compiler compiles itself. Zero external dependencies. The seed is a small ARM64 assembly file; everything above that is Forth.

GPU backend (weights-as-code)

Emits sm90a GPU machine code (64-bit instruction words) into an ELF binary. The compiler reads model.safetensors at compile time: Q4 weight values are encoded as instruction immediates in FMUL-IMM instructions. Each layer's weight matrix becomes a compiled instruction stream where the immediate fields ARE the packed weights. The resulting ELF .text section is the model — there is no separate weight tensor, no weight pointer parameter, no runtime LDG for weights. Hardware encoding empirically verified from live GH200 probing via ptxas + nvdisasm. All opcodes confirmed for: FP32 arithmetic, SFU (MUFU), memory (LDG/STG/LDS/STS/ULDC), integer/atomic (IMAD/LOP3/ATOMG). No ptxas, no CUDA driver, no cubin NVIDIA sections required.

ARM64 backend

Emits ELF executables for the ARM64 launcher: GPFIFO ring writes, doorbell writes, lithos.ko ioctls, and CPU-side sampling. Reuses Eighth's ARM64 encoding tables.

One compiler. One source language. Two outputs. No CUDA anywhere. Weights are compiled into instruction immediates — the model becomes the program.

Weights as Code

The compiler reads model.safetensors at compile time. For each layer, it extracts the Q4 weight matrix, and emits a stream of 64-bit FMUL-IMM instructions where the immediate fields encode the packed 4-bit weight values. The output is a per-layer instruction stream that, when executed, multiplies the activation vector by the weight matrix — with no GEMV loop, no weight pointer, and no LDG instruction.

Compile-time inputs

  • model.safetensors — weight values read at compile time, encoded into instruction immediates
  • config.json — model dimensions determine the instruction stream shape
  • .ls source — primitive families determine the instruction patterns

What the compiler emits per layer

For a projection with weight matrix W of shape [out_dim x in_dim] at Q4:

  • Each row of W becomes a sequence of FMUL-IMM instructions — the activation register is multiplied by each weight immediate in sequence
  • Accumulation is interleaved via FFMA instructions
  • The total instruction count per projection = out_dim x ceil(in_dim / values_per_instruction)
  • No loop counter, no weight address register, no LDG — the instruction stream IS the loop

The ELF output

The ELF .text section contains the weight-immediate instruction streams for all layers. The .text section IS the model weights. There is no .data section for weight tensors. At inference time, the GPU's instruction cache hierarchy (L1I → L2 → HBM) serves as the weight cache.

Why Not Other Languages

Every alternative considered, and why it was rejected.

Python

Wrong layer. Framework overhead, GIL, interpreter, the entire class of problem we are eliminating. Writing the compiler in Python is writing the problem into our solution. The thing Lithos exists to avoid cannot be the thing that builds Lithos.

Rust

Fast, yes. Native, yes. But introduces cargo, the borrow checker, a standard library — complexity that has nothing to do with compiling GPU kernels. A Rust Lithos compiler would spend more code on memory safety for its own AST than on emission. Wrong trade.

C

Would work. But C is prefix-and-infix. Our target languages (SASS, ARM64) are postfix assembly. Forth matches the target semantics; C fights them. Writing a stack-based emitter in a non-stack language means hand-rolling the stack in every function.

Gforth or other existing Forth

Someone else's interpreter. Dependency. The entire point is zero external dependencies. If a third-party Forth vanishes, breaks ABI, or gains a bug we cannot patch, the compiler breaks with it.

Eighth (native ARM64 Forth compiler)

The right answer eventually — once Eighth self-hosts. Blocked right now on the forward-reference depth bug. When fixed, the Lithos compiler compiles through Eighth to a native binary. Until then: forth-bootstrap interprets.

Language Surface

Primitive-based with dimensional notation. Lithos source (.ls) expresses computation as sequences of primitives that compile directly to GPU machine code. The implementation uses bootstrap Forth (.fs files) internally, but the language surface is Lithos primitives, not Forth syntax.

GPU kernel (→ sm90a machine code)

kernel vadd
    param: a u64
    param: b u64
    param: c u64
    param: n u32

    global-tid
    dup n >= if exit then
    dup ld.global.f32 a
    swap ld.global.f32 b
    add.f32
    st.global.f32 c
;

Each primitive dispatches to the GPU machine code emitter. Register allocation is explicit; memory spaces are explicit; the stack carries register IDs, not values. The emitter produces binary directly — no PTX text, no intermediate representation.

Host code (→ ARM64)

host main
    argc>
    1 argv slurp-file
    model-load
    engine-init
    loop-forever
;

Same word syntax. The parser sees host and routes to the ARM64 emitter. Stack operations compile to X0–X8 register stack manipulation; control flow compiles to B.cond/BL; I/O compiles to SVC 0 syscalls.

Backend routing

The router is per-definition, not per-file. A single .ls source can define several kernel words and several host words side by side. The host binary and the GPU ELF binaries it loads are built from the same source, and symbol names are shared — the launcher knows the kernels by the names they were defined under.

File Organization

compiler/
├── lithos.fs          Main entry point
├── lexer.fs           Whitespace tokenizer (thin — Forth does this natively)
├── parser.fs          Token dispatch table
├── emit-sass.fs       GPU machine code emission wrapper (loads gpu/emit.fs)
├── emit-arm64.fs      ARM64 machine code
├── elf-wrap.fs        Wraps GPU machine code into ELF binary for lithos.ko
└── arm64-wrap.fs      Wraps ARM64 into executable ELF
FileRole
lithos.fsArgument parsing, file I/O, top-level dispatch to the backend router.
lexer.fsThin wrapper over Forth's native parse-name. Lithos source is already whitespace-delimited tokens — Forth's host loop already tokenizes.
parser.fsDispatch table mapping word → handler. kernel, host, param:, if, then, ; are parser directives; everything else is emitted by the current backend.
emit-sass.fsThin wrapper that loads gpu/emit.fs and adds write-sass-raw. Raw 16-byte sm90 instruction emission — the only GPU-side emitter. Opcode constants from the probe disassembly. No driver JIT, no ptxas.
emit-arm64.fsARM64 machine code. Reuses Eighth's instruction encoders. Stack-machine model on X0–X8.
elf-wrap.fsWraps GPU machine code bytes into an ELF binary for loading by lithos.ko. The NVIDIA-specific sections (.nv.info, .nv.constant0) are emitted by build-cubin in gpu/emit.fs.
arm64-wrap.fsWraps ARM64 bytes into a standard Linux executable ELF with .text, .data, a program header, and an entry point.

Total expected size: < 3,000 lines of Forth. Small enough to read end-to-end in an afternoon. Small enough that every dispatch decision, every emission, every byte of output is traceable to a line of source.
