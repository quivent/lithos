Inference Engines

Why each is fast and where each fails.

What Makes Inference Fast

LLM inference at batch=1 is a memory bandwidth problem. The GPU must read every weight once per token. The arithmetic is trivial — a few multiply-adds per weight byte. The question is: how close to peak bandwidth can you get?

Five factors determine inference speed, in order of impact.

01 Memory Bandwidth Utilization

Are you reading weights at the physical speed limit? An H100 SXM has 3.35 TB/s HBM bandwidth. A GH200 has 4.0 TB/s. At batch=1, a 27B model in 4-bit quantization is ~14 GB of weights. At full bandwidth that is 3.5 ms per token — 285 tok/s. In practice, most engines achieve 10-30% of peak bandwidth. Every percentage point of utilization is a direct percentage point of speed.

Why is utilization so low? Global memory has ~400 cycle latency. You need hundreds of outstanding memory requests to saturate the bus. This requires careful prefetching, double-buffering, and enough concurrent warps to hide latency. Most framework-dispatched kernels don't achieve this.

02 Kernel Fusion

Are you eliminating redundant memory traffic? Every kernel boundary means: write results to HBM, synchronize, launch next kernel, read results back. If RMSNorm writes a 4096-element vector to HBM and the next GEMV reads it back, that is 32 KB of wasted traffic in each direction — and more critically, 1-5 us of dead time between kernels.

A fused kernel keeps intermediate results in registers (1 cycle access) or shared memory (~20 cycles) instead of round-tripping through HBM (~400 cycles). At 64 transformer layers with 20-50 kernels each, eliminating boundaries is worth 30-50% throughput.

03 Quantization Efficiency

Are you using fewer bytes per weight without destroying quality? FP16 uses 2 bytes per weight. INT4 uses 0.5 bytes — 4x less bandwidth needed. But the dequantization math must be fast: if your dequant kernel can't feed the ALUs at line rate, you lose the bandwidth savings to compute overhead.

The best quantization kernels interleave dequantization with the multiply-accumulate, keeping the ALU pipeline full while the memory system streams weights. Poor implementations dequantize first, store intermediate results, then compute — doubling memory traffic.

04 Launch Overhead

Is the GPU idle between kernels? Each CUDA kernel launch costs 3-10 us of CPU-side dispatch. CUDA graphs amortize this by recording a sequence and replaying it, but graphs are rigid — they can't handle dynamic shapes without re-recording.

The CUDA driver API (as used by Lithos) can launch kernels with ~1 us overhead. But the real win is not launching fewer kernels faster — it is launching fewer kernels, period. One fused kernel per layer means 64 launches for the entire model instead of 1,000+.

05 Batching

Are you sharing weight reads across multiple requests? At batch=1, each weight byte produces one output element. At batch=8, each weight byte produces eight output elements — arithmetic intensity increases 8x, and you shift from memory-bound to compute-bound.

Continuous batching (vLLM, SGLang) dynamically packs requests so the GPU always has a full batch. But this requires managing a KV cache per request, which is its own memory challenge. PagedAttention solves the fragmentation problem; without it, you waste 60-80% of KV cache memory to internal fragmentation.

Engine Deep Dives

01 TensorRT-LLM

NVIDIA — C++ runtime, Python build phase

Why it is fast

  • Pre-compiled execution plan. The build phase analyzes the model graph and selects optimal kernel implementations, tile sizes, and memory layouts for the specific GPU. No runtime decisions.
  • Layer fusion at build time. RMSNorm+residual, QKV projection+rope, attention+softmax — fused into single kernels during the build phase, not at runtime.
  • FP8/INT8 quantization with calibration. NVIDIA's own calibration tools find per-channel scale factors that minimize accuracy loss. Their kernels use tensor core MMA instructions directly.
  • Custom CUDA kernels for attention. Not cuBLAS — hand-written kernels that exploit the specific memory access patterns of self-attention. This is what cuBLAS cannot optimize for.
  • Multi-GPU tensor and pipeline parallelism. NVLink-aware partitioning. NCCL integration. Designed for data center deployment from the start.
  • C++ runtime with no Python in the inference path. After build, the engine is a serialized blob that a C++ process loads and executes.

Architecture

Python build phase (minutes) → serialized engine file → C++ runtime loads and executes. The build phase uses TensorRT's optimizer to convert a PyTorch/HuggingFace model into a target-specific execution plan. The runtime uses CUDA streams, events, and custom memory pools.

Tradeoffs

  • Minutes-long build phase. 5-30 minutes depending on model size and optimization level. Every configuration change requires a rebuild.
  • Fixed batch size and sequence length baked into the plan. Dynamic shapes require multiple plans or padded execution.
  • NVIDIA-proprietary. The engine runtime is closed source. You cannot inspect, modify, or port the generated kernels.
  • Complex setup. Dependency on TensorRT, cuDNN, NCCL, and specific driver versions. Version matrix is fragile.

Typical numbers

~250-400 tok/s single stream on H100 for 27B W4A16 (estimated). Highest throughput of any engine at high batch sizes due to multi-GPU scaling and optimized plans.

02 vLLM

UC Berkeley — Python + PyTorch + cuBLAS/FlashAttention

Why it is fast

  • PagedAttention. Virtual memory for the KV cache. Instead of allocating one contiguous buffer per sequence, the KV cache is split into fixed-size blocks allocated on demand. Eliminates fragmentation — the reason most engines waste 60-80% of KV memory.
  • Continuous batching. New requests are inserted into the running batch without waiting for the current batch to finish. The scheduler dynamically adjusts the batch composition every iteration.
  • CUDA graphs. After warmup, the kernel launch sequence is recorded and replayed, amortizing the 3-10 us per-launch CPU overhead across dozens of kernels.
  • Prefix caching. If two requests share a common system prompt, the KV cache for that prefix is computed once and shared. Significant for multi-turn conversations and few-shot prompting.
  • Speculative decoding with Multi-Token Prediction (MTP). A small draft model proposes multiple tokens; the main model verifies in one forward pass. When the draft model is accurate, throughput multiplies.

Architecture

Python scheduler → PyTorch operations → cuBLAS for GEMMs, FlashAttention for attention → CUDA graphs for replay. The scheduler is written in Python and runs on the CPU. Kernel execution is delegated to well-optimized libraries. torch.compile provides JIT fusion for some operations.

Tradeoffs

  • Python in the dispatch path. The scheduler, tokenizer, and sampling logic run in Python. At batch=1, this adds measurable latency per token.
  • ~2 minute startup. Model loading, torch.compile warmup, CUDA graph recording all happen at startup. Cold start is painful for interactive use.
  • cuBLAS for GEMMs. cuBLAS is general-purpose — it cannot fuse dequantization with the matrix multiply. The weight read and dequant are separate memory passes.
  • Framework memory overhead. PyTorch's memory allocator, autograd metadata (even in inference mode), and Python object overhead consume GPU memory that could hold KV cache.

Measured on GH200

179 tok/s with GPTQ quantization and MTP5 speculative decoding. 13% bandwidth utilization at batch=1. KV cache capacity: 274K tokens. The 87% bandwidth waste is the inter-kernel gap problem — hundreds of small kernels with synchronization between each.

03 llama.cpp

Georgi Gerganov — C/C++ with CUDA backend

Why it is fast

  • Custom quantization kernels. Q4_K_M, Q5_K, Q6_K — hand-written CUDA kernels for each quantization format. The dequantization is fused into the dot product, avoiding intermediate storage.
  • Minimal dependencies. No PyTorch, no cuBLAS, no cuDNN. The entire inference stack is one C++ codebase. Less code means less overhead, less memory waste, and faster builds.
  • GGUF format designed for fast loading. Memory-mapped weight files. The weights are laid out in the file exactly as they will be consumed by the GPU — no deserialization, no conversion.
  • No framework overhead. No Python interpreter, no garbage collector, no autograd graph, no tensor metadata. Every byte of GPU memory holds weights or KV cache.

Architecture

C/C++ with a CUDA backend for GPU acceleration. Custom memory allocator. mmap-based weight loading — the OS pages weights into memory on demand, so startup is near-instant. The CUDA backend uses custom kernels for quantized operations and cuBLAS only for large FP16 GEMMs.

Tradeoffs

  • No tensor core usage for some quantization formats. The K-quant kernels use CUDA cores (FP32 FMA) rather than tensor cores (MMA). This leaves the tensor core units idle during quantized inference.
  • Generic tile sizes. The kernel tile dimensions are not tuned per-GPU — the same code runs on RTX 3060 and H100. This leaves performance on the table for high-end hardware.
  • Single-developer optimization capacity. One person writing and maintaining all CUDA kernels. Optimization breadth is necessarily limited compared to a team.
  • Limited batching support. Designed for single-user, single-stream inference. Server-mode batching exists but is not as mature as vLLM's continuous batching.

Typical numbers

~80-120 tok/s on H100 for 27B Q4_K_M. Fast startup (~5s). Excellent for local inference where simplicity and low resource usage matter more than peak throughput.

04 MLX

Apple — Swift/C++ with Metal backend

Why it is fast

  • Unified memory. CPU and GPU share the same physical memory. No PCIe transfers, no explicit copies. The weight tensor allocated by the CPU is the same physical memory the GPU reads. This eliminates the single biggest bottleneck on discrete GPU systems.
  • Lazy evaluation. Operations are recorded but not executed immediately. When results are needed, the framework fuses the recorded operations before dispatching to the GPU. This is implicit kernel fusion without programmer effort.
  • Metal shader compilation. Apple's Metal compiler generates GPU-specific code at runtime. Compile-time specialization bakes constants (layer dimensions, head count) into the shader, enabling the compiler to optimize aggressively.
  • Zero-copy weight loading. Weights are memory-mapped directly into the unified address space. No deserialization, no conversion, no copy. Model load time is bounded by page fault handling, not data movement.

Architecture

Swift/C++ framework inspired by the PyTorch API but designed for Apple Silicon from the ground up. Metal backend for GPU compute. The Metal Performance Shaders (MPS) library provides optimized GEMM implementations. simdgroup_matrix instructions on M-series chips provide the equivalent of NVIDIA's tensor cores.

Tradeoffs

  • Apple-only. No path to NVIDIA GPUs, no data center deployment, no cloud instances. The fastest Apple Silicon (M4 Ultra) has ~800 GB/s bandwidth vs. 4 TB/s on GH200.
  • Metal compiler is opaque. No equivalent of SASS disassembly. You cannot inspect the generated GPU instructions, verify scheduling decisions, or hand-tune at the instruction level.
  • Cannot control GPU scheduling at instruction level. No equivalent of PTX inline assembly. The programmer cannot specify instruction interleaving, register allocation, or shared memory bank conflict avoidance.
  • Limited to Apple's simdgroup_matrix for tensor ops. The matrix multiply unit is smaller than NVIDIA's tensor cores, and the API surface for controlling it is minimal.

Measured on M4 Max 128GB

51.1 tok/s for Qwen 27B Q4 with adaptive MTP chain. Given the ~400 GB/s memory bandwidth of M4 Max, this represents solid utilization. The unified memory architecture means zero copy overhead — a genuine architectural advantage for models that fit in memory.

05 SGLang

UC Berkeley — Python + PyTorch + FlashInfer

Why it is fast

  • RadixAttention. A tree-based prefix caching system. Instead of a flat cache, the KV cache is organized as a radix tree where shared prefixes are stored once. Multi-turn conversations and branching prompts reuse cached computation automatically.
  • Optimized scheduling. The scheduler is designed for structured generation workloads — JSON output, function calling, constrained decoding — where the generation pattern is known in advance.
  • FlashInfer integration. Uses FlashInfer instead of FlashAttention for the attention kernel. FlashInfer is optimized for variable-length sequences and supports more attention variants.
  • Compressed finite state machine for structured generation. When generating JSON or following a grammar, the valid token set is precomputed and applied during sampling, avoiding wasted decoding steps.

Architecture

Python frontend with a focus on the programming model — structured generation, multi-turn conversation, branching, and constrained decoding. The backend shares the same PyTorch + FlashInfer foundation as other Python engines. Differentiated by the programming interface more than the kernel layer.

Tradeoffs

  • Same PyTorch/cuBLAS foundation as vLLM. The kernel-level performance ceiling is similar. Gains come from better scheduling and caching, not from faster kernels.
  • Python scheduling overhead. Same fundamental limitation as vLLM — the critical path includes Python execution between GPU kernel launches.
  • Differentiated by programming model more than raw speed. If your workload is simple text generation, SGLang and vLLM perform similarly. SGLang shines on structured, multi-step, branching workloads.

Typical numbers

Comparable to vLLM: ~150-180 tok/s on H100 for 27B quantized models. The advantage is in multi-request scenarios where RadixAttention reuse is high — structured generation throughput can be 2-5x higher than naive approaches.

06 ExLlamaV2

turboderp — Python + custom CUDA extensions

Why it is fast

  • Hand-optimized CUDA kernels for EXL2 quantization. EXL2 uses mixed-precision, per-layer bit allocation — important layers get more bits, unimportant layers get fewer. The kernels are written specifically for this mixed format.
  • Custom attention implementation. Not FlashAttention, not cuBLAS — a from-scratch attention kernel tuned for single-user, single-GPU inference. Avoids the overhead of general-purpose libraries.
  • Extremely efficient memory usage. The EXL2 format achieves high compression ratios with minimal quality loss by letting the quantizer allocate bits where they matter most. 3.5 bits per weight average is common.

Architecture

Python wrapper around custom CUDA extensions. The Python layer handles model loading and high-level control. The CUDA extensions handle all compute. Focused entirely on single-GPU, single-user inference — no multi-GPU, no serving layer, no batching infrastructure.

Tradeoffs

  • Single-GPU only. No tensor parallelism, no pipeline parallelism. The model must fit in one GPU's memory.
  • Limited batching. Designed for one user at a time. Not suitable for server deployment with concurrent requests.
  • One-developer project. Like llama.cpp, the optimization scope is limited by one person's bandwidth. But the kernels that exist are very well optimized.
  • EXL2 quantization format is not universally supported. Models must be quantized specifically for ExLlamaV2. Cannot use GPTQ, AWQ, or GGUF weights directly.

Typical numbers

~100-150 tok/s on RTX 4090 for 27B models. Among the fastest options for single-GPU consumer hardware. The per-layer bit allocation means you can often fit larger models than with uniform quantization.

07 MLC-LLM / Apache TVM

OctoAI / Apache Foundation — compiler-based approach

Why it is fast

  • Compiler-based kernel generation. TVM auto-schedules kernels for the target hardware — analyzing the computation graph and generating hardware-specific code. No hand-written kernels needed (in theory).
  • Cross-platform. The same model definition compiles to CUDA, Metal, Vulkan, OpenCL, and WebGPU. One optimization framework for every backend.
  • Can target mobile and edge devices. The compiler generates lightweight inference code for phones, browsers, and embedded systems — platforms where PyTorch cannot run.

Architecture

Python model definition → TVM intermediate representation → auto-scheduling and optimization → target-specific compiled code. The auto-scheduler searches the space of possible implementations (tile sizes, loop orderings, vectorization strategies) and selects the fastest on the target hardware.

Tradeoffs

  • Compilation time. Auto-scheduling requires profiling thousands of candidate implementations. Full model compilation can take hours.
  • Auto-scheduling may not match hand-tuned kernels. The search space is vast but the search is heuristic. On specific hardware, a hand-written kernel by someone who understands the architecture will often win.
  • Complex toolchain. TVM, LLVM, target-specific compilers, the auto-scheduler, the runtime — many moving parts, each with their own version requirements.

Typical numbers

~120-150 tok/s on high-end NVIDIA GPUs for 27B quantized models. Competitive but rarely the fastest on any single platform. The value proposition is breadth — reasonable performance everywhere rather than peak performance somewhere.

08 Lithos

This project — Forth/Rust, direct PTX/SASS emission

Why it will be fast

  • Direct PTX/SASS emission. No compiler chain between the programmer and the GPU. The Forth-derived language emits PTX instructions directly; the planned SASS backend will emit binary GPU instructions. Every scheduling decision is explicit.
  • Fused mega-kernels. One kernel per transformer layer. RMSNorm, attention, GEMV, activation, residual — all fused into a single kernel launch. Intermediate results stay in registers and shared memory. No HBM round-trips between operations.
  • mmap weight loading on unified memory (GH200). Weights are memory-mapped from NVMe. On GH200's unified memory, the GPU reads weights directly from the mapped region — no staging through CPU memory, no explicit transfers. Startup time is bounded by the first page faults, not by bulk data movement.
  • No framework, no Python in the inference path. The runtime is Rust calling the CUDA driver API. No PyTorch, no cuBLAS, no Python interpreter. The entire dispatch path is compiled native code.
  • DeltaNet-native kernels. Lithos is being designed around DeltaNet, a linear-complexity attention alternative. The kernel architecture is optimized for DeltaNet's access patterns from the start, rather than retrofitting a transformer engine.
  • 132 KB of cubins replacing 800 MB of cuBLAS. The entire kernel payload is smaller than a single cuBLAS kernel. Less code to load, less instruction cache pressure, faster startup.

Architecture

Forth-derived language → PTX assembly → CUDA driver API (cuModuleLoadData, cuLaunchKernel) → GPU execution. The Forth words encode known-good patterns: memory access strategies, instruction interleaving, register allocation. The programmer composes these patterns; the system emits the corresponding PTX.

Tradeoffs

  • Not yet running full inference. Proof of life is passing. Kernels compile. The engine skeleton works. But end-to-end model inference is not yet demonstrated.
  • Extreme programming model. Writing GPU code in a Forth dialect is not accessible to most developers. The tradeoff is maximum control for minimum audience.
  • Single-hardware target initially. Optimized for GH200 (SM 9.0). Porting to other architectures requires new SASS patterns.

Targets

>400 tok/s single stream. <2s startup. >70% bandwidth utilization at batch=1. These targets come from first-principles analysis of the GH200's memory subsystem: 4 TB/s bandwidth, 14 GB weights at Q4, fused kernels that eliminate inter-operation gaps.

What the table reveals

The engines cluster into three tiers by architectural approach:

Framework engines (vLLM, SGLang) build on PyTorch and cuBLAS. They optimize at the scheduling layer — batching, caching, memory management — but inherit the kernel granularity and dispatch overhead of their foundation. Ceiling: ~200 tok/s at batch=1.

Native engines (llama.cpp, ExLlamaV2, TensorRT-LLM) write their own kernels. They control the compute path but still launch many kernels per layer. TensorRT-LLM's build-time fusion gets closest to optimal, but at the cost of flexibility and startup time. Ceiling: ~300 tok/s at batch=1.

Direct-to-silicon (Lithos) eliminates the kernel boundary entirely. One fused kernel per layer, with the programmer controlling instruction scheduling. This is the only approach that can theoretically reach >70% bandwidth utilization at batch=1. The tradeoff is that it requires understanding the GPU at the instruction level.

Lithos — from lithography. Writing on stone. The process that puts circuits on silicon. Lithos writes on the GPU.
