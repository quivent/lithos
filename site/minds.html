Five Minds on Hardware-Matched Quantization

The problem: fit one transformer layer (180M weights) into 66 MB of on-chip GPU memory while utilizing all four compute unit types. Five perspectives, five solutions.

The Constraint

A Hopper SM has 256 KB of register file plus 256 KB of L1/shared memory. Across 132 SMs, this is roughly 66 MB of on-chip storage. One layer of a 27B transformer has 180 million weights. At FP16, that is 360 MB — over 5x too large.

The target rate: 553,648,128 bits / 180,000,000 weights = 3.076 bits/weight. But metadata (scales, codebooks, maps) consumes some of that budget, so the effective index rate must be closer to ~2.8–2.9 bits/weight.

Meanwhile, standard inference uses only tensor cores for matrix multiplication. The FP32, INT32, and FP64 ALUs sit idle. A hardware-matched quantization format should put all four unit types to work in a pipelined dequantization scheme.

Five minds were given this problem. Each approached it from the foundations of their own discipline. All five arrived at sub-3-bit formats that pipeline across compute units — but the details differ in revealing ways.

Core Insight

Transformer weights are Gaussian, not uniform. A Lloyd-Max quantizer with 6 or 8 levels on a Gaussian source has entropy strictly less than log2(k) bits. By using a non-power-of-two number of quantization levels (6 or 7) and packing groups of 8 indices via mixed-radix arithmetic, Shannon achieves implicit entropy coding without variable-length codewords — preserving random access and GPU parallelism. The scheme then allocates more levels to sensitive weights (51.5% get 8 levels) and fewer to insensitive ones (48.5% get 6 levels), following the reverse water-filling principle.

Key Formulas

R(D) = (1/2) log2(sigma^2 / D) -- rate-distortion for Gaussian source
D(R) = sigma^2 * 2^(-2R) -- minimum MSE at rate R bits/weight
6^8 = 1,679,616 < 2^21 = 2,097,152 -- 8 weights x 6 levels pack into 21 bits (2.625 bits/wt)
7^8 = 5,764,801 < 2^23 = 8,388,608 -- 8 weights x 7 levels pack into 23 bits (2.875 bits/wt)
R_total = alpha * 3.25 + (1-alpha) * 2.875 = 2.93 --> alpha = 0.5147
Weighted SQNR = 10 * log10(1 / 0.02504) = 16.01 dB (bound: 16.99 dB, gap: 0.98 dB)

Pipeline Design

INT32: Mixed-radix unpack (divide-and-modulo chain for 6-level, bit-shift for 8-level) FP32: Codebook lookup + affine dequant: w_hat = scale * codebook[idx] + zero Tensor: Matrix-multiply accumulate (weight tile x activation) FP64: High-precision partial-sum accumulation to prevent numerical drift

Shannon's approach is the most theoretically grounded. The scheme operates within 0.98 dB of the rate-distortion bound — meaning no encoding can do meaningfully better at this bit rate.

Core Insight

Von Neumann models each SM as four parallel automata sharing a data path. The quantization format is not chosen for compression optimality but for Nash equilibrium of compute unit utilization. Seven quantization levels (symmetric: {-3,...,+3}) pack 8 values into 23 bits via base-7 encoding. Each group of 128 weights occupies exactly 384 bits (48 bytes = 6 x 64-bit words) — including scale factor. No padding, no waste. The dequantization pipeline is formalized as a composition of four operators, each mapping to a specific hardware unit type.

Key Formulas

log2(7) = 2.807 bits/weight -- information content per 7-level symbol
7^8 = 5,764,801 < 2^23 -- sub-pack: 8 heptary digits in 23 bits
B_group = 16 * 23 + 16 = 384 bits = 48 bytes -- exact, no padding
D = T4 . T3 . T2 . T1 -- operator chain: extract, decode, scale, contract
T2(Pj) = LUT4(Pj mod 7^4) || LUT4(floor(Pj / 7^4)) -- 9.6 KB shared-mem LUT
eta_pipe = (K-3)/(K+2) = 1,406,247/1,406,252 = 0.999996

Pipeline Design

INT32: Bit extraction (23-bit sub-packs from 64-bit words) + LUT-based base-7 decode 64 ops/cycle = capacity. LUT4[0..2400] in shared memory (9.6 KB/SM). FP32: Affine scaling: w = (q - 3) * scale. 128 FMA ops = capacity. Tensor: Tile GEMM: 128 weights x activation sub-vector. 256 FMA = capacity. FP64: Kahan compensated summation: t = y + p; c = (t-y) - p; y = t-c. 64 ops = capacity. All four units at 100% utilization at Nash equilibrium. T* = 1 cycle/group/SM.

Von Neumann's design is the most formally rigorous. The 384-bit group achieves exact packing with zero waste, and the game-theoretic analysis proves that the pipeline balance is a unique Nash equilibrium.

Core Insight

Ferrucci refuses to separate the format design from the validation loop. Like Watson, the system combines four independent sensitivity signals to allocate bits where they matter most — then validates empirically and iterates. The critical architectural decision: separate calibration quality (which rounding decisions to make) from packing layout (how bits sit in memory). Use AQLM or QuIP# for calibration. Use a hardware-matched format for packing. Same quantized values, different byte arrangement. The format itself uses fixed-width {2, 3, 4}-bit codes with per-group allocation, avoiding the complexity of mixed-radix or entropy coding.

Key Formulas

Sensitivity = f(Hessian_diag, activation_stats, layer_ablation, quant_simulation)
Minimize: total_perplexity_impact
Subject to: sum(bits_per_group * group_size) <= 66 MB per layer
Greedy allocation: start at 2 bits (45 MB), upgrade by sensitivity until budget full
Error budget: quant_rounding (~0.3 ppl) + dequant_numerical (~0.01) + calibration_mismatch (~0.15) + group_size (~0.05) < 0.5 ppl

Pipeline Design

INT32: Shift-and-mask extraction. Fixed-width {2,3,4}-bit codes. No division, no LUT, no data-dependent branching. ~960 ops/tile (1024 weights) at 64 cores = ~15 cycles. FP32: Scale + zero-point: w = (float)q * scale + zero. ~2048 ops/tile at 128 cores = ~16 cycles. Tensor: Standard mma.sync on dequantized FP16 tiles. 8-16 cycles. FP64: Optional Kahan accumulation for long reductions. Enabled per-layer only where rounding error is measurable. ~1 cycle when active. Never the bottleneck.

Ferrucci's design is the most pragmatic. It treats format parameters as a search space (60 configurations) that Lithos JIT compilation can explore in under 10 minutes. The architecture is right even if the first parameter choice is not — you iterate on measurements, not theory.

Core Insight

Turing defines a complexity class he calls SHIFT-MASK-SCALE: the set of functions computable by shift, bitwise-AND, and FMA alone. Any dequantization requiring division, large LUTs, or data-dependent branching is inadmissible for the inner loop. The format uses three fixed tile types (T4 at 4-bit, T3 at 3-bit, T2 at 2-bit), mixed within a layer via Bayesian allocation. The format descriptor is universal; the parameters are model-specific. Lithos JIT compiles the descriptor into a specialized SASS kernel with shift amounts and masks baked in as constants.

Key Formulas

4.25*p4 + 3.25*p3 + 2.25*p2 = 2.93, p4+p3+p2 = 1
Solution: p4=0.10, p3=0.48, p2=0.42 (with g=128 metadata overhead)
importance(bit_j) ~ |dL/d(bit_j)| -- first-order (gradient)
importance(bit_j) ~ (dL/d(bit_j))^2 / (d^2L/d(bit_j)^2) -- second-order (OBS)
group_importance(g) = ||X_g||_F * ||W_g - Dequant(Quant(W_g))||_F
Total: 180M * 2.93 / 8 = 65.925 MB (75 KB headroom)

Pipeline Design

INT32: Shift-and-mask extraction. All tile types cost 2 INT32 ops/weight. 2048 ops/tile at 64 cores = 32 cycles. (Reducible to 16 cycles by sharing INT ops on FP32 datapath.) FP32: INT-to-float conversion + FMA scale: w = (float)q * scale + zero. 3072 ops/tile at 128 cores = 24 cycles. Tensor: mma.sync m16n8k16 on dequantized FP16. 16 cycles. FP64: Accumulate partial sums: accum_f64 += (double)tile_result. 16 ops/tile at 64 cores = <1 cycle. Vastly underutilized. Key insight: even at 2x pipeline imbalance (32 vs 16 cycles), on-chip decode (32 cycles) beats HBM read (400 cycles) by 12.5x.

Turing's design is the most computationally principled. The SHIFT-MASK-SCALE constraint guarantees that the dequantization function is deterministic, decomposable, and free of warp divergence. The format is universal (one kernel structure); the parameters are specialized (per model, per layer).

Core Insight

Lovelace alone rejects the premise that quantization is about compressing individual numbers. Weights are patterns of relations, like a Jacquard fabric. The encoding should be a multi-stratum decomposition where each stratum is expressed in the notation native to the hardware that must reconstruct it. The result is a four-voice fugue: FP64 units compute high-precision structural centroids, tensor cores execute a product-quantization codebook multiply, INT32 units unpack bitplane residuals, and FP32 units apply sparse harmonic corrections. All four voices sound simultaneously with zero data dependencies between strata.

Key Formulas

w = skeleton + fabric + texture + harmonics -- additive four-stratum composition
Stratum I (FP64): 0.031 bits/wt -- FP64 centroid per 4096-weight super-block
Stratum II (Tensor): 2.000 bits/wt -- product-quantized codebook indices
Stratum III (INT32): 0.750 bits/wt -- bitplane-coded residual with integer scale
Stratum IV (FP32): 0.150 bits/wt -- sparse harmonic corrections (~3.8% of weights)
Total: 2.931 bits/wt --> 65.9 MB per layer

Pipeline Design

FP64: Compute high-precision block centroids (base + delta). Zero cross-block error accumulation. Analogous to Bernoulli term precision. Tensor: Codebook matrix multiply: W_block = C_warp[P_rows,:] * S * C_weft[P_cols,:]^T The reconstruction IS a matrix multiply. The notation IS the hardware op. INT32: Bitplane residual unpack: mask decode, sign/magnitude extract, integer scale. Pure bitwise + integer multiply. Native INT32 work. FP32: Sparse harmonic corrections: mask decode, FP16 upcast, scale, accumulate. Targets the ~3.8% highest-sensitivity weights. All four strata decode in parallel (zero inter-stratum dependencies). Critical path depth: 3 (decode | convert | sum). Compare GPTQ: depth 5.

Lovelace's design is the most architecturally original. By decomposing weights into four strata matched to four unit types, she achieves the shallowest critical path (depth 3 vs. depth 5 for conventional dequant) and gives every compute unit purposeful, native work.

Synthesis

Where They Agree

Sub-3-bit is feasible. All five designs fit 180M weights into 66 MB at rates between 2.93 and 3.08 bits per weight. Shannon proves it information-theoretically (0.98 dB from the rate-distortion bound). Turing argues it from intrinsic dimensionality (trained weights have far fewer effective degrees of freedom than parameter count). Ferrucci validates it empirically (AQLM and QuIP# already achieve 2-bit with acceptable quality). The consensus is overwhelming: ~3 bits/weight preserves >97% of signal variance.

Pipeline across all four unit types works. Every design assigns meaningful work to INT32 (bit extraction/decode), FP32 (scale application), tensor cores (matrix multiply), and FP64 (precision accumulation or centroid computation). The insight is shared: current inference wastes >50% of on-chip compute by ignoring INT32 and FP64 ALUs. A hardware-matched format converts this waste into useful dequantization work that runs concurrently with tensor core consumption.

The format should be hardware-specific, not model-specific. All five reject the idea of a single universal quantization format. Shannon's mixed-radix packing, von Neumann's 384-bit groups, and Turing's tile types are all designed for the specific unit counts and throughputs of Hopper SMs. Turing and Ferrucci both emphasize that Lithos's microsecond recompilation makes per-model format specialization free. The universal kernel, specialized parameters pattern appears in three of the five designs.

Where They Differ

Encoding details: mixed-radix vs. fixed-width vs. multi-stratum. Shannon and von Neumann use non-power-of-two levels (6, 7, or 8) packed via mixed-radix arithmetic, which requires integer division in the decode path. Turing and Ferrucci insist on power-of-two levels ({2, 3, 4}-bit) with pure shift-and-mask extraction — no division, no LUT, no warp divergence. Lovelace takes a third path entirely: learned codebooks with product quantization, where the decode is a matrix multiply on tensor cores rather than integer arithmetic. The division-vs-shift debate is the sharpest technical disagreement.

LUT-based decode vs. direct decode. Von Neumann's design hinges on a 9.6 KB shared-memory lookup table that converts base-7 packed integers to heptary digits in two table lookups per sub-pack. Shannon's divide-and-modulo chain achieves the same result without a LUT but at higher INT32 cost. Turing explicitly rules out LUTs larger than the register file. Ferrucci avoids the question by using fixed-width codes. The tradeoff is shared-memory capacity vs. ALU cycles.

The role of FP64. This is the most divergent axis. Von Neumann and Shannon use FP64 for Kahan compensated summation of partial products — a precision-preservation role. Lovelace assigns FP64 a structural role: computing high-precision block centroids that anchor the coarse representation. Ferrucci makes FP64 optional, arguing that most layers do not need it and the register pressure cost is not justified everywhere. Turing acknowledges FP64 is vastly underutilized even in his scheme (<1 cycle per tile vs. 32 for INT32). The question remains open: is FP64 worth the pipeline stage, or should it be an opportunistic resource?
