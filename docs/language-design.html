Language Design

Draft syntax under review. Assembly-flavored. Literal PTX mnemonics.

Draft Syntax Under Review This page shows a preliminary proposal for Lithos surface syntax. It is not the way the language will be — it is the current shape on the table, awaiting review. Details will change. The examples below compile in your head, not yet in a compiler.

The shape: assembly-flavored, space-separated, one instruction per line. Every mnemonic is a literal PTX mnemonic. No parentheses, no commas, no expression trees. The compiler assigns physical registers; the programmer writes meaningful names.

Three worked examples follow, then twelve design decisions that fix the shape.

Examples

Example 1

Vector add

The simplest useful kernel. Thread index, bounds check, three loads/stores, one add.

kernel vadd  a *f32  b *f32  c *f32  n u32
    mov.u32       tid     thread.x
    mad.lo.u32    tid     block.x  block.dim.x  tid
    setp.ge.u32   p0      tid      n
    @p0 bra done

    ld.global.f32 x  [a + tid*4]
    ld.global.f32 y  [b + tid*4]
    add.f32       z  x y
    st.global.f32 [c + tid*4]  z

    done: exit
Example 2

RMSNorm

Compile-time constant, shared memory, warp-level reduction, mixed f16/f32, barrier, final cast-and-store.

kernel rmsnorm  x *f16  weight *f16  output *f16  eps f32
    const H 5120
    shared sum f32

    ld.global.f16 v         [x + thread.x*2]
    cvt.f32.f16   vf        v
    mul.f32       sq        vf vf

    warp_reduce_sum sq -> warp_total
    setp.eq.u32   p0        thread.x 0
    @p0 st.shared.f32 [sum] warp_total
    barrier

    ld.shared.f32 total     [sum]
    div.f32       mean      total  H
    add.f32       denom     mean   eps
    rsqrt.f32     scale     denom

    ld.global.f16 w         [weight + thread.x*2]
    cvt.f32.f16   wf        w
    mul.f32       scaled    vf scale
    mul.f32       out       scaled wf
    cvt.f16.f32   out16     out
    st.global.f16 [output + thread.x*2]  out16
Example 3

Tensor core tile

Shared/register declarations with shapes, a for loop, async copy with wait group, and a full-precision MMA mnemonic.

kernel gemm_tile  A *f16  B *f16  C *f32
    const M 64
    const N 64
    const K 16

    shared   at  f16[64, 16]
    shared   bt  f16[16, 64]
    register acc f32[32]

    for k in 0..K step 16
        cp.async.cg.shared.global at  [A + k*2]  1024
        cp.async.cg.shared.global bt  [B + k*2]  1024
        cp.async.wait_group 0
        barrier

        mma.sync.m16n8k16.f16.f32 acc at bt

    store_tile C acc

Design Decisions

Twelve choices fix the shape. Each one is defensible in isolation; together they commit the language to staying within a hand's reach of PTX.

01No parens, no commas

Parameters on the kernel line are space-separated. kernel vadd a *f32 b *f32 c *f32 n u32. Double spaces between logical groups are a convention, not syntax — whitespace between tokens is whitespace.

02Literal PTX mnemonics

Every instruction name is the actual PTX mnemonic: add.f32, ld.global.f32, setp.ge.u32, mad.lo.u32. No renaming, no prettification. If you know PTX, you know Lithos.

03Named registers

The programmer writes meaningful names (tid, x, y, z, scale, denom). The compiler assigns physical registers. This is the one concession to readability that the PTX surface itself already makes.

04Destination first, sources after

add.f32 z x y means z = x + y. This matches PTX convention exactly. Every three-operand instruction reads the same way: result, then operands.

05Memory syntax

[a + tid*4] for address computation. The compiler emits whatever address arithmetic the target needs — scaled indexing, separate mad, whatever. The source expresses intent; the emitter picks the encoding.

06Predication

@p0 bra done. Same as PTX. A predicate register gates a single instruction. Inverted form @!p0 behaves as expected.

07Labels

done: marks a jump target. Bare identifier followed by a colon. The label lives on the same line as an instruction or on its own — either is fine.

08Compile-time constants

const H 5120. The name is baked into the emitted code as an immediate. No runtime slot, no register pressure. Constants participate in arithmetic at compile time when possible.

09Memory spaces

Explicit keywords: shared, register. Pointer parameters are implicitly global. There is no "just memory" — every declaration names the space it lives in.

10Built-in globals

thread.x, block.x, block.dim.x, warp.id, lane.id. Dotted names for the fixed vocabulary of GPU-provided identifiers. They read as data, not as function calls.

11-> for non-PTX abstractions

warp_reduce_sum sq -> warp_total. The arrow marks a word that expands to more than one PTX instruction. When the thing you want doesn't map to a single mnemonic, the syntax reminds you of that at the call site.

12Tensor core mnemonics stay full-length

mma.sync.m16n8k16.f16.f32. The variant matters — the shape, the input type, the accumulator type are all part of the instruction's identity. Shortening the name would lose the information that makes the line meaningful.

Vocabulary

Total surface size: ~90 words.

That includes:

  • Structure keywords — kernel, const, shared, register, for, in, step
  • PTX mnemonics — the subset needed for transformer inference (loads, stores, arithmetic, comparison, branch, cp.async, mma.sync, barrier, exit)
  • Built-in identifiers — thread.x, block.x, block.dim.x, warp.id, lane.id, and a handful more
  • Types — f16, f32, u32, i32, pointer forms (*f32), array shapes (f16[64, 16])

A human can learn the whole surface in an afternoon. An LLM can hold the whole surface in a system prompt.

Functions, Not Kernels

The insight

There is no “kernel” in Lithos. A kernel is a math function that happens to run on the GPU. It can be as trivial as 1 + 1. It can be as large as a full DeltaNet layer. There is no minimum size, no special construct, no ceremony.

What GPU programmers call “fusion” is function composition. When the compiler inlines a call, two functions become one GPU launch. That is not an optimisation — that is what function calls do.

A “fused kernel” is a redundant phrase. Every function is already a composition of smaller functions. RMSNorm is square → sum → 1/√ → multiply. SiLU is negate → ex → +1 → 1/ → multiply. The word “fused” only exists because other systems make separation the default and fusion the exception. In Lithos, composition is the default. There is nothing to fuse because nothing was ever separated.

Status

Draft, not finalized

This syntax is a proposal. It has not been ratified by running code through a compiler and discovering which parts hurt. Expect movement in:

  • How -> abstractions are defined and composed
  • Array shape syntax inside shared and register declarations
  • Whether for loops are the right control-flow primitive or whether everything should be labels-and-branches like PTX itself
  • The exact spelling of built-in globals (thread.x vs tid.x vs %tid.x)
  • How immediate constants participate in address arithmetic vs operand lists

Review comments welcome. The goal at this stage is to find the parts that read wrong before they are set in a parser.
